Module 12 — Kubernetes on AWS: EKS

Minikube taught concepts; EKS is where they meet production — and where Kubernetes meets IAM. AWS manages the control plane (etcd, API server, upgrades). You own the nodes, add-ons, and workloads. The capstone cluster is created in this module.

1. Why Managed Kubernetes?

OptionYou manageAWS managesCost (est.)
EKS (managed)Nodes, add-ons, workloadsControl plane (API server, etcd, upgrades)$0.10/h + nodes
Self-managed K8s on EC2Everything including control planeEC2 infra onlyNode cost only, but high ops overhead
Fargate on EKSWorkloads only (no nodes to manage)Control plane + node-level computePer-pod pricing, higher per-unit cost
⚠️ EKS is not free

The control plane alone costs ~$72/month ($0.10/hr). Add t3.medium nodes (~$30/month each), NAT Gateway ($32/month), and ALB, and a minimal cluster costs ~$150-200/month. Always teardown lab clusters after learning.

2. Creating an EKS Cluster with eksctl

eksctl is the recommended CLI for EKS — it creates the cluster, node group, IAM roles, and VPC attachments in a single command. Under the hood it generates and applies CloudFormation stacks.

bash — install eksctl
curl --silent --location \
    "https://github.com/eksctl-io/eksctl/releases/latest/download/eksctl_Linux_amd64.tar.gz" \
    | tar xz -C /tmp
sudo mv /tmp/eksctl /usr/local/bin/
eksctl version
yaml — cluster config file (eksctl)
# cluster.yaml — the capstone cluster definition
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig
metadata:
  name: techitfactory-cluster
  region: us-east-1           # Module 10 region choice
  version: "1.29"

vpc:
  subnets:
    private:                   # worker nodes go in private subnets (Module 10 security model)
      us-east-1a: { id: subnet-0priv1 }
      us-east-1b: { id: subnet-0priv2 }
    public:                    # ALB goes in public subnets
      us-east-1a: { id: subnet-0pub1 }
      us-east-1b: { id: subnet-0pub2 }

managedNodeGroups:
- name: workers
  instanceType: t3.medium    # 2 vCPU, 4 GB — minimum for a real workload
  minSize: 2
  maxSize: 5
  desiredCapacity: 2
  privateNetworking: true    # nodes in private subnets
  labels: { role: worker }

iam:
  withOIDC: true              # enables IRSA (Section 7)
bash — create the cluster (~15 min)
# eksctl reads cluster.yaml, creates CloudFormation stacks
eksctl create cluster -f cluster.yaml

# Verify
kubectl get nodes
NAME                          STATUS   ROLES    AGE   VERSION
ip-10-0-3-142.ec2.internal   Ready    <none>   2m    v1.29.3-eks-ae9a62a
ip-10-0-4-87.ec2.internal    Ready    <none>   2m    v1.29.3-eks-ae9a62a

3. Node Groups

TypeWhat AWS managesFinOps consideration
Managed Node GroupNode lifecycle, OS patches, coordinated drainsOn-demand pricing — predictable
Self-managed Node GroupNothing — you own the EC2 ASGFull control, but you handle upgrades
Spot Node GroupSame as managed but on Spot instancesUp to 90% cheaper — for stateless workloads (capstone FinOps)
Fargate ProfileCompute + OS (serverless nodes)Per-pod/vCPU/GB — no idle capacity cost
💡 Spot node groups for stateless workloads

Add a second node group with Spot instances and a taint (spot=true:NoSchedule). Stateless Deployments with a spot=true:NoSchedule toleration run cheaply on Spot. Stateful workloads (databases, Prometheus) use a toleration only for on-demand nodes. The capstone FinOps section links here — this is how you cut a $200 cluster bill in half.

4. kubectl Authentication to EKS

The most common EKS surprise: "kubectl access denied" is usually an IAM problem, not a Kubernetes RBAC problem. Here's how the auth chain works:

EKS Authentication Flow
kubectl → aws-iam-authenticator
(generates token from IAM creds) → EKS API Server → aws-auth ConfigMap
(maps IAM → K8s user/group) → K8s RBAC
bash — configure kubectl for EKS
# Update your local kubeconfig (adds EKS cluster to ~/.kube/config)
aws eks update-kubeconfig \
    --region us-east-1 \
    --name techitfactory-cluster

# Verify you're talking to the right cluster
kubectl config current-context
arn:aws:eks:us-east-1:123456789012:cluster/techitfactory-cluster

kubectl get nodes   # should show your EC2 worker nodes
yaml — aws-auth ConfigMap (grant a CI role cluster access)
apiVersion: v1
kind: ConfigMap
metadata:
  name: aws-auth
  namespace: kube-system
data:
  mapRoles: |
    # Node group role (must be here — nodes use this to join the cluster)
    - rolearn: arn:aws:iam::123456789012:role/eksctl-workers-NodeInstanceRole
      username: system:node:{{EC2PrivateDNSName}}
      groups: [system:bootstrappers, system:nodes]
    # Jenkins role — gets deploy rights (Incident 1: this line missing = auth failure)
    - rolearn: arn:aws:iam::123456789012:role/JenkinsDeployRole
      username: jenkins
      groups: [deploy-group]           # bound to a K8s ClusterRole via RBAC

5. Deploy the Module 11 App on EKS

Node group instances have an instance role with ECR pull permissions — no imagePullSecrets needed (Module 10 instance-role payoff in action). The Helm chart from Module 11 deploys unchanged; only the image registry URL changes.

bash — deploy to EKS via Helm
# Ensure you're pointing at the EKS cluster
kubectl config use-context arn:aws:eks:us-east-1:123456789012:cluster/techitfactory-cluster

# Install with ECR image (no imagePullSecrets — node role handles auth)
helm upgrade --install techitfactory-prod ./techitfactory \
    --namespace production --create-namespace \
    --set image.repository=123456789012.dkr.ecr.us-east-1.amazonaws.com/techitfactory/api \
    --set image.tag=1.4.2 \
    -f values-prod.yaml

# Verify pods are running on real EC2 nodes
kubectl get pods -n production -o wide

6. VPC CNI & AWS Load Balancer Controller

VPC CNI: Unlike Minikube (where pods get an internal overlay IP), EKS pods get real VPC IPs from the subnet. Each node can host pods equal to its ENI limit × IPs per ENI — a t3.medium can host ~17 pods. This is efficient (no overlay overhead) but means your subnets must be sized for peak pod count.

AWS Load Balancer Controller: When you apply an Ingress manifest on EKS, the ALB controller creates a real AWS Application Load Balancer automatically. Service type LoadBalancer creates an NLB.

bash — install AWS Load Balancer Controller via Helm
# Add the EKS chart repository
helm repo add eks https://aws.github.io/eks-charts
helm repo update

# Install the controller (requires IRSA service account — see Section 7)
helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
    -n kube-system \
    --set clusterName=techitfactory-cluster \
    --set serviceAccount.create=false \
    --set serviceAccount.name=aws-load-balancer-controller

# Tag the public subnets (required for the controller to discover them)
aws ec2 create-tags --resources $PUB1 $PUB2 \
    --tags Key=kubernetes.io/role/elb,Value=1
⚠️ Subnet IP exhaustion

With VPC CNI, each pod consumes a real subnet IP. A /24 subnet (256 IPs) supports ~250 pods across all nodes in that AZ. Size your subnets generously — /23 (512) or /22 (1024) for production. Incident 2 (pods stuck Pending) is often this. Mitigation: enable prefix delegation (each ENI gets a /28 block instead of individual IPs).

7. IRSA — IAM Roles for Service Accounts

A node role grants all pods on a node the same AWS permissions — too broad. IRSA lets individual pods assume their own IAM role via a Kubernetes Service Account, with no credentials on disk. This is the capstone security table's IRSA row, delivered.

IRSA Authentication Flow
Pod → K8s Service Account
(annotated with IAM role ARN) → OIDC Provider
(on EKS cluster) → AWS STS
(issues temp creds) → AWS Service
(S3, DynamoDB…)
bash — create IRSA for the app to access S3
# Create IAM role + K8s Service Account in one eksctl command
eksctl create iamserviceaccount \
    --cluster techitfactory-cluster \
    --namespace production \
    --name api-service-account \
    --attach-policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess \
    --approve

# The service account gets an annotation:
kubectl describe sa api-service-account -n production
# Annotations: eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/...

# Reference it in the Deployment pod spec:
# spec.serviceAccountName: api-service-account
# The AWS SDK inside the pod automatically picks up the IRSA token — no code changes

8. CI/CD Pipeline → EKS

The Module 09 Jenkins pipeline's SSH deploy stage gets replaced with an EKS Helm upgrade. The image tag built in CI becomes the Helm release value. This is the Module 09 tag-naming discipline reaching its destination.

groovy — Jenkinsfile deploy stage for EKS
stage('Deploy to EKS') {
    steps {
        withCredentials([[$class: 'AmazonWebServicesCredentialsBinding',
                          credentialsId: 'aws-deploy-role']]) {
            sh """
                # Configure kubectl for the EKS cluster
                aws eks update-kubeconfig \\
                    --region us-east-1 \\
                    --name techitfactory-cluster

                # Helm upgrade — bumps image.tag to the new build
                helm upgrade --install techitfactory-prod ./techitfactory \\
                    --namespace production \\
                    --set image.tag=${IMAGE_TAG} \\    # Module 09 dynamic version
                    -f values-prod.yaml \\
                    --wait --timeout 5m               # block until rollout completes

                # Smoke test against the ALB DNS
                ALB_DNS=\$(kubectl get ingress -n production \\
                    -o jsonpath='{.items[0].status.loadBalancer.ingress[0].hostname}')
                curl -f http://\$ALB_DNS/health
            """
        }
    }
}

stage('Rollback on Failure') {
    when { expression { currentBuild.result == 'FAILURE' } }
    steps {
        sh 'helm rollback techitfactory-prod 0 -n production'  # 0 = previous
    }
}

9. Real-Time Incidents & Troubleshooting

Incident 1 — "kubectl: You must be logged in to the server"

StageDetails
SymptomJenkins pipeline fails: error: You must be logged in to the server (Unauthorized)
Root CauseThe Jenkins IAM role is not in the aws-auth ConfigMap. EKS only trusts IAM identities that are explicitly mapped in this ConfigMap.
FixEdit the ConfigMap (kubectl edit configmap aws-auth -n kube-system) and add a mapRoles entry for the Jenkins role ARN. Grant it the minimum necessary K8s group (a custom group with only helm upgrade rights).
PreventionStore the aws-auth ConfigMap in Git and apply it via CI — treat it as infrastructure code, not a manual config. Changes need to go through the same review process as IAM changes.

Incident 2 — Pods Stuck Pending (Subnet IP Exhaustion)

bash — diagnose and fix VPC CNI exhaustion
kubectl describe pod pending-pod | grep -A 5 Events:
Warning  FailedCreatePodSandBox  ... failed to assign an IP address to the container

# Check how many IPs are left in each subnet
aws ec2 describe-subnets \
    --filters "Name=vpc-id,Values=$VPC_ID" \
    --query 'Subnets[*].[SubnetId,AvailableIpAddressCount,CidrBlock]' \
    --output table

# Enable prefix delegation (each ENI gets a /28 = 16 IPs instead of 1)
kubectl set env daemonset aws-node \
    -n kube-system ENABLE_PREFIX_DELEGATION=true
kubectl rollout restart daemonset aws-node -n kube-system

Incident 3 — ALB Never Provisions

StageDetails
SymptomIngress resource created but kubectl get ingress shows empty ADDRESS column forever.
InvestigationCheck the controller logs: kubectl logs -n kube-system deploy/aws-load-balancer-controller. Common errors: controller not installed, subnet tags missing, IRSA not configured for the controller service account.
FixTag subnets: public subnets need kubernetes.io/role/elb=1; private subnets need kubernetes.io/role/internal-elb=1. Also verify the Ingress annotation: kubernetes.io/ingress.class: alb.

Incident 4 — Bill Shock: Forgotten EKS Cluster

bash — ⭐ teardown discipline
# Delete all workloads first (prevents ALBs from not cleaning up)
helm uninstall techitfactory-prod -n production
kubectl delete ns production staging development

# Delete the cluster (eksctl removes nodes, security groups, CloudFormation stacks)
eksctl delete cluster --name techitfactory-cluster --region us-east-1

# Verify no dangling ALBs (common bill source)
aws elbv2 describe-load-balancers \
    --query 'LoadBalancers[*].[LoadBalancerName,DNSName]' --output table

# Check for unattached EIPs
aws ec2 describe-addresses \
    --query 'Addresses[?AssociationId==null].[AllocationId,PublicIp]' --output table

Interview Questions

Click a card to reveal a strong candidate answer.

Q: What does EKS manage for you versus what you still own?

EKS manages the control plane — the API server, etcd, controller-manager, scheduler — across multiple AZs with automatic failover and managed upgrades. You own everything else: worker nodes (OS patches, instance types, scaling), add-ons (CNI, CoreDNS, ALB controller), workloads, namespaces, and networking configuration. The $0.10/hr control plane fee is the price of not running five control plane EC2s yourself. The real operational win is automatic etcd backups and control-plane HA.

Q: How does kubectl authenticate to an EKS cluster?

kubectl uses the aws-iam-authenticator exec plugin to exchange your current IAM credentials for a bearer token. The EKS API server validates this token against AWS STS. If the IAM identity is in the aws-auth ConfigMap, it's mapped to a Kubernetes user or group. From there, standard K8s RBAC takes over. The two failure points: (1) IAM identity not in aws-auth — "unauthorized" at the IAM layer, (2) K8s RBAC denies the action — "forbidden" at the K8s layer. Read the error message carefully — it tells you which layer failed.

Q: What is IRSA and when should you use it over a node role?

A node role grants every pod on a node the same AWS permissions — the blast radius of a compromised pod is the entire node's IAM scope. IRSA (IAM Roles for Service Accounts) uses OIDC federation to bind an IAM role to a specific K8s service account, so only pods using that service account get those permissions. A backend API pod gets S3 read; a batch job pod gets DynamoDB write; other pods get nothing. This is least-privilege at the pod level. Always use IRSA in production — node roles only for the node group itself.

Q: Why do EKS pods get real VPC IPs, and what can go wrong?

EKS uses the VPC CNI plugin, which assigns each pod an actual VPC IP address from the node's ENI. This is efficient — no overlay network — and pods are directly routable from the VPC without NAT. What breaks: your subnets can run out of IPs. A /24 subnet has 251 usable IPs; with many pods per node, a deployment surge can exhaust the subnet. Fix: size subnets generously (/23 or larger), enable prefix delegation (each ENI gets a /28 block = 16 IPs instead of 1), and monitor AvailableIpAddressCount in CloudWatch.

Q: How does an Ingress on EKS provision an ALB?

When you apply an Ingress resource with ingressClassName: alb, the AWS Load Balancer Controller (running as a Deployment in kube-system) watches for new Ingress objects. It calls the AWS ELBv2 API to create an ALB in the annotated subnets, creates target groups for each backend service, and registers the pod IPs directly (not the node ports). The ALB's address appears in kubectl get ingress after ~30-60 seconds. Prerequisites: subnets must be tagged, the controller needs IRSA with the LBC IAM policy.

Q: How would a CI system safely deploy to EKS?

The CI system (Jenkins, GitHub Actions) assumes a dedicated IAM role with only the minimum K8s permissions needed — specifically, only helm upgrade rights on specific namespaces, not cluster-admin. This role is mapped in aws-auth. The pipeline calls aws eks update-kubeconfig, runs helm upgrade --install --wait (blocking until the rollout is confirmed healthy), then does a smoke test. If any step fails, the pipeline calls helm rollback. No long-lived kubeconfig files in Jenkins — the IAM role credential is short-lived and rotates automatically.
