1. What Is an Artifact Repository?

You built a JAR file in Module 05. Why can't you just commit it to Git? (Because Git chokes on binaries). Why not upload it to an S3 bucket? (Because S3 doesn't understand Maven/npm protocols).

An Artifact Repository (like Sonatype Nexus, JFrog Artifactory, or AWS CodeArtifact) is a specialized database built strictly to store, version, and serve software packages.

Git (Code) → Jenkins (Build) → Nexus (Warehouse) → Ansible/K8s (Deploy)
Why Do We Need It?
  • Immutability: A true repository enforces that once app-1.0.0.jar is uploaded, it can never be overwritten.
  • Access Control: Developers can read libraries, but only the CI/CD server is allowed to publish new releases.
  • Proximity & Speed: Caching the internet locally.
💡 Real-World Scenario: "The Internet is Down"

Imagine npmjs.org goes offline for an hour. Normally, your Jenkins builds would instantly fail because they can't run npm install. If you have Nexus acting as a caching proxy, Jenkins pulls the cached packages from your local Nexus server instead. The build passes. Nobody even notices the internet is broken.

2. Installing & Running Nexus (systemd)

This is where Module 02 (Linux) pays off. We are going to provision a real server and install Nexus as a background service.

The Setup Steps
bash
# 1. Install Java (Nexus runs on Java)
sudo apt update && sudo apt install openjdk-8-jdk -y

# 2. Create a dedicated user (NEVER run Nexus as root!)
sudo adduser --system --no-create-home nexus

# 3. Download and extract Nexus to /opt/nexus
wget https://download.sonatype.com/nexus/3/latest-unix.tar.gz
tar -xvf latest-unix.tar.gz
sudo chown -R nexus:nexus /opt/nexus /opt/sonatype-work
Writing the systemd Service (Module 02 Callback)

We need Nexus to start automatically if the server reboots. We write a systemd unit file at /etc/systemd/system/nexus.service.

ini
[Unit]
Description=Nexus Service
After=network.target

[Service]
Type=forking
LimitNOFILE=65536         # Required: Nexus opens thousands of files
User=nexus                # Run as the restricted user
ExecStart=/opt/nexus/bin/nexus start
ExecStop=/opt/nexus/bin/nexus stop
Restart=on-abort

[Install]
WantedBy=multi-user.target
⚠️ Incident: Nexus Won't Start

Nexus is a Java application. It eats RAM. By default, it requires at least 2GB of memory. If you put it on a 1GB cloud server (like a $5 DigitalOcean droplet), the Linux OOM (Out of Memory) Killer will instantly kill it. Running sudo journalctl -u nexus will show you exactly that it failed due to heap size constraints.

Open port 8081 with sudo ufw allow 8081 (Module 04). The initial admin password is autogenerated and hidden at /opt/sonatype-work/nexus3/admin.password.

3. Reverse Proxy + HTTPS

Nexus runs on port 8081 over HTTP. You should never expose port 8081 directly to the internet. We must put Nginx in front of it to provide a clean port 443 HTTPS URL (e.g., https://nexus.mycompany.com).

nginx.conf
server {
    listen 443 ssl;
    server_name nexus.mycompany.com;

    ssl_certificate /etc/letsencrypt/live/nexus/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/nexus/privkey.pem;

    location / {
        # Proxy all traffic to the local Nexus port
        proxy_pass http://localhost:8081;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        
        # Required for uploading large artifacts (like a 500MB Docker image)
        client_max_body_size 1G;
    }
}

(This is the exact reverse proxy pattern taught in Module 04!)

4. Repository Types (Hosted, Proxy, Group)

Nexus is not just one big bucket. It is split into specific repository types. Understanding the difference between Hosted, Proxy, and Group is the conceptual heart of this module.

1. Hosted Repositories

This is where your artifacts live. You build a JAR file locally and push it here.

  • maven-releases: For immutable, production-ready artifacts (e.g., 1.0.0).
  • maven-snapshots: For mutable, in-development artifacts (e.g., 1.0.0-SNAPSHOT).
2. Proxy Repositories

A caching mirror of an external public registry (like Maven Central or npmjs.org). When a developer requests an npm package, Nexus checks if it has it. If not, it fetches it from the internet, caches it on disk, and serves it. Subsequent requests are instant and survive internet outages.

3. Group Repositories

A logical bundle. Instead of configuring developers to pull from 5 different URLs, you put the Hosted and Proxy repos inside a Group. The developer points to ONE URL, and Nexus searches all repos in the group automatically.

TypeWho writes to it?Example Use Case
HostedYour CI/CD Pipeline (Jenkins)Storing your company's proprietary payment-service.jar
ProxyNexus (fetching from internet)Caching react from npmjs.org so builds don't break if npm goes down
GroupNo one (it's read-only)The single URL developers put in their ~/.npmrc or settings.xml

5. Uploading Artifacts (Maven, npm, Docker)

How do we actually get our builds from Module 05 into Nexus?

Maven / Gradle

You configure your Nexus credentials (never commit these!) and the repository URL, then run the publish command.

bash
# Maven
mvn deploy

# Gradle
./gradlew publish
npm

You set up an .npmrc file pointing your registry to your Nexus Hosted repo, log in, and publish.

bash
# Publish to the Hosted repo
npm publish

# To download packages, developers install from the Group repo
npm install
Docker (Preview)

Nexus can host Docker registries! Because Docker requires HTTPS, this is where that Nginx reverse proxy becomes mandatory. (We will dive deep into this in Module 08).

bash
docker login nexus.mycompany.com:8082
docker tag my-app nexus.mycompany.com:8082/my-app:1.0
docker push nexus.mycompany.com:8082/my-app:1.0
⚠️ Never Commit Credentials

Do not hardcode Nexus passwords in pom.xml or package.json. They belong in environment variables or user-specific configuration files (like ~/.m2/settings.xml or ~/.npmrc) outside the Git repository.

6. Nexus REST API for Automation

A DevOps engineer automates everything. What if you are writing a deployment script and you need to ask Nexus: "Give me the latest version of the payment-service JAR"? You use the REST API.

The "Deploy Latest" Script Pattern ⭐

By combining curl and jq (from Module 02), you can query Nexus and extract the exact download URL of the newest artifact.

bash
# Search Nexus for the latest version of an artifact, parse the JSON with jq
DOWNLOAD_URL=$(curl -u admin:password -s \
  "http://localhost:8081/service/rest/v1/search/assets?repository=maven-releases&name=my-app&sort=version&direction=desc" \
  | jq -r '.items[0].downloadUrl')

# Download it
wget $DOWNLOAD_URL

This is exactly how Ansible or a bash script on a production server pulls the code before restarting the service.

7. Blob Stores & Cleanup Policies

A "Repository" is just a logical concept in Nexus. The actual physical bytes of the JAR files live on the hard drive in a Blob Store.

⚠️ Incident: Disk at 100%

The #1 incident with Artifact Repositories is running out of disk space. Developers build 1.0.0-SNAPSHOT 50 times a day. If each JAR is 100MB, that's 5GB of junk per developer, per day. If the disk fills up, Nexus crashes, and NO ONE can build or deploy code.

Cleanup Policies & Compact Blobstore

To prevent disk exhaustion, DevOps engineers create scheduled tasks in Nexus:

  • Cleanup Policy: "Delete any SNAPSHOT older than 30 days." (Note: We never delete Releases).
  • Compact Blob Store: When Nexus deletes a file, it just marks it as "soft deleted." The disk space is not actually freed until the scheduled Compact Blob Store task runs. (A classic trap for junior engineers: "I deleted 50GB of artifacts but df -h still shows 99% full!").

8. Users, Roles & Access Management

Nexus contains the keys to production. If an attacker gains write access to Nexus, they can replace payment-service.jar with a malicious version.

The Principle of Least Privilege
  • Anonymous Access: Turn this OFF on day one. No one should read or write without authenticating.
  • Developers: Assigned a read-only role. They can download dependencies, but they cannot upload to the Release repository.
  • CI/CD (Jenkins): We create a dedicated Service Account (e.g., user: jenkins-svc). It is assigned a deploy-only role. Only the automated pipeline is allowed to push to Nexus.
  • Admins: Humans never share the admin account. Nexus integrates with LDAP/Active Directory so engineers log in with their corporate credentials.
🚑

9. Backup & Restore (Disaster Recovery)

If the Nexus server catches fire, your company cannot deploy software. How do we back it up?

Nexus state is split in two: the database (metadata, users, settings) and the blob store (the actual JAR files). A proper backup script must backup both. Nexus provides an "Export Databases for Backup" scheduled task. You combine that with a nightly rsync or AWS S3 sync of the /opt/sonatype-work directory.

🛑

10. Supply Chain Security (Proxy as a Shield)

By forcing all developers and CI servers to pull internet dependencies through your Nexus Proxy Repositories, you gain a chokepoint. If a massive vulnerability is announced in log4j, an admin can log into Nexus, block the malicious version in the Proxy repo, and instantly protect the entire company from downloading it.

💡 Modern Deployment: Nexus in Docker

While installing via systemd teaches you how Linux works, modern teams deploy Nexus as a Docker container: docker run -d -p 8081:8081 -v nexus-data:/nexus-data sonatype/nexus3. (Notice the volume mount? If you don't map the data outside the container, you lose all artifacts when the container restarts. Full details in Module 08!)

🛠️ Real-Time Project: Nexus Setup & Artifact Publishing

You will provision a real server, install Nexus, configure its repositories, and publish artifacts to it from the command line.

Provision Server & Install Nexus

Provision a cloud server (or VM). Create a nexus user. Download Nexus, configure ownership, and write a systemd unit file. Start the service and verify it stays running with journalctl -u nexus.

Secure the Installation

Open port 8081 using ufw. Log into the UI using the autogenerated admin password. Immediately disable "Anonymous Access" in the security settings.

Create Roles and Users

Create two new Roles: dev-read (read access to repos) and ci-deploy (write access to hosted repos). Create two Users: a developer account and a jenkins-svc account, assigning them the respective roles.

Configure Repositories

Create a Hosted repository named maven-releases. Create a Proxy repository for Maven Central. Create a Group repository bundling them together.

Publish a Java Artifact

Take your Spring Boot app from Module 05. Add the jenkins-svc credentials to your ~/.m2/settings.xml or Gradle properties. Run ./gradlew publish and verify the JAR appears in the Nexus UI under maven-releases.

Automate with REST API

Write a Bash script that uses curl and jq to query the Nexus REST API, find the exact download URL of the latest version of your JAR, and download it.

Cleanup & Compact

Create a Cleanup Policy to delete old artifacts. Run it. Then, run the "Compact Blob Store" scheduled task. Document the disk space before and after using df -h and du -sh.

💬 Real DevOps Interview Questions

Artifact management is critical for production stability. Master these answers.

Q: Why use an artifact repository instead of Git or S3?

Git is awful at storing binaries (it stores every change, bloating the repo). S3 is just dumb storage. Artifact repositories enforce immutability (can't overwrite a release), understand language protocols (Maven/npm), and provide caching proxies for internet dependencies.

Q: Explain Hosted vs. Proxy vs. Group repositories.

Hosted: Stores our own internal, proprietary artifacts built by our CI server. Proxy: Caches external dependencies from the internet (like Maven Central) so we survive outages. Group: Combines Hosted and Proxy into a single URL for developers to use.

Q: What is the difference between SNAPSHOT and Release repos?

Releases are immutable—once a version is pushed, it can never be overwritten or deleted (except manually by an admin). SNAPSHOTs are mutable—they represent in-development code and are constantly overwritten by daily builds. We use Cleanup Policies to delete old SNAPSHOTs automatically.

Q: "The Nexus disk is 100% full." Walk me through recovery.

First, I find out what is consuming the space (usually SNAPSHOTs). I execute a Cleanup Policy to soft-delete artifacts older than 30 days. Crucially, I then manually run the Compact Blob Store scheduled task, which actually deletes the physical files from the Linux disk, freeing the space.

Q: How does a CI server (like Jenkins) authenticate to Nexus?

Via a dedicated Service Account (e.g., jenkins-svc). Following the Principle of Least Privilege, this account is assigned a role that only allows writing to specific Hosted repositories. It never uses human admin credentials.

Q: What happens when you run `npm install` and Nexus proxies npmjs?

Nexus checks its local blob store. If the requested package is there, it serves it instantly. If it's not, Nexus reaches out to the public internet (npmjs.org), downloads the package, saves a copy to its local disk, and then serves it to you. The next developer to request it gets the cached copy.

Q: How would you script "deploy the latest artifact"?

I would use curl to query the Nexus REST API (e.g., the /v1/search/assets endpoint), filter by the repository and artifact name, and sort descending by version. I would pipe the JSON response into jq to extract the downloadUrl, and then use wget to download the file.
