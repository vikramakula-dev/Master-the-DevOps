1. IaC Concepts

After Module 12, you can build an EKS cluster — but only you know how. If the cluster needs rebuilding (disaster recovery, a new environment), it's a day of console clicks with no audit trail and certain configuration drift. Infrastructure as Code solves this:

ApproachExampleProblem
Manual consoleClicking "Create VPC"Unrepeatable, undocumented, no audit trail
Imperative scriptsAWS CLI shell scriptsNot idempotent — running twice breaks things
Declarative IaCTerraform HCL✅ Idempotent, versioned, reviewable, reproducible

Terraform is declarative — you describe the desired end state, and Terraform figures out how to get there. This is the same reconciliation model as Kubernetes (M11) and ArgoCD (M17), applied to infrastructure provisioning.

🏗️

2. Terraform Architecture

The Terraform Triangle
HCL Code
(desired state) → Terraform Core
(plan + apply) → Provider
(AWS/Azure/GCP API) → Real Infrastructure
Real Infrastructure → State File
(current state) → Terraform Core
(diff = plan)

The state file (terraform.tfstate) is Terraform's memory — it maps your HCL resources to real infrastructure IDs. The plan is the diff between the desired state (HCL) and the current state (tfstate).

3. HCL Syntax

hcl — annotated main.tf
# Provider: the plugin that talks to AWS
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"   # ~> = pessimistic constraint (5.x but not 6.x)
    }
  }
  backend "s3" {          # remote state — covered in Section 5
    bucket = "techitfactory-tf-state"
    key    = "dev/terraform.tfstate"
    region = "us-east-1"
  }
}

provider "aws" {
  region = var.aws_region  # reference a variable
}

# Variable with type, description, and validation
variable "aws_region" {
  description = "AWS region to deploy into"
  type        = string
  default     = "us-east-1"
  validation {
    condition     = contains(["us-east-1", "us-west-2"], var.aws_region)
    error_message = "Unsupported region."
  }
}

# Resource: a real piece of infrastructure
resource "aws_vpc" "main" {  # type = aws_vpc, local name = main
  cidr_block           = "10.0.0.0/16"
  enable_dns_support   = true
  enable_dns_hostnames = true
  tags = {
    Name    = "techitfactory-vpc"
    Project = "techitfactory"    # cost allocation tag
  }
}

# Output: expose values after apply
output "vpc_id" {
  description = "ID of the created VPC"
  value       = aws_vpc.main.id    # reference: resource_type.local_name.attribute
}

# Data source: read existing infrastructure (not managed by this TF)
data "aws_availability_zones" "available" {
  state = "available"
}

4. Core Workflow — init → plan → apply → destroy

bash — ⭐ the four commands you'll run every day
# ⭐ init: download providers, initialize backend
terraform init

# Format and validate before planning (run in CI)
terraform fmt -check       # exits non-zero if formatting differs
terraform validate         # syntax + type checking

# ⭐ plan: show what will change — read this like a PR diff before applying
terraform plan -out=tfplan  # save the plan for deterministic apply

# ⭐ apply: converge infrastructure to the desired state
terraform apply tfplan       # applies the saved plan (no prompt)

# destroy: tear down everything (teardown checklists become one command)
terraform destroy

The plan output uses symbols you must learn to read — they mean the same as a git diff, but for infrastructure:

SymbolMeaningRisk level
+Create (new resource)Low — additive
-Destroy (delete resource)High — data loss possible
~Update in-place (modify attributes)Low to medium
-/+Destroy and recreate (immutable attribute changed)🚨 High — downtime
🚨 -/+ means downtime

-/+ is the one to fear: Terraform must destroy the resource and create a fresh one because you changed an immutable attribute (e.g., an EC2's AZ, an RDS engine version). Always read the plan before typing apply. In CI, require manual approval for any -/+ changes.

5. State — Remote Backend with S3 + DynamoDB

🚨 Never commit terraform.tfstate to Git

The state file contains plaintext secrets — database passwords, TLS private keys, anything Terraform read or created. Add *.tfstate and *.tfstate.* to .gitignore immediately (Module 03 discipline applies here). Use remote state exclusively.

bash — bootstrap the S3 + DynamoDB backend (run once)
# Create the state bucket with versioning (Module 06 backup thread: S3 edition)
aws s3 mb s3://techitfactory-tf-state --region us-east-1
aws s3api put-bucket-versioning \
    --bucket techitfactory-tf-state \
    --versioning-configuration Status=Enabled
aws s3api put-bucket-encryption \     # encrypt at rest
    --bucket techitfactory-tf-state \
    --server-side-encryption-configuration \
    '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"AES256"}}]}'

# Create DynamoDB table for state locking (prevents concurrent applies)
aws dynamodb create-table \
    --table-name techitfactory-tf-lock \
    --attribute-definitions AttributeName=LockID,AttributeType=S \
    --key-schema AttributeName=LockID,KeyType=HASH \
    --billing-mode PAY_PER_REQUEST
💡 Why locking?

Two engineers run terraform apply simultaneously. Without locking, they both read the same state, both generate plans, and then apply conflicting changes — corrupting state. DynamoDB locking ensures only one apply runs at a time: the second engineer gets "Error acquiring the state lock" and waits. This is why DynamoDB is paired with S3 for state.

6. Provisioning AWS — The Module 10 VPC in HCL

The VPC you built manually in Module 10 (nine CLI commands) becomes ~80 lines of idempotent, version-controlled HCL:

hcl — VPC, subnets, IGW, NAT, route tables
# VPC
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
  tags       = { Name = "techitfactory-vpc" }
}

# Public subnets (one per AZ)
resource "aws_subnet" "public" {
  count             = 2
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet("10.0.0.0/16", 8, count.index)  # 10.0.0.0/24, 10.0.1.0/24
  availability_zone = data.aws_availability_zones.available.names[count.index]
  map_public_ip_on_launch = true
  tags = { Name = "public-${count.index}", "kubernetes.io/role/elb" = "1" }
}

# Private subnets
resource "aws_subnet" "private" {
  count             = 2
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet("10.0.0.0/16", 8, count.index + 10) # 10.0.10.0/24, 10.0.11.0/24
  availability_zone = data.aws_availability_zones.available.names[count.index]
  tags = { Name = "private-${count.index}", "kubernetes.io/role/internal-elb" = "1" }
}

# Internet Gateway + attachment
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id
}

# NAT Gateway (one per region for dev, one per AZ for prod HA)
resource "aws_eip" "nat" { domain = "vpc" }
resource "aws_nat_gateway" "main" {
  allocation_id = aws_eip.nat.id
  subnet_id     = aws_subnet.public[0].id
  depends_on    = [aws_internet_gateway.main]  # explicit dependency
}

7. Terraform Modules

A module is a reusable directory of .tf files with defined inputs (variables) and outputs. Think of it as a function for infrastructure — the same VPC setup used in three environments without copy-paste.

hcl — calling a module
# environments/dev/main.tf — calls the vpc module
module "vpc" {
  source = "../../modules/vpc"  # local path

  vpc_cidr    = "10.0.0.0/16"
  environment = "dev"
  aws_region  = "us-east-1"
}

# Or use the community module from Terraform Registry
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.1.2"   # always pin registry modules

  name            = "techitfactory-${var.environment}"
  cidr            = "10.0.0.0/16"
  azs             = ["us-east-1a", "us-east-1b"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24"]
  enable_nat_gateway = true
}
💡 Write vs Reuse modules

For standard AWS patterns (VPC, EKS, RDS), the community modules from terraform-aws-modules are excellent and battle-tested. Write your own when you have unique requirements not covered by the community module, or when you need fine-grained control over all attributes. Don't write a VPC module from scratch unless you have a specific reason to.

10. Terraform in CI

groovy — Jenkinsfile for infrastructure PR pipeline
pipeline {
  agent any
  environment {
    TF_IN_AUTOMATION = 'true'   # suppresses interactive prompts
  }
  stages {
    stage('TF Validate') {
      steps {
        sh 'terraform init -backend=false'
        sh 'terraform fmt -check -recursive'
        sh 'terraform validate'
      }
    }
    stage('TF Plan') {
      steps {
        withCredentials([[$class: 'AmazonWebServicesCredentialsBinding',
                          credentialsId: 'aws-tf-role']]) {
          sh """
            terraform init
            terraform plan -out=tfplan -input=false
          """
        }
      }
    }
    stage('Approval') {   // Module 07 input() payoff
      when { branch 'main' }
      steps {
        input message: 'Review the Terraform plan. Apply?', ok: 'Apply'
      }
    }
    stage('TF Apply') {
      when { branch 'main' }
      steps {
        sh 'terraform apply -input=false tfplan'
      }
    }
  }
}

11. Real-Time Incidents & Troubleshooting

Incident 1 — State Lock Stuck After Killed Run

bash — diagnose and force-unlock
# Error: Error acquiring the state lock
# Lock Info: ID: 12345-... Operation: OperationTypeApply
# The previous apply was Ctrl+C'd — DynamoDB lock item wasn't released

# Verify: check if anyone else is running TF
# If you're certain the previous apply is dead:
terraform force-unlock 12345-abcd-...  # the lock ID from the error message

# Then re-run plan to see current state before applying again
terraform plan

Incident 2 — Drift (Someone Clicked in the Console)

StageDetails
Symptomterraform plan shows changes you didn't make — e.g., a Security Group rule was added manually.
Decision tree(1) The manual change is wrong → terraform apply to revert drift. (2) The manual change is intentional → terraform import the resource and update the HCL to match, then apply (idempotent). (3) Temporarily exclude from TF → lifecycle { ignore_changes = [tags] }.
PreventionIAM deny policy for console changes to tagged resources (all infra tagged ManagedBy=Terraform); enforce via SCP. At minimum, CloudTrail alerts on console changes to production resources.

Incident 3 — The Accidental Destroy

hcl — prevent_destroy lifecycle guard
# Never accidentally delete the production database:
resource "aws_db_instance" "prod" {
  identifier = "techitfactory-prod"
  # ... config ...
  lifecycle {
    prevent_destroy = true   # terraform destroy will error out rather than delete this
  }
}

# Also: save the plan file and review before applying in CI (Section 10)
# terraform plan -out=tfplan → human reviews → terraform apply tfplan
# The plan file is deterministic — apply does exactly what plan showed

Interview Questions

Click a card to reveal a strong candidate answer.

Q: What is Terraform state and why must it be remote and locked?

State is Terraform's record of what infrastructure it manages and the mapping from HCL resource names to real resource IDs (EC2 instance IDs, ARNs, etc.). Without it, Terraform can't know what already exists. Remote state (S3) means the team shares one source of truth instead of each engineer having a local state that diverges. Locking (DynamoDB) prevents concurrent applies from corrupting the state when two engineers or CI jobs run simultaneously. The state file also contains secrets in plaintext — S3 encryption + access control are mandatory.

Q: What's the difference between terraform plan and terraform apply?

Plan computes the diff between desired state (your HCL) and current state (tfstate + a real API refresh) and shows what would change — no side effects. Apply executes the changes. The best practice is to save the plan (-out=tfplan) so that when you apply, you're applying exactly what was reviewed, not a potentially different plan computed at apply time. In CI, plan runs on every PR (humans review it), apply only runs on merge to main after an approval gate.

Q: How do you design Terraform modules?

A module should have a single, clear responsibility — a VPC module creates the VPC and networking; an EKS module creates the cluster. It takes inputs (variables with types and descriptions) and produces outputs that other modules can reference. I avoid modules that do too many things — they become hard to test and reuse. For standard patterns, I use community modules from the Terraform registry (terraform-aws-modules) as a base and wrap them if I need additional standardization. I pin module versions the same way I pin Docker image tags.

Q: How do you manage multiple environments with Terraform?

I use a directory-per-environment pattern over workspaces. Each environment has its own directory with its own backend config (separate state bucket key), its own tfvars, and its own state. This makes blast radius explicit — a terraform destroy in environments/dev/ can't touch production. Workspaces share a backend config which makes it easy to accidentally target the wrong environment. The tradeoff is more code duplication, which I mitigate by extracting shared infrastructure into modules.

Q: How do you handle drift — when someone manually changes infrastructure?

Terraform plan shows drift as proposed changes — it refreshes state against the real API before comparing. My response depends on intent: if the manual change was wrong, I apply to revert. If it was intentional, I either codify it in HCL and apply (idempotent — no change needed), or use terraform import if the resource was created outside TF. Prevention: enforce IaC via IAM deny policies on direct resource modification (SCP in Organizations), and alert on CloudTrail events for console changes to tagged infra.

Q: What does -/+ in a Terraform plan mean, and why is it dangerous?

-/+ means destroy and recreate — Terraform must delete the resource and create a new one because you changed an immutable attribute (one that can't be updated in-place). For an RDS instance, this means a new database with no data (unless you restore from snapshot). For an EKS node group, it means draining nodes and creating new ones — brief capacity reduction. Always flag -/+ for human review in CI and schedule it as a maintenance window. prevent_destroy = true in the resource lifecycle blocks accidental destruction of critical resources.
