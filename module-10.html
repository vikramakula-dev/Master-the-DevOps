1. Cloud Computing — IaaS, PaaS, SaaS

Cloud computing is renting compute, storage, and networking from a datacenter that someone else operates — by the second, via API. The three service models define how much of the stack they manage for you:

ModelAWS ExampleYou manageAWS manages
IaaS — InfrastructureEC2, VPCOS, runtime, app, dataPhysical servers, network hardware, facilities
PaaS — PlatformElastic Beanstalk, RDSApp code & dataOS, runtime, patching, scaling
SaaS — SoftwareWorkMail, ChimeYour data & configEverything including the application

This roadmap lives in IaaS — you control the OS just like your local VMs, but AWS handles the datacenter. The shift from CapEx (buying servers) to OpEx (paying per hour) is why startups can scale globally before they're profitable.

💡 Why AWS?

AWS has the largest market share (~32%), the deepest service catalog, and the most DevOps job postings. The concepts transfer to Azure/GCP — the service names change, the networking model doesn't. Learn one deeply.

2. Regions & Availability Zones

AWS divides the world into Regions — geographic areas like us-east-1 (N. Virginia) or eu-west-1 (Ireland). Each region contains multiple Availability Zones (AZs) — physically separate datacenters connected by low-latency fiber. AZs within a region share the same IP address space but have independent power, cooling, and networking.

Region → AZ → Resource hierarchy
Region: us-east-1 → AZ: us-east-1a → Subnet: 10.0.1.0/24 → EC2 instance

The capstone deploys across two AZs. If us-east-1a loses power, us-east-1b keeps serving traffic. Single-AZ deployments are development shortcuts — production always spans at least two.

⚠️ Data residency matters

Regulated data (HIPAA, GDPR) must stay in specific regions. Choose your region before building — migrating later is painful. us-east-1 is cheapest and gets new services first; choose based on user location and compliance.

3. IAM — Identity & Access Management

IAM is the security backbone of AWS. Before you create a single EC2 instance, you lock down IAM. It has four components:

EntityWhat it isUse case
UserA person or application with long-term credentialsYour developer login
GroupCollection of users sharing the same policies"developers" group → ReadEC2 policy
RoleTemporary credentials assumed by services or usersEC2 instance role to access S3
PolicyJSON document listing allowed/denied actionsAmazonEC2ReadOnlyAccess

Policies are the JSON rules. Here's the ECR pull policy you already saw in Module 09, fully annotated:

json — IAM Policy (annotated)
// Minimal policy: allow an EC2 to pull images from ECR (no keys needed)
{
  "Version": "2012-10-17",       // always this literal string
  "Statement": [
    {
      "Effect": "Allow",           // Allow | Deny
      "Action": [
        "ecr:GetAuthorizationToken",  // get a temp docker login token
        "ecr:BatchGetImage",          // pull images
        "ecr:GetDownloadUrlForLayer"  // pull image layers
      ],
      "Resource": "*"             // GetAuthorizationToken requires *
    }
  ]
}
⭐ Root account lockdown ritual (do this on day 1)

1. Enable MFA on root. 2. Create an admin IAM user — use that instead. 3. Delete root access keys. 4. Never use root for daily work. Root can do things that can't be undone — like deleting the account.

Roles vs Access Keys: Access keys (a key ID + secret) are long-lived credentials that can be leaked. Roles issue short-lived tokens (15 min – 12 h) that auto-rotate. Whenever an AWS service (EC2, Lambda, ECS) needs to call other AWS services, use a role — never put access keys in your code or user-data. This is the least-privilege thread from Modules 06, 07, and 09 at the AWS level.

bash — debug "access denied" (⭐ daily use)
aws sts get-caller-identity        # who am I right now? (account, user/role, ARN)
{
    "UserId": "AIDA...",
    "Account": "123456789012",
    "Arn": "arn:aws:iam::123456789012:user/vikram"
}

# Then use the IAM Policy Simulator in the console to test exactly what a
# user/role can do — paste the ARN, choose the action, see Allow/Deny + reason

4. EC2 — Virtual Machines in AWS

EC2 (Elastic Compute Cloud) is a rented virtual machine. The instance type encodes hardware: t3.micro = family t (burstable), generation 3, size micro (2 vCPU, 1 GB RAM). For the capstone, t3.small is the minimum; m5.large for production API servers.

FamilyOptimized forExample use
t3/t4gBurstable general-purposeDev/staging, low-traffic apps
m6i/m7gBalanced compute/memoryApp servers, API backends
c6i/c7gCompute-optimizedCI agents, batch processing
r6i/r7gMemory-optimizedDatabases, in-memory caches

Key EC2 concepts you'll touch immediately:

bash — launch, connect, and query EC2 via CLI
# Launch an instance (uses the same SSH key pair as Module 02)
aws ec2 run-instances \
    --image-id ami-0c02fb55956c7d316 \    # Amazon Linux 2 AMI (us-east-1)
    --instance-type t3.micro \
    --key-name my-keypair \               # Module 02 SSH key, now in AWS
    --security-group-ids sg-0abc123 \
    --subnet-id subnet-0def456 \
    --iam-instance-profile Name=ECRPullRole \  # role, not keys!
    --user-data file://server-setup.sh \  # Module 02's script runs at first boot
    --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=web-1}]'

# ⭐ List running instances with their private IPs
aws ec2 describe-instances \
    --filters "Name=instance-state-name,Values=running" \
    --query 'Reservations[*].Instances[*].[InstanceId,PrivateIpAddress,Tags[?Key==`Name`].Value|[0]]' \
    --output table

# SSH using the key pair (Module 02 payoff)
ssh -i ~/.ssh/my-keypair.pem ec2-user@<public-ip>
⚠️ Security Groups are stateful firewalls

A Security Group is the M04 stateful firewall table — return traffic is automatically allowed. Inbound rules: 22 from your IP (SSH), 80/443 from 0.0.0.0/0 (web). Outbound: allow all (default). NACLs are stateless — you need both directions. When you can't connect, check SG → NACL → route table → key pair — in that order (the Module 04 layered-debug ladder, AWS edition).

5. VPC — Your Private Network

A VPC (Virtual Private Cloud) is a logically isolated network inside AWS — Module 04's subnetting, routing, and firewall concepts, implemented in software. The capstone VPC is 10.0.0.0/16 — 65,536 addresses, carved into public and private subnets across two AZs.

Capstone VPC — built by hand, later automated by Terraform (M13)
Internet → IGW → Public Subnets
10.0.1.0/24 (1a)
10.0.2.0/24 (1b) → ALB
ALB → Private Subnets
10.0.3.0/24 (1a)
10.0.4.0/24 (1b) → EC2 (app) → NAT GW → Internet
bash — build the VPC step by step
# 1. Create VPC (Module 04 CIDR payoff — decode: 10.0.0.0 with 16-bit prefix)
VPC_ID=$(aws ec2 create-vpc --cidr-block 10.0.0.0/16 \
    --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=techitfactory-vpc}]' \
    --query 'Vpc.VpcId' --output text)

# 2. Create public subnets (one per AZ)
PUB1=$(aws ec2 create-subnet --vpc-id $VPC_ID \
    --cidr-block 10.0.1.0/24 --availability-zone us-east-1a \
    --query 'Subnet.SubnetId' --output text)
PUB2=$(aws ec2 create-subnet --vpc-id $VPC_ID \
    --cidr-block 10.0.2.0/24 --availability-zone us-east-1b \
    --query 'Subnet.SubnetId' --output text)

# 3. Create private subnets
PRIV1=$(aws ec2 create-subnet --vpc-id $VPC_ID \
    --cidr-block 10.0.3.0/24 --availability-zone us-east-1a \
    --query 'Subnet.SubnetId' --output text)
PRIV2=$(aws ec2 create-subnet --vpc-id $VPC_ID \
    --cidr-block 10.0.4.0/24 --availability-zone us-east-1b \
    --query 'Subnet.SubnetId' --output text)

# 4. Internet Gateway (the VPC's default route to the internet)
IGW=$(aws ec2 create-internet-gateway --query 'InternetGateway.InternetGatewayId' --output text)
aws ec2 attach-internet-gateway --internet-gateway-id $IGW --vpc-id $VPC_ID

# 5. Elastic IP + NAT Gateway (private subnets route outbound through this)
EIP=$(aws ec2 allocate-address --domain vpc --query 'AllocationId' --output text)
NAT=$(aws ec2 create-nat-gateway --subnet-id $PUB1 \
    --allocation-id $EIP --query 'NatGateway.NatGatewayId' --output text)
aws ec2 wait nat-gateway-available --nat-gateway-ids $NAT

# 6. Route tables — public: 0.0.0.0/0 → IGW; private: 0.0.0.0/0 → NAT
RT_PUB=$(aws ec2 create-route-table --vpc-id $VPC_ID --query 'RouteTable.RouteTableId' --output text)
aws ec2 create-route --route-table-id $RT_PUB --destination-cidr-block 0.0.0.0/0 --gateway-id $IGW
aws ec2 associate-route-table --route-table-id $RT_PUB --subnet-id $PUB1
aws ec2 associate-route-table --route-table-id $RT_PUB --subnet-id $PUB2
💡 Module 04 Payoff

Every CIDR, subnet mask, and routing concept from Module 04 just landed. The public subnets have a route to the IGW — their instances can have public IPs and reach the internet directly. The private subnets route through NAT — their instances can pull updates outbound but have no public IP and can't be reached inbound. That's the security model.

6. S3 — Object Storage

S3 (Simple Storage Service) stores files as objects inside buckets. There's no filesystem hierarchy — the slash in images/logo.png is just part of the key name. Objects can be up to 5 TB; buckets are effectively unlimited.

bash — S3 daily commands (⭐)
# Create a bucket (globally unique name — prefix with your account/project)
aws s3 mb s3://techitfactory-assets-123456 --region us-east-1

# ⭐ Sync a local directory to S3 (one-way mirror, skips unchanged files)
aws s3 sync ./dist/ s3://techitfactory-assets-123456/static/ \
    --delete                   # remove S3 objects not in ./dist/

# Enable versioning (Module 06 Nexus cleanup-policy analogy — keep N versions)
aws s3api put-bucket-versioning \
    --bucket techitfactory-assets-123456 \
    --versioning-configuration Status=Enabled

# Lifecycle policy: move to Glacier after 30d, delete after 365d
aws s3api put-bucket-lifecycle-configuration \
    --bucket techitfactory-assets-123456 \
    --lifecycle-configuration file://lifecycle.json

# Generate a presigned URL (time-limited, no auth needed by the requester)
aws s3 presign s3://techitfactory-assets-123456/report.pdf --expires-in 3600
🚨 Never make a bucket public without intent

In 2023, thousands of S3 breaches were "public access" misconfigurations. AWS now blocks public access by default at the account level. For static site hosting, use CloudFront in front of a private bucket — never flip the public-access block off globally. If you need a file public, use a presigned URL.

7. ECR — Elastic Container Registry

You met ECR in Module 09 as the push destination. Now the full picture: ECR is a managed Docker registry, private by default, integrated with IAM (no separate credentials once your instance or CI role has the ECR policy), and capable of scanning images for CVEs on every push (we'll act on those alerts in Module 18).

bash — ECR full workflow
# Create repository
aws ecr create-repository \
    --repository-name techitfactory/api \
    --image-scanning-configuration scanOnPush=true \  # Module 18 teaser
    --region us-east-1

# ⭐ Log Docker into ECR (token valid 12 hours)
aws ecr get-login-password --region us-east-1 | \
    docker login --username AWS \

# Tag and push (Module 09 tags land here)
docker tag techitfactory/api:1.4.2 \

docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/techitfactory/api:1.4.2

# Lifecycle rule: keep only the 10 most recent images per tag prefix
aws ecr put-lifecycle-policy \
    --repository-name techitfactory/api \
    --lifecycle-policy-text file://ecr-lifecycle.json

# Create a target group (instances the ALB forwards to)
TG_ARN=$(aws elbv2 create-target-group \
    --name techitfactory-tg \
    --protocol HTTP --port 3000 \
    --vpc-id $VPC_ID \
    --health-check-path /health \        # Module 09 smoke-test endpoint
    --health-check-interval-seconds 30 \
    --healthy-threshold-count 2 \
    --query 'TargetGroups[0].TargetGroupArn' --output text)

# Register instances into the target group
aws elbv2 register-targets \
    --target-group-arn $TG_ARN \
    --targets Id=$INSTANCE_ID

# Create the ALB in the public subnets
ALB_ARN=$(aws elbv2 create-load-balancer \
    --name techitfactory-alb \
    --subnets $PUB1 $PUB2 \
    --security-groups $ALB_SG \
    --query 'LoadBalancers[0].LoadBalancerArn' --output text)

# Set up a billing alarm (prevents the Module 10 incident #1)
aws cloudwatch put-metric-alarm \
    --alarm-name BillingAlert50USD \

    --statistic Maximum --period 86400 \
    --threshold 50 --comparison-operator GreaterThanThreshold \

10. Real-Time Incidents & Troubleshooting

Incident 1 — Surprise Bill: Forgotten NAT Gateway + Unattached EIPs

StageDetails
SymptomAWS bill is $80 when you expected $10. Nothing is "running" in the console.
InvestigationAWS Cost Explorer → group by Service. NAT Gateway: $32 (idle hourly fee). EC2 → Elastic IPs: $7.20 (unattached EIPs bill by the hour). Check CloudTrail for what was created and never destroyed.
Root CauseNAT Gateways charge ~$0.045/hr even with zero traffic. EIPs charge when not attached to a running instance. A VPC teardown that misses these resources keeps billing silently.
FixDelete the NAT GW + release the EIP. In Cost Explorer, filter last 30 days to confirm charge stops.
PreventionCreate a billing budget: Console → Billing → Budgets → $20/month alert. Tag every resource (Project=techitfactory). Create a teardown checklist: EIP → NAT GW → IGW → subnets → VPC (in that order).

Incident 2 — "Can't SSH to Instance"

StageDetails
Symptomssh: connect to host ... port 22: Connection timed out
InvestigationFollow the Module 04 layered debug ladder: (1) SG inbound rule — port 22 from your IP? (2) NACL — stateless, needs both inbound 22 and outbound ephemeral ports 1024-65535. (3) Route table — does the public subnet route to the IGW? (4) Does the instance have a public IP? (5) Is the instance in a public subnet?
Root CauseInstance launched in a private subnet (no route to IGW). Private instances aren't reachable directly — use a bastion host or AWS SSM Session Manager.
FixUse AWS Systems Manager Session Manager (no SSH, no key, no open port 22 needed): aws ssm start-session --target i-0abc123
PreventionPrefer SSM over SSH — no open port 22, all sessions logged to CloudWatch. Only open port 22 to a bastion, never to 0.0.0.0/0.

Interview Questions

Click a card to reveal a strong candidate answer.

Q: What's the difference between a Region and an Availability Zone?

A Region is a geographic area containing multiple, isolated datacenters called Availability Zones. AZs within a region are connected by low-latency, high-bandwidth fiber but have completely independent power, cooling, and networking — so a fire in us-east-1a doesn't affect us-east-1b. For production, you always span at least two AZs. Regions are chosen for user proximity and data residency requirements.

Q: What's the difference between a Security Group and a NACL?

Security Groups are stateful and operate at the instance level — if you allow port 443 inbound, the return traffic is automatically allowed. NACLs are stateless and operate at the subnet level — you must explicitly allow both directions. SGs have only allow rules (deny by default); NACLs can have explicit deny rules and are evaluated in numbered order. For most use cases, Security Groups are sufficient — NACLs add a second layer for compliance requirements.

Q: When would you use an IAM Role instead of an IAM User with access keys?

Always prefer roles for AWS services. Roles issue short-lived credentials that auto-rotate — no credentials to rotate, lose, or accidentally commit to Git. An EC2 instance with an instance profile (role attached) can call S3 or ECR without a single access key anywhere. Access keys are only for humans logging in from outside AWS, and even then you should use SSO or federated access. The rule: if it's running inside AWS, it gets a role.

Q: Why put app servers in private subnets with a NAT Gateway instead of public subnets?

Private subnets have no route from the internet — a scanner can't reach your app servers directly even if port 22 is accidentally open. The ALB in the public subnet is the only inbound path, which narrows your attack surface to a managed service. The NAT Gateway allows outbound traffic (package updates, ECR pulls) without exposing the instance to inbound connections. Compliance frameworks like PCI-DSS explicitly require this architecture.

Q: How does an ALB health check work?

The ALB sends periodic HTTP(S) requests to a configured path (e.g., /health) on each target. If a target returns a 2xx response within the timeout and does so for a configured number of consecutive checks (healthy threshold), it becomes healthy and receives traffic. If it fails enough consecutive checks (unhealthy threshold), the ALB removes it from rotation silently — no traffic reaches a broken instance. This is why exposing a /health endpoint that verifies the app is genuinely ready (DB connection, cache, etc.) is critical.

Q: How do you debug an "Access Denied" error in AWS?

First: aws sts get-caller-identity — confirm you're authenticated as who you think. Then use the IAM Policy Simulator to test the exact action. Check for explicit Deny (Deny wins over Allow) in: attached policies, inline policies, and if using AWS Organizations, SCPs. For S3, also check the resource-based bucket policy — bucket policies can deny even if the user policy allows. Finally, CloudTrail logs the exact denied event with the error code, which is the ground truth.

Q: What's the difference between S3 versioning and lifecycle policies?

Versioning keeps every version of every object — a delete creates a delete marker, and you can restore any previous version. It's your protection against accidental overwrites. Lifecycle policies act on object age and version count — you might say "move to S3-IA after 30 days, Glacier after 90, delete after 365." They're cost management tools. You typically use both: versioning for safety, lifecycle policies to prevent versioning from becoming a storage bill.
