1. Why Containers Exist (The "Works on My Machine" Problem)

In Module 05, we built a Java app and a Node.js app. What happens if Developer A has Java 11 installed, but the Production server has Java 17? Or if two Node.js apps on the same server need conflicting versions of node_modules? The app crashes in production with the classic excuse: "Well, it works on my machine!"

Containers solve this by packaging the code and the entire environment (OS libraries, Java/Node runtime, dependencies) into a single, immutable box.

FeatureVirtual Machines (VMware/EC2)Containers (Docker)
ArchitectureHardware → Host OS → Hypervisor → Guest OS → AppHardware → Host OS → Docker Engine → App
IsolationFull hardware virtualization (heavy)Linux Namespaces & cgroups (lightweight)
Boot TimeMinutes (booting an entire OS)Milliseconds (starting a process)
SizeGigabytesMegabytes
💡 The Matrix Reality

A container is not a tiny VM. It is just a regular Linux process that has been lied to. Linux Namespaces restrict what the process can see (it thinks it's the only process on the machine). Linux cgroups restrict what it can use (it thinks the machine only has 512MB RAM). If you run ps aux on the Docker host, you can see the containerized process running right alongside your system services!

2. What is Docker, Exactly? (Docker vs Container vs Kubernetes)

Docker is a platform for building, shipping, and running containers. That single sentence hides an important distinction people mix up constantly: the container is the technology, Docker is the tool that made it usable, and Kubernetes is the system that runs containers at scale. Let's untangle all three.

Docker Is Actually Four Things in One Box
PieceWhat It IsYou Use It When…
Docker Engine (dockerd)The daemon that creates and supervises containers on a hostAlways — it does the actual work
Docker CLIThe docker command — a client that talks to the daemon over a socketdocker run, docker build, docker ps…
Image formatThe layered, portable package standard (now the open OCI standard)Every docker build produces one
Registry ecosystemDocker Hub + private registries (Nexus, ECR) that store and distribute imagesdocker push / docker pull
The Lifecycle in One Line
Dockerfile (recipe) → docker build → Image (frozen package) → docker run → Container (running process)
Why Docker Won — The Shipping Container Story

Containers (Linux namespaces + cgroups) existed before Docker — LXC, Solaris Zones, FreeBSD jails. What Docker added in 2013 was standardization and ergonomics: one file format (the Dockerfile), one package format (the image), one command to run anything, anywhere. It's the same revolution as the physical shipping container: goods (apps) always existed, but a standard box meant every crane, ship, and port (every server, cloud, and CI system) could handle any cargo without knowing what's inside.

Docker vs Kubernetes, Head to Head

They are not competitors — Docker builds the image; Kubernetes runs it in production. The comparison that matters is Docker/Compose on one host vs Kubernetes on a cluster:

💡 The Interview Trap: "Docker vs Kubernetes — which is better?"

Wrong question, and interviewers ask it to see if you know that. The senior answer: they're complementary layers — Docker (build-time) produces OCI images; Kubernetes (run-time) orchestrates them. Bonus point that proves depth: Kubernetes actually removed Docker as its runtime (dockershim, v1.24) and uses containerd directly — yet every Docker-built image still runs unchanged, because the OCI image standard is the real contract, not Docker itself.

⚠️ Terms You'll Hear (Quick Decoder)

containerd — the lower-level runtime Docker itself uses internally (and K8s uses directly). OCI — the open standard for images & runtimes; why images run anywhere. Podman — a daemonless Docker alternative, CLI-compatible. Docker Desktop — the Mac/Windows GUI app (runs a Linux VM under the hood — containers are always Linux processes). Docker Swarm — Docker's own orchestrator; simpler than K8s but the industry chose Kubernetes.

3. Docker Architecture

Docker Client (CLI) ↔ Docker Daemon (dockerd) ↔ Registry (Docker Hub / Nexus)

Image vs. Container: Think of an Image as a Class (the blueprint, stored on disk), and a Container as an Object (a running instance of that image in RAM). You can spin up 100 containers from 1 image.

Security Warning (Module 02 Callback)

When you install Docker on Ubuntu, it creates a docker user group. Adding your user to this group allows you to run docker commands without sudo. WARNING: Anyone in the docker group effectively has root access to the host server, because they can mount the host's root filesystem into a container.

4. First Commands, Step-by-Step

Let's start an Nginx web server. Notice how we map port 8080 on the host to port 80 inside the container (Module 04 payoff).

bash
# Run Nginx in the background (-d), mapping ports (-p host:container)
docker run -d -p 8080:80 --name my-web nginx

# See what is running
docker ps

# Follow the logs (The most important debugging command!) ⭐
docker logs -f my-web

# "SSH" into the running container (Technically, spawn a bash process inside its namespace) ⭐
docker exec -it my-web bash

# Kill and remove it
docker stop my-web
docker rm my-web

5. Images — Pull, Tag, Push

How do images travel from your laptop to production? Through a Registry (like Docker Hub, AWS ECR, or our Nexus server from Module 06).

Image naming syntax: registry_url/repository:tag

bash
# Pull an image
docker pull ubuntu:22.04

# Rename (tag) it for our private Nexus registry
docker tag ubuntu:22.04 nexus.mycompany.com:8082/my-ubuntu:v1

# Push it (Requires `docker login` first)
docker push nexus.mycompany.com:8082/my-ubuntu:v1
⚠️ The `latest` Trap

The tag :latest does NOT mean "the newest version." It is just a default string applied if you don't specify a tag. If a developer pushes a breaking change tagged as latest, and your prod server pulls it, production goes down. In DevOps, we ALWAYS use specific, immutable tags (like the Git commit SHA from Module 05).

6. Dockerfile, Line by Line

A Dockerfile is the script used to build an image. It executes from top to bottom. Each line creates a new "Layer", which is cached.

Example 1: Node.js (Layer Caching)
Dockerfile (Node)
# 1. Base image (Alpine is a tiny 5MB Linux distro)
FROM node:20-alpine

# 2. Set the working directory inside the container
WORKDIR /app

# 3. COPY package.json FIRST. Why? Docker caches layers. 
# If we copied the whole codebase first, any code change would bust the npm install cache!
COPY package*.json ./
RUN npm ci

# 4. Now copy the rest of the application code
COPY . .

# 5. Security: Don't run as root!
USER node

# 6. Document the port (does not actually publish it)
EXPOSE 3000

# 7. The command to run when the container starts
CMD ["node", "server.js"]
Code Explanation
  • FROM node:20-alpine: Defines the foundation of the image. "Alpine" is a security-focused, tiny Linux distribution (~5MB). We use it to keep our container small and reduce the attack surface.
  • WORKDIR /app: Creates a directory named /app inside the container and CD's into it. All subsequent COPY and RUN commands happen here.
  • COPY package*.json ./: Copies only the dependency definitions first. Docker caches this layer! If we don't change package.json, Docker skips the npm install on the next build, saving minutes of build time.
  • RUN npm ci: Runs a clean install of our Node dependencies inside the image.
  • COPY . .: Copies the rest of our application code (index.js, etc.) from our host laptop into the image.
  • USER node: Switches from the root user to a limited user named node. If a hacker breaches our app, they won't have root permissions to destroy the container.
InstructionBehavior
CMDProvides a default command. Can be easily overridden at runtime (e.g., docker run my-image bash overrides CMD).
ENTRYPOINTHardcodes the executable. Arguments passed at runtime are appended to it. (e.g., if ENTRYPOINT is ping, docker run my-image google.com runs ping google.com).
Example 2: Java Multi-Stage Build

A JDK (Java Development Kit) is 500MB+. A JRE (Java Runtime Environment) is 50MB+. You need the JDK to build the code, but only the JRE to run it. Multi-stage builds solve this.

Dockerfile (Java)
# Stage 1: Build (Huge image)
FROM eclipse-temurin:17-jdk-alpine AS builder
WORKDIR /app
COPY . .
RUN ./gradlew build

# Stage 2: Run (Tiny image)
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
# Copy ONLY the compiled JAR from Stage 1. Discard the rest of the 500MB build environment!
COPY --from=builder /app/build/libs/app.jar .
CMD ["java", "-jar", "app.jar"]
Code Explanation
  • AS builder: We name this first stage "builder". It contains the heavy 500MB JDK required to run Gradle.
  • COPY --from=builder ...: The magic of multi-stage builds. In Stage 2, we grab the single `.jar` file we compiled in Stage 1 and move it into our new, tiny JRE image. The 500MB JDK from Stage 1 is completely thrown away.
  • CMD ["java", "-jar", "app.jar"]: The default command executed when a container spins up from this final image.

7. Docker Networking

The biggest beginner trap: inside a container, localhost means the container itself, NOT your Mac/Windows host. To let containers talk to each other, we put them on a Custom Bridge Network.

Custom Network: "my-net"

Container A (name: web) ↔ Container B (name: db)

⭐ The Magic Feature: On a custom network, Docker provides built-in DNS. Container A can connect to Container B simply by pinging http://db:5432. Docker resolves the name to the correct internal IP automatically.

8. Volumes & Persistence

Containers are ephemeral (cattle, not pets). If you run a PostgreSQL container, write 10 rows of data, and then docker rm the container... your data is gone forever.

To persist data beyond the container's lifecycle, we use Volumes.

TypeSyntaxUse Case
Named Volume-v my-data:/var/lib/postgresql/dataProduction. Managed by Docker (lives in /var/lib/docker/volumes/). Fast and safe.
Bind Mount-v /Users/vikra/code:/appDevelopment. Mounts your laptop folder into the container so live-reloads work when you save a file.

9. Docker Compose

Typing docker run with 15 flags for the web app, and another 15 flags for the database, is tedious. Docker Compose allows you to define the entire stack in a single YAML file.

docker-compose.yml
version: '3.8'

services:
  database:
    image: mongo:6
    volumes:
      - mongo_data:/data/db
    networks:
      - backend-net

  api:
    build: ./backend        # Builds the Dockerfile in this folder
    ports:
      - "3000:3000"         # Host:Container
    env_file: .env          # Inject secrets without hardcoding them in YAML
    depends_on:
      - database            # Starts DB before API
    networks:
      - backend-net

volumes:
  mongo_data:               # Declares the named volume

networks:
  backend-net:              # Declares the custom bridge network
Code Explanation
  • services:: The core block where you define your containers (the database and the API).
  • image: mongo:6: Tells Compose to pull a pre-built image from Docker Hub instead of building it from scratch.
  • build: ./backend: Tells Compose to look in the ./backend folder for a Dockerfile and build it on the fly.
  • env_file: .env: Loads secrets from a `.env` file that is in `.gitignore` (M03 callback). This keeps database passwords out of your source code repository!
  • depends_on:: Ensures the Database container starts before the API container. (Note: It only waits for the container to start, not for the database to be "ready" to accept connections. For that, use a healthcheck).
  • volumes: / networks:: Explicitly declares the Named Volume and Custom Network so that Compose can create them if they don't exist.
The Compose Daily Loop ⭐
  • docker compose up -d (Start all services in background)
  • docker compose logs -f api (Watch the API logs)
  • docker compose down (Stop and remove containers)
  • docker compose down -v (⚠️ DANGER: Removes containers AND deletes the named volumes/data)
✅

10. Best Practices & Optimization

A poorly built container is a security risk and costs money in cloud bandwidth. The production checklist:

  • Small Base Images: Use alpine or distroless to shrink images from 1GB to 50MB. Faster pulls, smaller attack surface.
  • Non-Root User: Always declare USER node or USER appuser. If a hacker breaches your app, they are trapped as an unprivileged user inside the container.
  • One Process per Container: Don't run Nginx and Node in the same container. Run two containers. If Nginx crashes, it shouldn't take down Node.
  • Read-Only Filesystems: Run with --read-only. If an attacker gets in, they can't download malware to the disk.
  • Resource Limits: Run with --memory="512m" --cpus="1.0". This prevents a memory leak in one container from crashing the entire host server (a preview of Kubernetes Limits).

11. Under the Hood (How it actually works)

If you want to pass a Senior DevOps interview, you need to know how Docker fakes a computer.

OverlayFS (Copy-on-Write)

When you pull an image, Docker downloads it in read-only layers. When you run a container, Docker slaps a thin, read-write layer on top of it. If your app modifies a file, Docker copies that file from the read-only layer up into the read-write layer, and modifies it there. This is Copy-on-Write. This is why you can run 100 containers from a 1GB image, and it only consumes 1GB of disk space total!

Namespaces

If you run docker inspect my-container | grep Pid, you will get the actual Linux Process ID (e.g., PID 4512). If you run ls /proc/4512/ns on the host, you will see the isolated namespaces (network, mount, pid) that make this process think it's a container.

12. Real-Time Incidents & Troubleshooting

Incident 1: Container Exits Immediately

Symptom: You run docker-compose up -d, but docker ps shows nothing. docker ps -a shows the container Exited (137).

Investigation: Exit Code 137 means "Killed by signal 9 (SIGKILL)". Usually, this is the Linux OOM (Out of Memory) killer.

Fix: Check dmesg -T | grep -i oom (Module 02 callback!). Increase the memory limit in Compose or the VM.

Incident 2: "Cannot connect to database"

Symptom: The API logs say MongoNetworkError: failed to connect to server [localhost:27017].

Investigation: The API code is hardcoded to connect to localhost. But inside the API container, localhost is itself! It is not looking at the database container.

Fix: Change the code (or Env Var) to connect to mongodb://database:27017, utilizing Docker's built-in DNS.

Incident 3: Host Disk is 100% Full

Symptom: Everything crashes. df -h shows /var/lib/docker is consuming 100GB.

Investigation: Run docker system df. You will see Gigabytes of "Dangling Images" (untagged images from old builds) and stopped containers.

Fix: Run docker system prune -a --volumes. Prevention: Setup log rotation in daemon.json (max-size: 10m) and run prune via cron weekly.

Incident 4: Build is painfully slow

Symptom: docker build takes 5 minutes every time you change 1 line of CSS.

Investigation: You put COPY . . before RUN npm install. Because the CSS file changed, Docker busts the cache for the COPY layer, which forces it to re-run the `npm install` layer every time!

Fix: Reorder the Dockerfile. Copy package.json first, run install, THEN copy the code.

🛠️ Real-Time Project: Dockerize a Full-Stack App

You will take the raw Node.js + React + MongoDB application from Module 05, write optimal Dockerfiles, and orchestrate them with Compose.

Install Docker Securely

Install Docker on your M02 Ubuntu VM. Add your user to the docker group so you don't need `sudo` (remembering the root-equivalence security warning).

bash
sudo apt update && sudo apt install -y docker.io docker-compose
sudo usermod -aG docker $USER
# Log out and log back in for group changes to take effect
docker run hello-world

Dockerize the Node Backend (Optimized)

Write an optimized single-stage Dockerfile for the backend: use alpine, implement the COPY package.json caching trick, and add a non-root USER node.

backend/Dockerfile
FROM node:20-alpine
WORKDIR /usr/src/app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
# Secure the container by running as a non-root user
USER node
EXPOSE 5000
CMD ["node", "index.js"]

Build it: docker build -t my-backend:v1 ./backend

Dockerize React (Multi-stage)

Write a multi-stage build for the frontend. Stage 1 (Node) runs npm run build. Stage 2 (Nginx) copies the static files from Stage 1 into /usr/share/nginx/html (M04 payoff!).

frontend/Dockerfile
# Stage 1: Build the React static files
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Serve with Nginx
FROM nginx:alpine
# Copy ONLY the static build output. Discard the heavy Node image!
COPY --from=builder /app/build /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Build it: docker build -t my-frontend:v1 ./frontend

Write the Compose File

Create docker-compose.yml at the root of your project. Define Mongo DB (with a named volume), the API (with an env_file), and the React frontend.

docker-compose.yml
version: '3.8'

services:
  mongodb:
    image: mongo:6-jammy
    volumes:
      - mongo-data:/data/db
    networks:
      - app-network

  backend:
    build: ./backend
    ports:
      - "5000:5000"
    env_file: .env
    depends_on:
      - mongodb
    networks:
      - app-network

  frontend:
    build: ./frontend
    ports:
      - "80:80"
    depends_on:
      - backend
    networks:
      - app-network

volumes:
  mongo-data:

networks:
  app-network:
    driver: bridge

Run & Chaos Engineering Drills

Run the stack: docker-compose up -d. Then, deliberately trigger Incident 1 (OOM kill a container by modifying compose to limit memory to 10MB), Incident 2 (break the DNS by renaming the mongodb service but forgetting to update the backend `.env`), and Incident 4 (ruin the layer cache by moving COPY . . to line 3). Fix them using the exact methods taught.

Post-Mortems

Write an incident report for each chaos drill (Symptom, Root Cause, Fix, Prevention). Add these to your portfolio repository. This proves you know how to debug, not just write YAML.

💬 Real DevOps Interview Questions

Docker is the foundation of modern infrastructure. You must answer these flawlessly.

Q: What is the exact difference between a Container and a VM?

A VM virtualizes the hardware and runs a full Guest OS (heavy, slow boot). A container virtualizes the OS. It shares the host's Linux Kernel and uses Namespaces and cgroups for isolation. Containers are just regular processes (lightweight, instant boot).

Q: What is the difference between CMD and ENTRYPOINT?

CMD provides default arguments that are easily overridden by typing a command at the end of docker run. ENTRYPOINT hardcodes the executable so it cannot be easily overridden; any arguments passed to docker run are appended to the ENTRYPOINT.

Q: Why is layer order important in a Dockerfile?

Docker caches each line. If a layer changes, all subsequent layers are invalidated. We copy package.json and run npm install BEFORE copying the rest of the source code. If we didn't, changing a single line of CSS would bust the cache and force a 5-minute npm reinstall.

Q: What is a multi-stage build and why use it?

It allows us to use a heavy image (like a JDK) to compile our code in Stage 1, and then copy ONLY the compiled artifact into a tiny runtime image (like a JRE) in Stage 2. This drastically reduces image size and attack surface.

Q: A container exits with code 137. What happened?

Exit Code 137 means it was killed by SIGKILL (Signal 9). In Docker, this almost always means the Linux Out of Memory (OOM) Killer terminated the container because it exceeded its memory limit or the host ran out of RAM.

Q: How do containers on a Custom Bridge Network find each other?

Docker provides an embedded DNS server for custom bridge networks. If my web container needs to talk to my DB container (named `database`), it simply requests `http://database`. Docker automatically resolves the container name to its internal IP address.

Q: What is the difference between a Named Volume and a Bind Mount?

A Named Volume is fully managed by Docker and stored in /var/lib/docker/volumes; it is used in production for databases. A Bind Mount maps a specific host directory (like /Users/me/code) into the container; it is used in development for live-reloading code changes.

Q: How do you optimize a Docker image for production?

I use small base images (alpine/distroless), utilize multi-stage builds to discard build tools, run as a non-root user, carefully order layers to maximize cache hits, and avoid installing unnecessary packages like curl or vim.
