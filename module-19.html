💡 SRE is DevOps with a job title and more math

Google coined SRE in 2003. The core insight: treat operations as a software problem. SRE teams have coding standards for runbooks (they call them playbooks), measure their own toil like a bug backlog, and use error budgets to have objective, data-driven conversations with product teams about release velocity vs reliability.

2. SLIs, SLOs, SLAs — The Reliability Vocabulary

TermDefinitionExample
SLI (Service Level Indicator)A quantitative measure of service behavior. A ratio: good events / total events% of API requests returning 2xx in <200ms
SLO (Service Level Objective)An internal reliability target on an SLI. Binding within engineering.99.9% of requests succeed within 200ms over 30 days
SLA (Service Level Agreement)External contractual commitment. Penalty if breached (credits, lawsuits).AWS EC2 SLA: 99.99% monthly uptime; credits if below
Error Budget1 - SLO. The amount of unreliability you're allowed in the SLO window.99.9% SLO → 0.1% budget → 43 min/month
yaml — SLO definition (OpenSLO format)
apiVersion: openslo/v1
kind: SLO
metadata:
  name: techitfactory-api-availability
spec:
  service: techitfactory-api
  description: API request success rate SLO
  sli:
    metricSource:
      spec:
        # Module 16 PromQL payoff — SLI expressed as a PromQL query
        query: >
          sum(rate(http_requests_total{status=~"2.."}[5m]))
          /
          sum(rate(http_requests_total[5m]))
  objectives:
  - value: 0.999          # 99.9% SLO
    op: gte
    displayName: Monthly availability
⚠️ Why 100% SLO is wrong

A 100% availability SLO means any change risks breaching it — so you'd never deploy. SLOs should be "good enough that users are happy, but realistic enough to allow innovation." Google's rule: the SLO should be slightly better than users notice, but never 100%. Typical targets: user-facing APIs: 99.9%, critical payment flows: 99.99%, batch jobs: 99%.

3. Error Budgets — The Policy that Changes Conversations

The error budget converts a reliability debate ("we need 100% uptime!") into a business trade-off conversation. Error budget math:

bash — error budget calculations
# SLO = 99.9% over 30 days
# Error budget = (1 - 0.999) × 30 days × 24 hr × 60 min = 43.2 minutes

# SLO = 99.95% over 30 days → 21.6 minutes
# SLO = 99.99% over 30 days → 4.3 minutes

# Error budget burn rate: how fast are we consuming the budget?
# Burn rate 1 = consuming budget at exactly the SLO rate (will exhaust at end of window)
# Burn rate 14 (fast burn) = exhausting 30-day budget in 2 days — PAGE IMMEDIATELY

# PromQL: 1-hour burn rate alert (Module 16 AlertManager payoff)
# 1 - (sum(rate(http_requests_total{status=~"2.."}[1h])) / sum(rate(http_requests_total[1h]))) > 0.001 * 14
Budget stateMeaningEngineering response
Budget healthyOn track — consuming <1× burn rateDeploy freely, run experiments
Budget below 50%Burning faster than expectedSlow risky deploys, focus on reliability improvements
Budget exhaustedSLO breached for the windowFeature freeze; only reliability fixes ship until budget resets

4. Toil — Engineering's Technical Debt Equivalent

Toil is manual, repetitive, automatable operational work that scales with service growth. SRE practice: cap toil at <50% of team time (the rest must be engineering). Every runbook task is a toil item; every automation eliminates it.

Toil exampleAutomationModule link
Manually scaling deployments during high trafficHPA (Horizontal Pod Autoscaler)Module 11
Manually rotating IAM credentialsIRSA — no long-lived keysModule 12
Manually provisioning servers for new servicesTerraform modules + CI applyModule 13
Manually deploying new image versionsGitOps — commit triggers deployModule 17
Manually running the same kubectl describe triage flowPython runbook automation (boto3/subprocess)Module 14
Manually checking all 20 services for new CVEsTrivy in CI + ECR scanningModule 18
✅ Toil elimination is the SRE pitch in interviews

"In my current role, I noticed we were spending 4 hours a week manually checking CloudWatch alarms and creating tickets. I automated the correlation logic in Python (Module 14 boto3), which eliminated ~180 hours of toil per year and let the team focus on the alerting architecture instead of reading emails." This kind of answer wins SRE roles.

5. Incident Response — Command Structure

Incidents are chaotic without structure. The SRE Incident Command model assigns clear roles so responders aren't tripping over each other:

RoleResponsibility
Incident Commander (IC)Owns the incident. Coordinates response, decides escalations, controls the communication channel. Does NOT debug — delegates.
Tech LeadLeads technical diagnosis and mitigation. Focuses on the system, not communication.
Communications LeadWrites status page updates, stakeholder comms. Shields Tech Lead from interruptions.
ScribeDocuments timeline in real time: what was tried, what failed, what fixed it. The post-mortem raw material.
text — incident timeline template (scribe output)
INCIDENT: API 5xx rate above 50%          Severity: P1
IC: Vikram                                Started: 2026-07-14 14:32 UTC

14:32 - PagerDuty alert: http_error_rate > 10% for 5 minutes
14:33 - IC declared P1, opened war room
14:35 - No recent deploys in last 2 hours (checked ArgoCD)
14:38 - kubectl describe pods: 3/5 pods OOMKilled (Module 11 incident type)
14:40 - Memory limit: 256Mi — recent traffic spike doubled memory usage
14:41 - MITIGATION: kubectl patch deploy api -p '{"spec":{"template":{"spec":{"containers":[{"name":"api","resources":{"limits":{"memory":"512Mi"}}}]}}}}'
14:43 - All pods running. Error rate back to 0.1%.
14:44 - Post-mortem scheduled. Terraform/GitOps PR to update base limits.

6. Blameless Post-Mortems

The blameless post-mortem assumes that engineers are smart people working in a complex system — the system (process, tooling, alerting) failed, not the person. Blame produces cover-ups. Blameless culture produces learning and systemic improvement.

SectionContent
ImpactWhat users experienced, how many, how long, revenue/SLO impact
TimelineChronological from first signal to resolution (scribe log)
Root Cause5-Whys analysis. Not "the engineer ran the wrong command" — what system allowed that?
Contributing FactorsAlerting gaps, runbook gaps, deploy process gaps, on-call fatigue
Action ItemsSpecific, assigned, time-bounded. Not "be more careful" — "add memory limit review to deploy checklist by 2026-07-21 (owner: Vikram)"
💡 The 5-Whys for the OOMKilled incident

Why did the API fail? → Pods were OOMKilled. Why? → Memory limit was 256Mi. Why? → Default from the Helm chart wasn't updated for traffic growth. Why? → No process to review resource limits as traffic grows. Why? → We don't have an SLO-driven capacity review cadence. Fix: schedule quarterly resource review + add HPA (Module 11).

8. Backstage — The IDP Framework

Backstage (Spotify OSS, adopted by CNCF) is the most common IDP framework. It provides a service catalog (every microservice, its owner, runbook, SLO), Software Templates (scaffold new services with your golden-path CI/CD baked in), and plugin integrations (PagerDuty, ArgoCD, Grafana, GitHub).

bash — bootstrap Backstage
# Requires Node 18+
npx @backstage/create-app@latest

# Run locally
cd my-backstage
yarn dev    # http://localhost:3000
yaml — catalog-info.yaml (add to every service repo)
# Add catalog-info.yaml to the root of each microservice repo
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: techitfactory-api
  description: Core API for the TechitFactory platform
  annotations:
    github.com/project-slug: techitfactory/api
    pagerduty.com/service-id: P12345
    argocd/app-name: techitfactory-prod
    grafana/dashboard-selector: techitfactory-api
spec:
  type: service
  lifecycle: production
  owner: team-backend
  system: techitfactory-platform

9. On-Call Engineering — The Sustainable Model

On-call principleDetails
Alert only on SLO threatsIf an alert doesn't require action within 30 minutes, it's not a page — it's a ticket. PagerDuty fatigue kills teams. Only page when the error budget is burning fast (Module 3 concept from Section 3).
Runbooks for every alertEvery AlertManager rule links to a runbook. The responder shouldn't debug from scratch at 2 AM — the runbook walks through triage. Module 16 AlertManager annotations: runbook_url.
Fix the alert or fix the systemIf you're paged for the same thing three times, that's a toil item — automate the mitigation or fix the root cause. Never just silence the alert.
Rotation and compensationOn-call rotations should be 5–7 days per person, with minimum 2-person teams (primary + secondary). Compensate for on-call load.

    (1 - sum(rate(http_requests_total{status=~"2.."}[1h]))

  for: 5m
  labels:
    severity: critical
    team: platform

    description: "30-day budget will be exhausted in 2 days at current rate"

10. Real-Time Scenarios & War Room Practice

Scenario 1 — Error Budget Burned in 3 Days

StageDetails
SituationMonday: 30-day SLO budget exhausted by Wednesday after a bad deploy on Friday.
Response(1) Freeze new feature deploys per the error budget policy. (2) Roll back the Friday deploy via git revert (Module 17). (3) Schedule post-mortem for Thursday. (4) Only reliability improvements deploy until next month's budget resets.
PreventionFast-burn AlertManager rule (Section 9) would have paged at 14× burn rate — catching it 2 hours after the deploy instead of 3 days later. SLO monitoring dashboard (Module 16) visible to the on-call team.

Scenario 2 — Toil Explosion: 40% of Sprint on Manual Work

StageDetails
SituationNew services onboarding doubles. Team spends 40% of time provisioning namespaces, creating ArgoCD apps, and setting up Grafana dashboards manually for each new service.
SRE responseIdentify the toil. Build a Backstage Software Template: developer fills a form, template runs Terraform to create the namespace, applies the ArgoCD Application, scaffolds the Grafana dashboard. New service onboarding: 2 weeks → 15 minutes.
MeasurementTrack toil ratio before and after: 40% → 10%. Present at the next sprint review with numbers. This is the SRE career proof point.

Scenario 3 — Major Incident: Post-Mortem Culture Test

StageDetails
SituationA database migration caused 45 minutes of downtime. A manager wants to "find out who's responsible."
SRE responseBlameless post-mortem: the system allowed the migration to run without a canary or rollback plan. The action item is a process change (all database migrations require a rollback script and staging validation), not a reprimand. Blame prevents learning and causes engineers to hide incidents in the future.

Interview Questions

Q: What is an SLO and how does it differ from an SLA?

An SLO (Service Level Objective) is an internal reliability target expressed as a threshold on an SLI — for example, "99.9% of API requests succeed within 200ms over any 30-day window." It's a binding engineering commitment. An SLA (Service Level Agreement) is an external contract with customers or stakeholders — it typically has penalties if breached (credits, termination rights). SLOs should be set tighter than SLAs to give you a buffer. The SLO is owned by engineering; the SLA is owned by the business/legal team. Breaking your SLO is an engineering problem to fix; breaking the SLA is a customer/legal problem.

Q: What is an error budget and how does it change team behavior?

Error budget = 1 - SLO. For a 99.9% SLO over 30 days, you have 43.2 minutes of allowed unreliability. When the budget is healthy, teams can move fast — deploy features, run experiments. When the budget is low or exhausted, it's a forcing function: freeze new features, focus on reliability improvements. This transforms the conversation between product and engineering from subjective ("we need to ship faster") to objective ("we've burned 80% of our error budget this month — we can't take on more risk without violating our SLO"). Error budgets make reliability a shared team responsibility, not just an ops team problem.

Q: What is toil? Give an example of how you'd eliminate it.

Toil is manual, repetitive, automatable work that scales with service growth — it has no lasting engineering value. Examples: manually approving deploys, rotating credentials, scaling pods during traffic spikes. SRE practice is to cap toil at <50% of team time. Example elimination: we were manually creating Kubernetes namespaces, ArgoCD Applications, and Grafana dashboards for every new microservice (about 2 weeks of work per onboarding). I built a Backstage Software Template that runs Terraform and applies K8s manifests automatically when a developer submits a form. Onboarding time dropped to 15 minutes. That freed 3 weeks per quarter for reliability engineering work.

Q: What is a blameless post-mortem and why does it matter?

A blameless post-mortem analyzes an incident assuming that the people involved made reasonable decisions given the information they had at the time — the system (tooling, process, alerting, runbooks) is what failed, not the individual. Why it matters: blame produces cover-ups, silence, and fear of taking action during incidents. Blameless culture produces learning: engineers share the full timeline honestly because they won't be punished, which surfaces real root causes and systemic improvements. The key structure: timeline → root cause (5-Whys analysis of system failures, not people failures) → action items (specific, assigned, time-bounded). "Be more careful" is never an action item.

Q: What is an Internal Developer Platform?

An IDP (Internal Developer Platform) is a self-service product that lets application developers deploy and operate services without deep infrastructure expertise. It abstracts the underlying platform (Kubernetes, Terraform, ArgoCD, monitoring) behind a simpler interface — typically a service catalog, template wizard, and observability dashboards. Teams build IDPs on frameworks like Backstage. The goal is to encode the "golden path" — your organization's approved, secure, observable way to run a service — into a template developers can use in minutes. IDPs reduce toil for both the platform team and developers, improve consistency (every service gets monitoring, security policies, GitOps by default), and accelerate onboarding.

Q: How do you prevent on-call burnout?

SRE approach to sustainable on-call: (1) Alert only on SLO threats — if an alert doesn't require action within 30 minutes it should be a ticket, not a page. Most alert noise comes from alerts that fire but don't need immediate action. (2) Every alert links to a runbook — responders shouldn't debug from scratch at 2 AM. (3) Fix the alert or fix the system: if the same page fires 3 times, that's a toil item — automate the mitigation or fix root cause. (4) Measure pages per shift — target <2 actionable pages per on-call week, escalate if exceeded. (5) Rotation covers minimum 2 people (primary/secondary), and compensate for on-call load. Burnout is a system design problem, not a personal resilience problem.
