1. The Three Observability Pillars

PillarWhat it answersToolWhen to reach for it
Metrics"What is the system doing right now?" — numbers over timePrometheus + GrafanaAlerting, trending, capacity planning
Logs"What happened, and exactly when?"Loki + PromtailPost-incident forensics, debugging
Traces"Where did this request spend its time?"Jaeger, TempoLatency attribution in microservices

Monitoring vs Observability: monitoring is "did the known thing break?" (alert fires). Observability is "can I answer arbitrary questions about the system's internal state from its outputs?" — the three pillars together make that possible.

2. Prometheus Architecture

Pull-based scraping model
App/Exporter
(/metrics endpoint) ← scrape (pull) ← Prometheus Server
(TSDB, rules engine) → AlertManager
(routing, grouping) → Slack / PagerDuty
💡 Why pull vs push?

Prometheus pulls metrics from targets on a schedule. Push-based systems (StatsD, Graphite) require apps to know the monitoring server's address. Pull has two advantages: (1) if a target goes down, Prometheus knows — a push-based system would just see silence and might not alert. (2) The targets list is in Prometheus config (or service discovery) — not distributed across every app. The pull model is why Prometheus can alert on "target down" as a first-class concept.

4. PromQL — Prometheus Query Language

PromQL has four metric types. Understanding them determines which functions to use:

TypeWhat it isExampleQuery pattern
CounterMonotonically increasing number, resets on restartHTTP requests totalrate() or increase()
GaugeCan go up or downMemory usage, pod countDirect, or avg_over_time()
HistogramBucketed observations + count + sumRequest latencyhistogram_quantile()
SummaryPre-computed quantiles (client-side)Go GC pause durationDirect quantile labels
promql — ⭐ the 5 queries every engineer needs
# ⭐ 1. CPU utilization per node
1 - avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m]))

# ⭐ 2. Memory usage (used / total)
(node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes) / node_memory_MemTotal_bytes

# ⭐ 3. HTTP request rate by status code
sum by (status_code) (rate(http_requests_total[5m]))
#   rate() = per-second average over [5m] window
#   5m is the window — should be ≥ 4× the scrape interval (15s → use [1m] minimum)

# ⭐ 4. p95 request latency
histogram_quantile(0.95, sum by (le) (rate(http_request_duration_seconds_bucket[5m])))
#   le = "less than or equal" — the bucket label
#   0.95 = 95th percentile

# ⭐ 5. Pod restart count (detect crash loops)
increase(kube_pod_container_status_restarts_total[1h])
#   increase() = total increase over window (use for counters over fixed window)

groups:
- name: kubernetes.rules
  rules:
  - alert: PodOOMKilled
    expr: |
      increase(kube_pod_container_status_restarts_total[1h]) > 0
      and on(pod, namespace)
      kube_pod_container_status_last_terminated_reason{reason="OOMKilled"} == 1
    for: 0m                              # fire immediately (OOMKill = urgent)
    labels:
      severity: warning
    annotations:
      summary: "Pod {{ $labels.pod }} in {{ $labels.namespace }} was OOMKilled"
      description: "Container {{ $labels.container }} exceeded memory limit. Increase limit or investigate leak."

7. Grafana — Dashboards

Grafana turns PromQL queries into charts. Key workflow: add Prometheus as a data source (URL: http://prometheus-operated:9090 inside the cluster), create dashboards, use template variables for namespace/pod dropdowns.

💡 Import community dashboards first

Don't build from scratch. Import: Node Exporter Full (#1860), Kubernetes Cluster Monitoring (#3119), NGINX Ingress (#9614). Customize from there. Each dashboard ID works via Grafana → Dashboards → Import → paste ID. Version-control your final dashboard JSONs in Git — Grafana exports them via API.

9. AlertManager — Routing, Grouping, Silence

yaml — AlertManager config (severity-split routing)
global:
  slack_api_url: "https://hooks.slack.com/services/..."

route:
  group_by: [alertname, namespace]    # group related alerts together (prevents storm)
  group_wait: 30s                    # wait for more alerts before sending the first notification
  group_interval: 5m                 # wait before sending updates to an existing group
  repeat_interval: 4h                # re-notify if alert is still firing after 4h
  receiver: slack-warning            # default receiver
  routes:
  - match: { severity: critical }
    receiver: pagerduty-critical       # critical → page the on-call
  - match: { severity: warning }
    receiver: slack-warning            # warning → Slack ticket channel (M09 notify thread done)

receivers:
- name: slack-warning
  slack_configs:
  - channel: "#alerts-warning"
    text: "{{ .CommonAnnotations.summary }}"

- name: pagerduty-critical
  pagerduty_configs:
  - routing_key: "YOUR_PD_INTEGRATION_KEY"
    severity: critical
⭐ Alert fatigue doctrine

Every alert must be actionable — if you can't tell an engineer what to do when the alert fires, it shouldn't page. Severity split: critical = wake someone up at 3am (service is down / SLO breach imminent); warning = create a ticket, investigate during business hours. Review your alert history monthly — any alert that fires and is silenced without action is noise and should be deleted or downgraded.

10. Application Metrics — prom-client

javascript — instrument a Node.js app (Module 08 app)
const client = require('prom-client');
const express = require('express');
const app = express();

// Default metrics: Node.js process metrics (event loop lag, memory, GC)
client.collectDefaultMetrics();

// Custom counter: HTTP requests by method + status
const httpRequests = new client.Counter({
  name: 'http_requests_total',
  help: 'Total HTTP requests',
  labelNames: ['method', 'status_code', 'route']
});

// Custom histogram: request duration (needed for p95 calculation)
const httpDuration = new client.Histogram({
  name: 'http_request_duration_seconds',
  help: 'HTTP request duration in seconds',
  labelNames: ['method', 'route'],
  buckets: [0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5]  // latency SLO thresholds
});

// Middleware: record every request
app.use((req, res, next) => {
  const end = httpDuration.startTimer({ method: req.method, route: req.path });
  res.on('finish', () => {
    httpRequests.inc({ method: req.method, status_code: res.statusCode, route: req.path });
    end();
  });
  next();
});

// /metrics endpoint — Prometheus scrapes this
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', client.register.contentType);
  res.end(await client.register.metrics());
});
yaml — ServiceMonitor to scrape the app
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: api-monitor
  namespace: production
spec:
  selector:
    matchLabels: { app: api }         # selects the api Service
  endpoints:
  - port: http
    path: /metrics
    interval: 15s

12. Real-Time Incidents & Troubleshooting

Incident 1 — Prometheus Disk Full (Cardinality Explosion)

StageDetails
SymptomPrometheus pod restarts; kubectl top pod shows it at memory limit. Grafana dashboards show gaps.
Investigationtopk(10, count by (__name__)({__name__=~".+"})) in PromQL — this query finds the metrics with the most series. If http_requests_total{user_id="..."} has 10M series (one per user), you found it.
Root CauseA developer added a user_id label to HTTP metrics. High-cardinality labels (unbounded values like user IDs, UUIDs, IP addresses) create one time series per unique value — Prometheus TSDB can't handle millions of series.
FixRemove the label (relabeling rules in the scrape config) or drop the metric entirely. Set TSDB retention time and storage limits. Bump Prometheus memory limit.
PreventionOnly use labels with bounded, low-cardinality values (method, status_code, route — not user_id, session_id). Code-review any new prom-client instrumentation.

Incident 2 — Alert Storm at 3am (200 Alerts from One Node Down)

StageDetails
SymptomOn-call engineer gets 200 Slack/PagerDuty notifications in 5 minutes. All from different pods on the same node.
Root CauseAlertManager grouping not configured. Each pod fires a separate alert.
FixAlertManager group_by: [alertname, node] — group all alerts from the same node into one notification. Add inhibit_rules: if NodeDown fires, suppress all child pod alerts from that node.

Incident 3 — "Dashboard Says Fine, Users Say Down"

StageDetails
Root CauseYou monitored infrastructure (CPU, memory, pod count) but not the user-visible path. A misconfigured ALB was returning 502 to users while pods were healthy by all internal metrics.
FixAdd a blackbox exporter: it makes real HTTP requests from outside the cluster and records success/failure. Now Prometheus scrapes the user-visible endpoint the same way a user does. RED method dashboard on the app metrics (Section 8) catches this too.

Interview Questions

Click a card to reveal a strong candidate answer.

Q: Why does Prometheus use a pull model instead of push?

Pull means Prometheus knows which targets exist and actively scrapes them. If a target goes down, Prometheus detects the failed scrape immediately and can fire an "instance down" alert. In a push model, a dead target just stops pushing — you'd see silence and might not know if it's dead or just very quiet. Pull also centralizes the target list in Prometheus config or service discovery, rather than distributing the monitoring server address to every application. The tradeoff: targets must be reachable from Prometheus, which requires network planning in multi-AZ or multi-cluster setups.

Q: What's the difference between a counter and a gauge?

A counter only goes up (and resets to zero on restart). Use rate() to get the per-second rate of change — never query raw counter values, they're cumulative totals. A gauge can go up or down — current memory usage, number of concurrent connections, replica count. You can query gauges directly. The most common mistake: treating request counts as a gauge and forgetting that the raw number is meaningless without a time window. Always apply rate() to counters.

Q: Explain rate() and what the time window means.

rate(metric[5m]) computes the average per-second rate of increase over the last 5 minutes, accounting for counter resets. The window [5m] is the look-back window for the calculation — it should be at least 4× the scrape interval (default 15s → minimum [1m]). A longer window smooths spikes; a shorter window is more responsive but noisier. For alerting, I use [5m] for stability. For SLO burn-rate alerts, I use two windows ([5m] and [1h]) together to catch both fast burns and slow burns.

Q: How do you avoid alert fatigue?

Three rules: (1) Every alert must be actionable — there must be a runbook entry for what to do. If the response is "watch it for a while," it's noise. (2) Severity split: critical = page at 3am, warning = ticket for business hours. Most alerts should be warnings. (3) Regular alert review — anything silenced more than once in a month is either misconfigured or should be a lower severity. Also: group with AlertManager (one node failure → one alert, not 50), and use inhibition rules (NodeDown inhibits all pod alerts on that node).

Q: What is cardinality, and why can high cardinality kill Prometheus?

Cardinality is the number of unique label value combinations for a metric. http_requests_total{method, status_code, route} might have 100 series (3 methods × 5 status codes × 7 routes). Add a user_id label and it jumps to 100 × millions of users. Prometheus stores each time series separately in TSDB — millions of series consumes gigabytes of memory and disk. The scraper slows to a crawl, memory OOM kills the Prometheus pod, and you lose monitoring. Rule: only use labels with low, bounded cardinality. User IDs, request IDs, IP addresses — never as labels.

Q: How would you monitor a new microservice from scratch?

Start with the RED method: instrument the app with a counter for requests (labelled by method, route, status code) and a histogram for request duration. Export a /metrics endpoint, create a ServiceMonitor. Build a Grafana dashboard: Rate (req/sec), Error ratio (5xx/total), Duration (p50/p95/p99). Add three alerts: error ratio > 1% (warning), p95 latency > SLO threshold (warning), target down (critical). Then add the USE method for the underlying resources: CPU utilization, memory saturation, disk/network saturation. That gives you the full picture within a day of launch.
