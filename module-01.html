1. What is DevOps — Culture, Principles & Practices

DevOps is a combination of Development and Operations. It is not a tool, not a job title you can buy, and not just automation — it is a culture and set of practices that brings development teams (who want to ship changes fast) and operations teams (who want systems stable) together, so software can be delivered faster, more reliably, and continuously.

The Wall of Confusion — What DevOps Removes
Developers: "It works on my machine" ⚡ Operations: "It broke my server"
↓
DevOps: One team, shared ownership, shared responsibility
Core Principles — CALMS
PrincipleMeaning
CultureBreak down silos — Dev, Ops, QA, and Security collaborate and share responsibility for the product end-to-end
AutomationAutomate everything repeatable — builds, tests, deployments, infrastructure provisioning
LeanShip small batches frequently, eliminate waste, shorten feedback loops
MeasurementMeasure everything — deployment frequency, lead time, failure rate, recovery time (DORA metrics)
SharingShare knowledge, tools, and post-mortems openly — blameless culture
Key Practices
  • Continuous Integration (CI) — merge and test code changes frequently
  • Continuous Delivery / Deployment (CD) — release software in short, automated cycles
  • Infrastructure as Code (IaC) — manage servers and networks through versioned code (Terraform, Ansible)
  • Monitoring & Observability — know the health of systems in real time (Prometheus, Grafana)
  • Microservices & Containers — small, independently deployable services (Docker, Kubernetes)
  • Blameless Post-Mortems — learn from failures instead of punishing them
💡 Remember This

DevOps = People + Process + Tools, in that order. Companies fail at DevOps when they buy tools but skip the culture change. You can have Jenkins and Kubernetes and still not "do DevOps".

2. Software Development Life Cycle (SDLC)

The SDLC is the structured process every piece of software goes through — from an idea to a running product in production. DevOps doesn't replace the SDLC; it automates and accelerates it.

The 6 Classic SDLC Phases
1. Planning → 2. Analysis → 3. Design → 4. Implementation → 5. Testing → 6. Deployment & Maintenance
PhaseWhat HappensWho Is Involved
PlanningDefine goals, scope, budget, and timelinesProduct Owner, Business Analysts
AnalysisGather detailed requirements (functional & non-functional)Analysts, Architects
DesignSystem architecture, database design, UI/UX designArchitects, Designers
ImplementationActual coding of the applicationDevelopers
TestingUnit, integration, system, and acceptance testingQA Engineers
Deployment & MaintenanceRelease to production, monitor, patch, improveOperations / DevOps
⚠️ The Pain Point DevOps Solves

In traditional SDLC, each phase is a hand-off to a different team — and every hand-off adds delay, miscommunication, and finger-pointing. DevOps turns these phases into a continuous, automated loop owned by one collaborative team.

⚖️

3. Waterfall vs Agile vs DevOps

These three represent the evolution of software delivery. Each one fixed the biggest problem of the previous approach.

Waterfall — The Sequential Era

Phases run strictly one after another — you cannot start Testing until all Development is done. Requirements are frozen upfront. A release could take 6–18 months, and customer feedback arrived only at the very end, when changes were most expensive.

Agile — The Iterative Era

Agile (2001) broke the work into sprints of 1–4 weeks, delivering small working increments and getting customer feedback continuously. But Agile only sped up development — the finished code still sat waiting for the Ops team to deploy it manually, sometimes for weeks.

DevOps — The Continuous Era

DevOps extends Agile all the way to production. Automated pipelines build, test, and deploy every change, so software can be released daily — or even multiple times per day.

AspectWaterfallAgileDevOps
ApproachSequential phasesIterative sprintsContinuous flow
Release CycleMonths / YearsWeeks (per sprint)Days / Hours / On-demand
ScopeDev → Test → Deploy as silosDev + QA collaborateDev + QA + Ops + Security as one team
FeedbackAt the end of the projectEnd of every sprintReal-time from production monitoring
DeploymentManual, high-risk eventManual / semi-automatedFully automated pipeline
Change HandlingVery expensiveWelcomed between sprintsContinuous, small, low-risk changes
Best ForFixed-scope projects (e.g., regulated hardware)Evolving product requirementsCloud-native, always-on services
✅ Key Insight

Agile and DevOps are not competitors — DevOps is the natural extension of Agile. Agile answers "how do we build the right thing fast?", DevOps answers "how do we deliver and run it fast and safely?"

4. Roles & Responsibilities of a DevOps Engineer

A DevOps Engineer is the bridge between development and operations — building the automation, infrastructure, and pipelines that let developers ship code to production safely and repeatedly.

Day-to-Day Responsibilities
  • CI/CD Pipelines — design, build, and maintain automated build → test → deploy pipelines (Jenkins, GitHub Actions)
  • Infrastructure as Code — provision and manage cloud infrastructure with Terraform / Ansible instead of manual clicks
  • Containerization & Orchestration — package apps with Docker, run them at scale with Kubernetes
  • Cloud Management — architect and operate workloads on AWS / Azure / GCP with cost awareness
  • Monitoring & Alerting — set up Prometheus, Grafana, and log aggregation; define alerts before users notice problems
  • Security (DevSecOps) — embed vulnerability scanning, secret management, and least-privilege access into pipelines
  • Incident Response — troubleshoot production issues, participate in on-call, write blameless post-mortems
  • Collaboration — enable developer self-service, document runbooks, spread automation culture across teams
DevOps Engineer vs Related Roles
RolePrimary Focus
DevOps EngineerAutomating the path from code to production (pipelines, IaC, deployment)
Site Reliability Engineer (SRE)Reliability of production systems — SLIs/SLOs, error budgets, incident management
Platform EngineerBuilding the internal developer platform (IDP) other teams build on
Cloud EngineerDesigning and managing cloud infrastructure and services
Build/Release EngineerBuild systems, artifact management, release coordination
💡 Career Tip

Job postings use these titles interchangeably. Read the responsibilities, not the title — the skill set in this roadmap (Linux, Git, CI/CD, Docker, Kubernetes, Terraform, AWS, monitoring) qualifies you for all of them.

6. Continuous Integration, Delivery & Deployment

These three "CDs" are the engine of DevOps. They differ in how far automation goes before a human is involved.

Continuous Integration (CI)

Developers merge small changes into the shared main branch frequently — at least daily. Every merge automatically triggers a build and test run, so integration bugs are caught within minutes instead of at the end of the month ("integration hell").

CI Flow
Commit → Push to Git → Automated Build → Automated Tests → ✅ Feedback in Minutes
Continuous Delivery (CDelivery)

Extends CI: every change that passes the pipeline is packaged and kept in a deployable state. Deployment to production requires one manual approval — the business decides when to release, but the release itself is a button-press, not a project.

Continuous Deployment (CDeployment)

The final step of automation: every change that passes all tests goes to production automatically — no human approval at all. This is how Netflix and Amazon deploy thousands of times per day. It demands excellent test coverage and monitoring.

Continuous IntegrationContinuous DeliveryContinuous Deployment
AutomatesBuild + TestBuild + Test + ReleaseBuild + Test + Release + Deploy
Manual StepMerge reviewApproval to deployNone
Production Deploys—On demandEvery passing commit
RequirementAutomated tests+ Staging environment+ High test confidence & monitoring
⚠️ Common Confusion

"CD" means Delivery or Deployment depending on context. The difference is one thing only: is the final push to production approved by a human (Delivery) or fully automatic (Deployment)?

Interview Questions

Q1

What is DevOps and why do companies adopt it?

DevOps is a culture and set of practices that unifies software development and IT operations to deliver software faster and more reliably. Companies adopt it to increase deployment frequency, reduce failure rates, recover from incidents faster, and shorten the time from idea to production — measured by the four DORA metrics.
Q2

Explain the difference between Continuous Delivery and Continuous Deployment.

Both keep every change in a deployable state through automated build and test. In Continuous Delivery, the final deployment to production requires a manual approval — a human decides when. In Continuous Deployment, every change that passes the pipeline goes to production automatically with no human gate.
Q3

How is DevOps different from Agile?

Agile is about how software is built — iteratively, in short sprints with continuous customer feedback. DevOps extends Agile to how software is delivered and operated — automating the path to production and unifying Dev and Ops. Agile ends at "working software"; DevOps ends at "working software running reliably in production". They complement each other.
Q4

Walk me through what happens when a developer pushes code to production.

The push to Git triggers the CI server (Jenkins/GitHub Actions), which builds the code, runs unit tests and quality/security scans (SonarQube, Trivy), builds a Docker image, and pushes it to a registry (ECR). The CD stage updates the GitOps repository with the new image tag; ArgoCD detects the change and syncs it to the Kubernetes cluster. Prometheus and Grafana then monitor the new version, and alerts fire if anything degrades — enabling automatic or manual rollback.
Q5

What are the DORA metrics?

The four key metrics from DevOps Research and Assessment used to measure DevOps performance: (1) Deployment Frequency — how often you release to production; (2) Lead Time for Changes — commit to production time; (3) Change Failure Rate — percentage of deployments causing failures; (4) Mean Time to Recovery (MTTR) — how fast you recover from incidents. Elite teams deploy on demand, with lead times under a day and MTTR under an hour.
Q6

Is DevOps a role, a tool, or a culture?

Primarily a culture — shared ownership of software from idea to production, supported by automation and measurement (CALMS: Culture, Automation, Lean, Measurement, Sharing). The industry also uses "DevOps Engineer" as a role that builds the enabling automation, and there are many DevOps tools — but tools without the culture change fail to deliver DevOps outcomes.
