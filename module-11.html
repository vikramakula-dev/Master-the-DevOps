1. Why Orchestration?

Docker Compose is excellent up to one host. Four problems break it at scale:

ProblemCompose realityKubernetes answer
Multi-host schedulingRuns only on the machine you SSH'd intoScheduler places pods across a cluster of nodes
Self-healingContainer crashes → stays dead until you restart itController detects drift, replaces failed pods automatically
Rolling updatesDown-gap between docker compose down and upDeployments replace pods one-at-a-time, zero downtime
Service discovery at scaleHardcoded container names in the same Compose filekube-dns resolves every Service by name across the cluster
💡 The Key Insight

Kubernetes is declarative — you describe the desired state ("3 replicas of this image"), and controllers continuously converge reality toward it. This reconciliation loop is the core pattern, and you'll see it again in Terraform (M13) and ArgoCD (M17).

3. Local Setup — Minikube & kubectl

bash — install and start
# Install minikube (single-node K8s cluster on your machine)
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
sudo install minikube-linux-amd64 /usr/local/bin/minikube

# Install kubectl (the CLI for every K8s cluster, local or cloud)
curl -LO "https://dl.k8s.io/release/$(curl -Ls https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
sudo install -o root -g root -m 0755 kubectl /usr/local/bin/kubectl

# Start a local cluster (uses Docker as the driver)
minikube start --driver=docker --cpus=2 --memory=4096

# Verify — both should show "Running" / "Ready"
minikube status
kubectl get nodes

4. Pods — The Smallest Deployable Unit

A pod wraps one or more containers that share a network namespace (same IP) and optional volumes. You rarely create pods directly — Deployments manage them. But understanding the pod YAML is the "decode ls -l" moment of Kubernetes:

yaml — annotated Pod manifest
apiVersion: v1                    # K8s API group/version — always check docs
kind: Pod                         # resource type
metadata:
  name: api-pod                   # unique name in the namespace
  labels:
    app: api                      # labels = the glue (Services use selectors to find pods)
spec:
  containers:
  - name: api                     # container name
    image: techitfactory/api:1.4.2 # always pin a tag — never :latest in prod
    ports:
    - containerPort: 3000          # documentation only — doesn't open firewall rules
    env:                           # env vars (or use envFrom + ConfigMap/Secret)
    - name: NODE_ENV
      value: production
    resources:                     # Section 11 — always set these
      requests: { cpu: "100m", memory: "128Mi" }
      limits:   { cpu: "500m", memory: "256Mi" }

5. Deployments & Rolling Updates

A Deployment manages a ReplicaSet, which manages pods. You almost never touch ReplicaSets directly — just set replicas and strategy in the Deployment.

yaml — Deployment with rolling update strategy
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-deployment
spec:
  replicas: 3                      # desired state — controller converges to this
  selector:
    matchLabels:
      app: api                    # must match pod template labels below
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1                 # 1 extra pod during update (4 total briefly)
      maxUnavailable: 0           # never take a pod down before new one is ready
  template:                        # pod template — same spec as a Pod manifest
    metadata:
      labels:
        app: api
    spec:
      containers:
      - name: api
        image: techitfactory/api:1.4.2
        readinessProbe:             # must pass before pod receives traffic
          httpGet: { path: /health, port: 3000 }
          initialDelaySeconds: 5
bash — rolling update lifecycle (⭐ daily use)
# Apply (create or update)
kubectl apply -f deployment.yaml

# ⭐ Watch the rollout in real time
kubectl rollout status deployment/api-deployment

# ⭐ View rollout history
kubectl rollout history deployment/api-deployment

# ⭐ Rollback to previous version (the Module 09 zero-downtime promise, delivered)
kubectl rollout undo deployment/api-deployment

# Rollback to a specific revision
kubectl rollout undo deployment/api-deployment --to-revision=2

# Scale manually
kubectl scale deployment api-deployment --replicas=5

6. Services — Stable Networking for Pods

Pods are ephemeral — they die and get new IPs. A Service gives you a stable virtual IP and DNS name that routes to matching pods via label selectors. kube-dns resolves api-service.default.svc.cluster.local to the Service's ClusterIP — the Module 08 container-name networking concept, cluster-wide.

TypeReachable fromUse case
ClusterIPInside cluster onlyInternal service-to-service (default)
NodePortAny node IP + port (30000-32767)Dev/testing, not production
LoadBalancerExternal (cloud provisions an LB)Prod external access, replaced by Ingress in practice
yaml — ClusterIP Service
apiVersion: v1
kind: Service
metadata:
  name: api-service          # DNS name: api-service.<namespace>.svc.cluster.local
spec:
  selector:
    app: api                 # routes to all pods with label app=api
  ports:
  - port: 80                 # port clients connect to
    targetPort: 3000         # port the pod actually listens on
  type: ClusterIP
🚪

7. Ingress — One Load Balancer, Many Services

Instead of one LoadBalancer Service per microservice (expensive and unmanageable), Ingress provides L7 routing: one LB → nginx-ingress controller → routes by host/path to the right Service. This is the Module 04 reverse-proxy concept at cluster scale.

bash — enable ingress on minikube
minikube addons enable ingress     # installs nginx-ingress controller
kubectl get pods -n ingress-nginx   # verify controller is Running
yaml — Ingress with host routing + TLS
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: techitfactory-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /  # strip path prefix
spec:
  ingressClassName: nginx
  tls:
  - hosts: [app.techitfactory.com]
    secretName: tls-secret        # TLS cert in a K8s Secret
  rules:
  - host: app.techitfactory.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service: { name: api-service, port: { number: 80 } }
      - path: /
        pathType: Prefix
        backend:
          service: { name: frontend-service, port: { number: 80 } }

8. ConfigMaps & Secrets

ConfigMaps and Secrets externalize configuration from the container image — the Module 08 and 09 env_file pattern, but cluster-native. ConfigMaps hold non-sensitive config; Secrets hold credentials.

bash — create and consume
# Create ConfigMap from literals
kubectl create configmap api-config \
    --from-literal=NODE_ENV=production \
    --from-literal=PORT=3000

# Create Secret (values get base64-encoded automatically)
kubectl create secret generic api-secrets \
    --from-literal=DB_PASSWORD=supersecret \
    --from-literal=JWT_SECRET=mykey123
yaml — consume via envFrom in a Deployment
spec:
  containers:
  - name: api
    image: techitfactory/api:1.4.2
    envFrom:
    - configMapRef:
        name: api-config       # all keys become env vars
    - secretRef:
        name: api-secrets      # secret keys also become env vars
🚨 base64 ≠ encryption

K8s Secrets are base64-encoded, not encrypted. Anyone with cluster access who can kubectl get secret can decode them. For production, store secrets in HashiCorp Vault or AWS Secrets Manager and inject them at runtime — covered in Module 18. Never commit Secret YAMLs with real values to Git (Module 03 warning applies).

9. Namespaces, Volumes & StatefulSets

Namespaces are virtual clusters within a cluster — the capstone uses three: development, staging, production. Resources in different namespaces are isolated by default; cross-namespace communication uses the full DNS name.

Volumes and StatefulSets — Deployments are for stateless workloads; StatefulSets are for databases. Key differences:

FeatureDeploymentStatefulSet
Pod namesRandom (api-7d4f9b-xkzpl)Stable ordinals (mongo-0, mongo-1)
Persistent storageShared PVC or noneEach pod gets its own PVC, retained on delete
Scaling orderParallel, randomSequential (0 → 1 → 2), ordered rollout
Use caseAPI servers, web frontendsDatabases, message queues, any stateful app
yaml — StatefulSet for MongoDB with PVC template
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mongo
spec:
  serviceName: mongo-svc     # headless Service for stable DNS (mongo-0.mongo-svc)
  replicas: 1
  selector:
    matchLabels: { app: mongo }
  template:
    metadata:
      labels: { app: mongo }
    spec:
      containers:
      - name: mongo
        image: mongo:6.0
        volumeMounts:
        - name: data
          mountPath: /data/db
  volumeClaimTemplates:       # each pod gets its own PVC — persists on pod restart
  - metadata: { name: data }
    spec:
      accessModes: [ReadWriteOnce]
      resources:
        requests: { storage: 10Gi }

10. Helm — Kubernetes Package Manager

Helm is the package manager for Kubernetes. A chart is a directory of templated YAML files + a values.yaml file. Instead of maintaining separate manifests for dev/staging/prod, you maintain one chart + per-environment values.

bash — ⭐ Helm daily commands
# Install Helm
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash

# Create a chart skeleton
helm create techitfactory-api
# Creates: Chart.yaml, values.yaml, templates/ (Deployment, Service, Ingress, etc.)

# ⭐ Install a release
helm install techitfactory-dev ./techitfactory-api \
    --namespace development --create-namespace \
    -f values-dev.yaml

# ⭐ Upgrade (update image tag, config, etc.)
helm upgrade techitfactory-dev ./techitfactory-api \
    -f values-dev.yaml \
    --set image.tag=1.5.0          # override a single value

# ⭐ Rollback to previous release
helm rollback techitfactory-dev 1  # revision 1

# List, status, uninstall
helm list -A                       # all namespaces
helm status techitfactory-dev
helm uninstall techitfactory-dev
Docker Compose conceptHelm equivalent
docker-compose.ymltemplates/*.yaml (rendered manifests)
.env filevalues.yaml
docker compose up -dhelm install
docker compose downhelm uninstall

11. Resource Requests, Limits & HPA

Requests = what the scheduler uses to place the pod (guaranteed minimum). Limits = the hard ceiling. Exceeding the CPU limit → throttled; exceeding the memory limit → OOMKilled (exit code 137 — the Module 08 callback).

yaml — HorizontalPodAutoscaler
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-deployment
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70   # scale out when avg CPU > 70% of request
bash — ⭐ resource monitoring
# ⭐ Top nodes and pods (like htop for K8s — requires metrics-server)
kubectl top nodes
kubectl top pods --sort-by=memory

# Watch HPA in action during a load test
kubectl get hpa api-hpa --watch

12. Real-Time Incidents & Troubleshooting

Incident 1 — CrashLoopBackOff

StageDetails
Symptomkubectl get pods shows CrashLoopBackOff. This is Module 09 Incident 6 reborn at the cluster level.
Investigationkubectl logs api-pod → logs from current (crashed) container. kubectl logs api-pod -p → logs from the previous crash (often the real error). kubectl describe pod api-pod → Events section shows exit codes and restart count.
Root CauseMissing environment variable — the app starts, immediately errors on process.env.DB_URL undefined, exits with code 1. K8s restarts with exponential backoff (5s→10s→20s…→5min), hence the "back-off" in the name.
FixAdd the missing key to the ConfigMap or Secret, then kubectl rollout restart deployment/api-deployment.

Incident 2 — ImagePullBackOff

bash — debug ImagePullBackOff
# Read the Events section — it tells you exactly why
kubectl describe pod failing-pod | grep -A 10 Events:
# Common causes:
# 1. Image tag typo: "no such image: techitfactory/api:1.4.22"
# 2. Private registry: "unauthorized: authentication required"
# → add imagePullSecrets or use IRSA (M12) / instance role
# 3. Registry rate limit (Docker Hub): "too many requests"
# → authenticate (logged-in users get 200 pulls/6h vs 100 anonymous)

Incident 3 — OOMKilled

StageDetails
SymptomPod restarts periodically; kubectl describe pod shows OOMKilled, exit code 137 (Module 08 callback — that was the signal, now you see it in K8s).
Investigationkubectl top pod api-pod — watch memory trend over time. If it consistently approaches the limit, the limit is too low or there's a memory leak.
FixRaise the memory limit in the Deployment. Separately, profile the app for leaks (heap dump, APM tool). Don't just raise limits blindly — find the root cause.

Incident 4 — Pod Stuck "Pending" Forever

bash — debug Pending pods
# ⭐ Describe always shows you what the scheduler is thinking
kubectl describe pod pending-pod | grep -A 5 "Events:"
# "Insufficient cpu" → node has less available CPU than the pod's request
# "Insufficient memory" → same for memory
# "no nodes available" → all nodes are at capacity or tainted

# Check node available resources
kubectl describe nodes | grep -A 5 "Allocated resources"

Interview Questions

Q: What's the difference between a Pod, a Container, and a Deployment?

A container is the running process (Docker). A pod is a K8s wrapper around one or more containers that share a network namespace and lifecycle — pods are the smallest deployable unit. A Deployment is a controller that declares the desired number of pod replicas and manages rolling updates — it creates a ReplicaSet, which creates pods. You almost never create pods directly; you define a Deployment and let K8s manage the pods.

Q: Walk me through what happens when you run `kubectl apply`.

kubectl sends the manifest to the API Server. The API Server validates it, writes it to etcd (the source of truth), and returns 200. The Deployment controller (part of controller-manager) watches etcd, notices the desired state changed, creates/updates the ReplicaSet. The Scheduler watches for unscheduled pods (ReplicaSet creates them), picks the best node (by resource availability, affinity rules, etc.), and writes the binding to etcd. The kubelet on that node watches etcd for pods assigned to it, pulls the image via containerd, and starts the container.

Q: ClusterIP vs NodePort vs LoadBalancer vs Ingress — when to use each?

ClusterIP (default) — internal cluster communication; use for service-to-service calls. NodePort — exposes the service on a port on every node; fine for testing, too clunky for production. LoadBalancer — provisions a cloud LB per service; works but gets expensive with many services. Ingress — the production choice: one cloud LB runs an nginx/ALB controller that does L7 routing to many ClusterIP services by host/path rules. One LB, many services, TLS termination at the edge.

Q: How do you debug a CrashLoopBackOff?

First: kubectl logs <pod> — current container logs. If the container crashed before logging anything, kubectl logs <pod> -p for previous container logs (often the actual error). Then: kubectl describe pod <pod> — the Events section shows exit codes and the exact failure reason. Common causes: missing env var (app panics at startup), wrong command/entrypoint, liveness probe too aggressive, or OOM (check exit code 137). Fix the root cause, never just increase restart limits.

Q: What's the difference between resource requests and limits?

Requests are what the scheduler uses — a pod with cpu: 500m request is only placed on a node that has 500m CPU available. It's the guaranteed minimum. Limits are the hard cap — if the container exceeds its CPU limit, it's throttled (slowed, not killed); if it exceeds its memory limit, it's OOMKilled. Setting requests without limits is dangerous (noisy neighbor); setting limits too low causes OOMKilled. Start with requests = typical usage, limit = 2×.

Q: Why use a StatefulSet instead of a Deployment for a database?

Databases need stable identity and stable storage. StatefulSets give pods predictable names (mongo-0, not random hashes) that don't change on restart — critical for replica sets where members address each other by name. Each pod gets its own PVC from a volumeClaimTemplate — if pod mongo-0 restarts, it remounts the same volume (the same 10GB of data). Deployments share a PVC or use ephemeral storage, which is wrong for anything with persistent state.

Q: What's the difference between a rolling update and a recreate strategy?

Rolling update (default): replaces pods gradually — new pods come up, pass readiness probes, then old pods come down. With maxUnavailable: 0 and maxSurge: 1, you always have at least the full replica count serving traffic — zero downtime. Recreate: terminates all old pods first, then starts new ones — there's a downtime gap. Use Recreate only when you cannot run two versions simultaneously (e.g., a DB schema migration where v1 and v2 are incompatible). Default to Rolling for everything else.
