1. Why Configuration Management?

Scenario: you have 10 servers. In month 1 you SSH'd into each and ran your setup script. By month 6, someone hotfixed a config on server 3, someone else updated nginx on servers 1-5 but forgot 6-10. Now each server is slightly different — this is configuration drift and it causes "works on my server" bugs at 3am.

ApproachIdempotent?Scalable?Drift-proof?
Manual SSH + commands❌❌ (n servers = n SSH sessions)❌
Bash scripts (M02)❌ (run twice → breaks)Partial (for-loop over IPs)❌
Ansible✅✅ (parallel execution)✅ (re-run to converge)

Idempotency means "ensure nginx is installed" — Ansible checks if nginx is already installed; if so, it does nothing. Run the playbook 10 times, same result. Your Module 02 script would try to install nginx 10 times.

2. Architecture — Agentless over SSH

Ansible is agentless — no daemon to install on managed nodes. It connects via SSH (Module 02 keys payoff — Ansible literally is "SSH in a for-loop with a brain"), pushes Python code to /tmp, executes it, and cleans up.

4. Ad-hoc Commands vs Playbooks

bash — ⭐ ad-hoc commands (one-offs)
# ⭐ Restart nginx on all web servers
ansible web -i hosts.ini -m service -a "name=nginx state=restarted" --become

# ⭐ Check disk usage across all hosts
ansible all -i hosts.ini -m shell -a "df -h" --become

# Copy a file to all web servers
ansible web -i hosts.ini -m copy \
    -a "src=./nginx.conf dest=/etc/nginx/nginx.conf mode=0644" --become

# Run a command and show output
ansible all -i hosts.ini -m command -a "uptime"

Ad-hoc commands are for one-offs. If you'll run the same set of tasks more than once, write a playbook — it's documented, version-controlled, and idempotent.

5. Playbooks — The Module 02 server-setup.sh Reborn

yaml — annotated playbook: setup.yml
---
- name: Configure web servers          # play name (shown in output)
  hosts: web                            # inventory group to target
  become: true                          # sudo — equivalent to M02's sudo
  vars:
    app_port: 3000
    nginx_worker_processes: auto

  handlers:                             # run only when notified, and only once
  - name: Restart nginx
    service: { name: nginx, state: restarted }

  tasks:
  - name: Install nginx                 # declarative: "ensure installed"
    apt:
      name: nginx
      state: present                    # idempotent — skips if already present
      update_cache: yes

  - name: Deploy nginx config from template
    template:
      src: nginx.conf.j2                # Jinja2 template — vars injected at render
      dest: /etc/nginx/nginx.conf
      validate: nginx -t -c %s         # test config before deploying
    notify: Restart nginx               # trigger handler only if file changed

  - name: Start and enable nginx
    service:
      name: nginx
      state: started
      enabled: yes

  - name: Allow app port through firewall
    ufw:
      rule: allow
      port: "{{ app_port }}"            # Jinja2 variable reference
      proto: tcp
    when: ansible_os_family == "Debian" # conditional: skip on non-Debian (gather_facts)
jinja2 — nginx.conf.j2 template (Module 04 nginx config, templated)
worker_processes {{ nginx_worker_processes }};
events { worker_connections 1024; }
http {
    upstream app {
        server 127.0.0.1:{{ app_port }};
    }
    server {
        listen 80;
        server_name _;
        location / {
            proxy_pass http://app;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }
    }
}

7. Roles — Reusable Playbook Components

A role is a structured directory that packages tasks, handlers, templates, and variables for one concern. Just as Terraform modules are reusable infrastructure, Ansible roles are reusable configuration. Roles compose into a site playbook.

bash — create a role scaffold
ansible-galaxy init roles/docker   # creates standard directory structure
# roles/docker/
# ├── tasks/main.yml      ← what to do
# ├── handlers/main.yml   ← what to do when notified
# ├── templates/          ← Jinja2 files
# ├── vars/main.yml       ← non-overridable vars
# ├── defaults/main.yml   ← overridable defaults
# └── meta/main.yml       ← role dependencies

# site.yml: compose roles into a site playbook
yaml — site.yml (the capstone's one-command convergence)
---
- name: Configure all servers
  hosts: all
  become: true
  roles:
  - role: common          # OS hardening, NTP, syslog
  - role: docker          # Docker install + compose deploy (M08 app)
  - role: nginx           # nginx + templated reverse proxy (M04 config)
  - role: node_exporter   # installs Prometheus exporter (Module 16 setup!)

8. Ansible Vault — Encrypting Secrets in Git

bash — vault lifecycle
# Create an encrypted file
ansible-vault create group_vars/all/vault.yml
# Opens editor — add secrets:
# vault_db_password: "supersecret123"
# vault_jwt_secret: "mykey456"
# File is now AES-256 encrypted — safe to commit to Git

# Edit an existing vault
ansible-vault edit group_vars/all/vault.yml

# View without editing
ansible-vault view group_vars/all/vault.yml

# Run playbook with vault (prompt for password)
ansible-playbook -i inventory site.yml --ask-vault-pass

# In CI: store vault password in Jenkins credentials, pass via file
ansible-playbook -i inventory site.yml \
    --vault-password-file /run/secrets/ansible-vault-pass
⚠️ Vault password leaked in CI log

If the vault password appears in Jenkins console output, immediately rotate it: (1) decrypt all vault files (ansible-vault decrypt), (2) create a new password, (3) re-encrypt (ansible-vault encrypt), (4) update the Jenkins credential. Use Jenkins' built-in secret masking — never pass the vault password as a plain environment variable.

The handoff: Terraform outputs (VPC IDs, EC2 IPs) feed Ansible dynamic inventory. Ansible configures the nodes and installs the container runtime. Kubernetes (on EKS) then schedules the application workloads. They don't overlap — each owns its layer.

10. AWS Dynamic Inventory

Hardcoded IPs in a static inventory break the moment Terraform replaces an instance. The aws_ec2 inventory plugin queries AWS for current instances, grouped by tag — no more hardcoded IPs.

yaml — aws_ec2.yml dynamic inventory plugin
# aws_ec2.yml — Ansible reads this as a dynamic inventory source
plugin: aws_ec2
regions:
- us-east-1
filters:
  instance-state-name: running
  tag:Project: techitfactory     # only instances tagged with our project
keyed_groups:
- key: tags.Role                  # groups by the Role tag: tag_Role_web, tag_Role_db
  prefix: tag_role
compose:
  ansible_host: private_ip_address  # use private IP (instances are in private subnets)
bash — use dynamic inventory
# Verify what Ansible sees (debug — lists hosts and groups)
ansible-inventory -i aws_ec2.yml --graph
@all:
  |--@tag_role_web:
  |  |--10.0.3.10
  |  |--10.0.3.11
  |--@tag_role_db:
  |  |--10.0.4.10

# Run playbook against EC2 instances from dynamic inventory
ansible-playbook -i aws_ec2.yml site.yml \
    --vault-password-file /run/secrets/vault-pass

11. Real-Time Incidents & Troubleshooting

Incident 1 — Playbook Works on One Host, Fails on Another

StageDetails
SymptomThe nginx role succeeds on web-1 (Ubuntu 22.04), fails on web-2 (Amazon Linux 2) with "no apt package nginx".
Root CauseThe role used apt module unconditionally. Amazon Linux 2 uses yum. Drift in the server fleet caused an inconsistency the playbook exposed.
FixUse when: ansible_os_family == "Debian" for apt tasks and when: ansible_os_family == "RedHat" for yum, or use the package module (OS-agnostic).
PreventionStandardize the OS across the fleet. Use the same AMI for all instances (Terraform AMI data source). The playbook then documents the drift it just fixed.

Incident 2 — Hung Task (SSH Prompt / sudo Password)

yaml — prevent SSH/sudo hangs
# ansible.cfg — required for automation (non-interactive SSH)
[defaults]
host_key_checking = False   # don't prompt for unknown host key (M02 StrictHostKey analogy)
timeout = 30

[ssh_connection]
pipelining = True           # avoids sudo password prompt — requires requiretty off in sudoers
ssh_args = -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null

If pipelining is enabled but sudo still prompts, check the remote /etc/sudoers for requiretty — comment it out or add Defaults:ansible !requiretty. This is the same non-interactive discipline from Modules 02 and 09.

Interview Questions

Click a card to reveal a strong candidate answer.

Q: What's the difference between Ansible and Terraform?

They solve different problems in the same stack. Terraform provisions infrastructure — it creates EC2 instances, VPCs, load balancers. Ansible configures existing infrastructure — it installs software, manages files, ensures services are running. Terraform is declarative with a state file; Ansible is also declarative (idempotent modules) but stateless — it re-runs the desired state on each execution. In practice: Terraform → EC2 instances exist; Ansible → those instances are configured correctly.

Q: What is idempotency and give a concrete example?

Idempotency means running an operation multiple times produces the same result as running it once. Concrete Ansible example: apt: name=nginx state=present — if nginx is already installed, Ansible reports "ok" and does nothing. If it's not installed, it installs it and reports "changed." Compare to apt-get install nginx in a Bash script — running it twice tries to install again, outputs noise, might fail. The Ansible task is safe to run in a cron job or at every boot as a convergence check.

Q: What are handlers in Ansible and when do you use them?

Handlers are tasks that only run when notified and only once per play, regardless of how many tasks notify them. The classic use case: you have three tasks that might change an nginx config file, and you want nginx restarted after any of them change — but only once. Without handlers, you'd restart nginx three times if all three tasks changed. Handlers run at the end of the play, in notification order, once. This is cleaner and avoids multiple unnecessary restarts during long playbook runs.

Q: How does AWS dynamic inventory work?

The aws_ec2 inventory plugin calls the AWS EC2 API at playbook runtime to discover running instances. You configure it with filters (only instances in your project, in your region, in the running state) and keyed groups (group by tag values — so instances tagged Role=web appear in the tag_role_web group). Ansible then runs tasks against those dynamically-discovered groups. This means your playbook always targets the current fleet, even after Terraform replaces instances with new IPs.

Q: How do you handle Ansible Vault in CI?

Store the vault password in the CI system's secret store (Jenkins Credentials, GitHub Actions secret, AWS Secrets Manager). In the pipeline, write it to a temp file or pass it via the ANSIBLE_VAULT_PASSWORD_FILE environment variable. Never pass it as a command-line argument (--vault-pass gets logged by process lists). Enable secret masking in the CI system. Rotate the vault password if you suspect exposure — the encrypted vault files in Git remain safe as long as the password is secret.

Q: What are the pros and cons of Ansible being agentless?

Pros: nothing to install on managed nodes — you can run Ansible against a freshly-provisioned EC2 immediately. No agent to maintain, upgrade, or troubleshoot. Works on any host with SSH + Python. Cons: SSH overhead means Ansible is slower than agent-based tools (Chef/Puppet) at scale — each task opens SSH connections. For thousands of nodes, you need strategies like connection pipelining and fact caching. Also, Ansible can't continuously converge like an agent that runs on a schedule — you must re-run the playbook explicitly (or schedule it). For most deployments under 1000 nodes, agentless is the right tradeoff.
