1. CI/CD Concepts Recap & Practice

Module 01 introduced the theory of Continuous Integration. Now you have the tools to actually build it. A "Pipeline" is simply a robot that automatically runs the Bash commands you learned in Module 02, the Git pulls from Module 03, the builds from Module 05, and the uploads from Module 06.

Git Push (M03) → Jenkins Webhook → Build & Test (M05) → Publish to Nexus (M06)

The Social Contract: "Broken Build = Stop the Line." If the Jenkins test stage fails (exit code > 0), the developer stops writing new code and fixes the build immediately. This ensures the main branch is always deployable. This drives our DORA metrics (Deployment Frequency & Lead Time).

2. Installing Jenkins on Ubuntu

Let's provision a server next to our Nexus server. Jenkins runs on Java, and we install it as a systemd service (the exact ritual you learned in Module 06).

bash
# 1. Install Java 17
sudo apt update && sudo apt install openjdk-17-jre -y

# 2. Add Jenkins Debian Repo Key
curl -fsSL https://pkg.jenkins.io/debian-stable/jenkins.io-2023.key | sudo tee /usr/share/keyrings/jenkins-keyring.asc > /dev/null
echo deb [signed-by=/usr/share/keyrings/jenkins-keyring.asc] https://pkg.jenkins.io/debian-stable binary/ | sudo tee /etc/apt/sources.list.d/jenkins.list > /dev/null

# 3. Install Jenkins
sudo apt update && sudo apt install jenkins -y

# 4. Open Firewall (Jenkins defaults to port 8080)
sudo ufw allow 8080

# 5. Fetch Initial Admin Password
sudo cat /var/lib/jenkins/secrets/initialAdminPassword
💡 JVM Sizing for Jenkins

Jenkins is memory-hungry. A 1GB VM will crash. Use a 2GB+ VM, and edit /lib/systemd/system/jenkins.service (or /etc/default/jenkins depending on OS version) to set JAVA_OPTS="-Xmx1g" to limit the heap size.

3. Jenkins UI, Jobs & Credentials

The Jenkins UI is old but powerful. The most critical configuration screen for DevOps engineers is Manage Credentials.

The Golden Rule of Secrets

In Module 03, we learned never to commit passwords to Git. Your Jenkinsfile is stored in Git. Therefore, you can never hardcode the Nexus password in the Jenkinsfile. You store it securely inside Jenkins' encrypted Credential Store, and inject it at runtime.

  • Username with Password: (e.g., nexus-deploy-creds)
  • Secret Text: (e.g., a Slack Webhook token or GitHub PAT)
  • SSH Username with Private Key: (e.g., to connect to remote Agents)

5. Jenkinsfile — Declarative Syntax, Line by Line

A Jenkinsfile is written in Groovy. Modern Jenkins uses Declarative Pipeline syntax.

Jenkinsfile
/* Step 1: The Wrapper */
pipeline {
    /* Step 2: Where does this run? */
    agent any

    /* Step 3: Global settings */
    options {
        timeout(time: 1, unit: 'HOURS')
        timestamps()
    }

    /* Step 4: The Meat */
    stages {
        stage('Build') {
            steps {
                sh './gradlew build'
            }
        }
        stage('Deploy') {
            /* Only run this stage on the main branch */
            when { branch 'main' }
            steps {
                /* Manual Approval Gate! */
                input message: 'Deploy to Prod?'
                sh './deploy.sh'
            }
        }
    }

    /* Step 5: Cleanup & Notifications */
    post {
        always {
            cleanWs() # Delete the workspace to save disk space
        }
        failure {
            echo 'Send Slack message here!'
        }
    }
}
Code Explanation
  • pipeline {}: The mandatory root block. Everything inside this is Declarative Pipeline syntax.
  • agent any: Tells Jenkins to run this on any available worker node.
  • options {}: Global configuration. timeout guarantees the build won't hang forever, and timestamps adds time prefixes to console logs.
  • stages {}: The sequence of work blocks. Each stage shows up as a separate column in the Jenkins UI.
  • when { branch 'main' }: A condition. The "Deploy" stage will only execute if this commit is on the main branch.
  • input message: Pauses the pipeline and waits for a human to click "Proceed" or "Abort" in the UI.
  • post {}: Runs after all stages finish (or fail). cleanWs() deletes the workspace to free disk space.
Parallel & Stash

You can run stages simultaneously (e.g., UI tests and API tests at the same time) using parallel {}. To pass files between different agents, use stash (save) and unstash (load).

Note: You may see "Scripted Pipeline" (using the node {} block) in older codebases. You should always write new pipelines in Declarative format.

6. The Real Pipeline: Build → Test → Publish

Here is the complete pipeline that automates the Java build from Module 05 and pushes to the Nexus repository from Module 06.

Jenkinsfile
pipeline {
    agent any
    
    stages {
        stage('Checkout') {
            steps {
                # Pull the code from Git
                checkout scm
            }
        }
        
        stage('Test') {
            steps {
                sh './gradlew test'
            }
            post {
                always {
                    # Tell Jenkins to read the XML report to draw the test graph
                    junit 'build/test-results/**/*.xml'
                }
            }
        }
        
        stage('Build') {
            steps {
                sh './gradlew assemble'
            }
        }
        
        stage('Publish to Nexus') {
            steps {
                # Inject the Nexus credentials securely!
                withCredentials([usernamePassword(
                    credentialsId: 'nexus-deploy-creds', 
                    usernameVariable: 'NEXUS_USER', 
                    passwordVariable: 'NEXUS_PASS'
                )]) {
                    sh './gradlew publish -PnexusUser=${NEXUS_USER} -PnexusPass=${NEXUS_PASS}'
                }
            }
        }
    }
}
Code Explanation
  • checkout scm: A built-in Jenkins command that automatically clones the exact Git commit that triggered the webhook.
  • sh './gradlew test': Runs the unit tests via Bash. If tests fail (exit code > 0), Jenkins instantly stops the pipeline and marks it RED.
  • junit '...': The post block reads the generated XML test files and renders a beautiful test-success-trend graph on the Jenkins dashboard.
  • withCredentials([...]): Securely fetches the Nexus username and password from Jenkins Credentials, assigning them to local variables (NEXUS_USER). It masks these values from the console logs.
  • sh './gradlew publish ...': Passes those secure variables into the Gradle build to authenticate against Nexus.

7. Multi-Branch Pipelines, Webhooks & Triggers

A "Multibranch Pipeline" scans your Git repository (Module 03). For every branch that contains a Jenkinsfile, Jenkins automatically creates a pipeline job. If you open a Pull Request, Jenkins builds it. This is how GitHub "Require CI to pass before merging" actually works.

How does Jenkins know when you push code?
MethodHow it worksVerdict
WebhookGitHub sends an HTTP POST to YOUR_JENKINS/github-webhook/ the millisecond you push.Best. Instant and efficient.
Poll SCMJenkins runs git fetch every 5 minutes to see if anything changed.Terrible. Wastes CPU and API rate limits.
Cron (Scheduled)Build runs at 2am every night (e.g., H/15 * * * *).Good for nightly deep-scans, not for CI. Note the H (Hash) instead of * to distribute load!

8. Shared Libraries (Don't Repeat Yourself)

Imagine you have 50 microservices. Copying and pasting the 50-line Jenkinsfile above into 50 different Git repos is a nightmare. If you need to change the Nexus URL, you have to open 50 Pull Requests.

Shared Libraries allow you to write Groovy code in a central Git repo, and call it from any Jenkinsfile.

Jenkinsfile (Using Library)
# Import the library at the top of the file
@Library('devops-lib@v1') _

pipeline {
    agent any
    stages {
        stage('Build & Publish') {
            steps {
                # Call the custom function we wrote in the library!
                buildAndPublishJava(repoUrl: 'nexus.mycompany.com')
            }
        }
    }
}
Code Explanation
  • @Library('devops-lib@v1') _: Tells Jenkins to clone the Git repository named "devops-lib" (version/tag "v1") before starting the pipeline. The underscore _ imports everything globally.
  • buildAndPublishJava(...): This is not a built-in Jenkins command. It is a custom function (a .groovy script) that your DevOps team wrote and stored in the shared library. It hides the 50 lines of complex Gradle and Credentials logic from the developer, leaving a clean, simple interface.

This is the seed of "Platform Engineering" (Module 19), where DevOps provides abstract, easy-to-use paved roads for developers.

9. Agents & Distributed Builds

Jenkins Controller (Brain) ↔ Agent 1 (Ubuntu/Java)

Jenkins Controller (Brain) ↔ Agent 2 (Windows/C++)
The Golden Rule of Architecture

The Controller builds nothing. The Controller's only job is to serve the UI, manage credentials, and assign tasks. If you run npm install on the Controller, it will run out of memory and crash the UI for the whole company. We set the Controller's Executors to 0.

We connect worker machines ("Agents" or "Nodes") via SSH (Module 02 callback!). We assign them labels. In the Jenkinsfile, we ask for them:

Groovy
agent { label 'linux && java17' }

10. GitHub Actions & GitLab CI (Alternatives)

Why learn Jenkins if GitHub Actions is newer? Because the concepts transfer 1:1. If you understand Jenkins, you can learn GitHub Actions in 20 minutes. Here is the exact same pipeline in GitHub Actions format (.github/workflows/ci.yml):

yaml
name: Java CI
on:
  push:
    branches: [ "main" ]

jobs:
  build:
    runs-on: ubuntu-latest  # This is the "agent"
    steps:
    - uses: actions/checkout@v3
    
    - name: Set up JDK 17
      uses: actions/setup-java@v3
      with:
        java-version: '17'
        
    - name: Build and Publish
      env:
        NEXUS_USER: ${{ secrets.NEXUS_USER }}  # This is "withCredentials"
        NEXUS_PASS: ${{ secrets.NEXUS_PASS }}
      run: ./gradlew publish -PnexusUser=$NEXUS_USER -PnexusPass=$NEXUS_PASS
Code Explanation
  • on: push: Equivalent to the Jenkins GitHub Webhook trigger.
  • runs-on: ubuntu-latest: Equivalent to agent { label 'linux' } in Jenkins.
  • uses: actions/checkout@v3: Equivalent to checkout scm.
  • ${{ secrets.NEXUS_USER }}: Equivalent to Jenkins' withCredentials block. Fetches the secret from GitHub Repository Secrets and injects it as an environment variable securely.

11. Real-Time Incidents & Troubleshooting

When the CI server breaks, the entire engineering organization is blocked. Here is how you debug the most common real-world Jenkins failures.

Incident 1: "Jenkins is down" (UI dead)

Symptom: 502 Bad Gateway or the UI won't load.

Investigation: SSH into the controller. Run systemctl status jenkins. Then read the logs: journalctl -u jenkins -n 50. You see java.lang.OutOfMemoryError: Java heap space after a plugin update.

Fix: Increase memory in the systemd override (-Xmx2g), or rollback the plugin from /var/lib/jenkins/plugins/*.bak.

Prevention: Setup monitoring (Module 16) to alert when JVM Heap is > 85%.

Incident 2: "The build hangs forever"

Symptom: A pipeline has been running for 4 hours on a 2-minute stage.

Investigation: Is it waiting for manual input, or is it a network hang? Go to YOUR_JENKINS/threadDump to see exactly what Java is doing. It's stuck trying to download from a dead Nexus proxy.

Fix: Abort the build manually.

Prevention: NEVER write a pipeline without a timeout. options { timeout(time: 15, unit: 'MINUTES') }.

Incident 3: "Disk full, all builds failing"

Symptom: Builds fail immediately with No space left on device.

Investigation: (Module 02 forensics). Run df -h. See `/` is at 100%. Run sudo du -sh /var/lib/jenkins/*. Discover /workspaces and /jobs are 500GB.

Fix: Delete old workspaces manually. Configure buildDiscarder(logRotator(numToKeepStr: '10')) on jobs to only keep the last 10 build logs. Add cleanWs() to the post { always {} } block in all Jenkinsfiles.

Incident 4: "Webhook doesn't trigger"

Symptom: A developer pushes code to GitHub, but Jenkins does nothing.

Investigation (The 4-layer check):
1. GitHub side: Go to repo Settings > Webhooks. Look at "Recent Deliveries". Is it getting a 200 OK or a 403 Forbidden?
2. Network side: Did the Jenkins IP change? Is the AWS Security Group blocking port 8080 from GitHub IPs? (Module 04).
3. Path side: Is the webhook URL exactly /github-webhook/ (with the trailing slash)?
4. Job side: Did the Multibranch Pipeline actually scan and find the branch?

Incident 5: "Credentials leaked in console log"

Symptom: Someone added set -x to their bash script to debug it, and now the Nexus password is printed in plain text in the Jenkins log.

Fix: IMMEDIATE ROTATION. Treat the password as compromised. Generate a new password in Nexus, update the Jenkins Credential Store, and delete the offending Jenkins build log.

Prevention: Use withCredentials which automatically masks variables with *** in logs, or wrap the step in MaskPasswordsBuildWrapper.

Incident 6: "Works on Controller, fails on Agent"

Symptom: The build worked yesterday. Today it ran on agent-02 and failed saying mvn: command not found.

Investigation: Environmental drift. agent-02 was provisioned yesterday and nobody installed Java/Maven on it.

Fix/Prevention: Stop patching pets. We will solve this in Module 09 using Docker Agents, where the pipeline defines its own environment: agent { docker { image 'maven:3-eclipse-temurin-17' } }.

12. Backup & Jenkins Hygiene

Jenkins is the most frequently attacked service in a company because it literally holds all the keys to AWS, Nexus, and production databases.

  • Backup: $JENKINS_HOME (usually /var/lib/jenkins) contains EVERYTHING. Your jobs, credentials, and plugins. Backing this directory up daily is mandatory. Advanced teams use JCasC (Jenkins Configuration as Code) to store settings in Git.
  • Hardening: Disable Anonymous Read access. Enable CSRF protection. Use "Matrix-based security" to ensure Developers can trigger builds but cannot configure jobs or read credentials.

💬 Real DevOps Interview Questions

Jenkins is the most asked-about tool in DevOps interviews. Be ready.

Q: What is the difference between Declarative and Scripted pipelines?

Declarative uses the pipeline {} block with strict, readable syntax (stages, steps, post). Scripted uses the node {} block and is essentially raw Groovy code. Declarative is the modern standard because it is easier to read, validate, and maintain.

Q: How do webhooks work, and how do you debug one that fails?

GitHub sends an HTTP POST request to Jenkins when code is pushed. If it fails, I check the 4 layers: GitHub's delivery log (did it send?), the network firewall (is 8080 open?), the URL path (is it /github-webhook/?), and Jenkins (is the job configured to listen?).

Q: Where do you store secrets, and how do you keep them out of logs?

Secrets are stored in the Jenkins Credential Store, never in the Git repo. We inject them into the pipeline using the withCredentials block, which automatically masks the secret value with *** in the console output.

Q: Why should you never run builds on the Jenkins Controller?

Builds consume CPU, memory, and disk IO. If a runaway npm install eats all the RAM, the Controller will crash, taking down the UI and breaking CI for the entire company. The Controller should only manage jobs; Agents do the actual work.

Q: What problem does the `post { always { cleanWs() } }` block solve?

It solves the "Disk Full" incident. Jenkins keeps the workspace (source code + build artifacts) on the agent's disk after the build finishes. Running cleanWs() ensures the workspace is deleted, preventing the agent from running out of space over time.

Q: Why use a Multibranch Pipeline instead of a Freestyle job?

A Multibranch Pipeline automatically discovers branches and Pull Requests, building them instantly without human intervention. Freestyle jobs require manual UI clicks to create, can't be version-controlled, and don't scale to feature-branch workflows.

Q: Jenkins disk is full at 3am. Walk me through your response.

I log into the server and run df -h and du -sh /var/lib/jenkins/* to find the culprit. It's usually old workspaces or archived artifacts. I manually delete them to restore service, then configure buildDiscarder and cleanWs() in the pipeline to prevent it from happening again.

Q: How would you migrate a Jenkins pipeline to GitHub Actions?

The concepts are 1:1. Jenkins Agents become GH Runners. Jenkins Stages become GH Jobs. Shared Libraries become Composite Actions. withCredentials becomes GitHub Secrets. I would translate the Groovy steps into YAML run: commands.
