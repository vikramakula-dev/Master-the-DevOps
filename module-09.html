2. Building Docker Images in Jenkins

Before any pipeline code: the Jenkins agent must be able to run Docker. This is the #1 setup stumble.

bash — on the Jenkins agent
# Install Docker (Module 08), then let the jenkins user use it:
sudo usermod -aG docker jenkins     # -aG! (Module 02 warning applies)
sudo systemctl restart jenkins      # group changes need a fresh login/session

# Verify AS the jenkins user:
sudo -u jenkins docker ps           # must work without sudo
🚨 docker group = root

Anyone in the docker group can trivially become root on that machine (mount / into a container). This is exactly why Module 07's rule exists: agents build, the controller never does. Your controller holds all credentials — don't hand it root-equivalent Docker access too.

The Build Stage
Jenkinsfile — build stage
stage('Build Image') {
    steps {
        sh 'docker build -t myapp:${BUILD_NUMBER} .'
    }
}

The Docker Pipeline plugin offers docker.build("myapp") as an alternative, but plain sh steps win in practice: what runs in CI is exactly what you can run and debug in a terminal — no plugin behavior to second-guess.

Why CI Builds Are Slower Than Local
  • Cold layer cache — a fresh agent has no cached layers, so every RUN executes. Module 08's layer-ordering lesson (COPY package*.json first) matters more in CI, not less.
  • .dockerignore — without it, the whole workspace (including .git/ and test reports) ships to the daemon as build context on every build.
Level-Up: Build Stages Inside Containers

Module 07's "works on controller, fails on agent" incident had a band-aid fix (install the JDK). The real fix: run the stage inside a container — the toolchain ships with the pipeline.

Jenkinsfile — containerized stage
stage('Test') {
    agent { docker { image 'node:20-alpine' } }   // stage runs IN this image
    steps {
        sh 'npm ci && npm test'                     // exact Node version, every time
    }
}

No more toolchain drift: the agent only needs Docker itself.

3. Pushing to Docker Hub / ECR

Three registry options, one rule: login happens inside the pipeline with stored credentials — never hardcoded, never with -p on the command line.

Jenkinsfile — Docker Hub push, done right
stage('Push') {
    steps {
        withCredentials([usernamePassword(
            credentialsId: 'dockerhub-creds',       // from Jenkins credential store (M07)
            usernameVariable: 'DOCKER_USER',
            passwordVariable: 'DOCKER_PASS'
        )]) {
            // --password-stdin keeps the secret OUT of the console log & process list
            sh 'echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin'
            sh 'docker push myuser/myapp:${IMAGE_TAG}'
        }
    }
}
ECR — The AWS Way

ECR logins are short-lived tokens fetched with the AWS CLI. Decode the registry URL once and it stops being scary:

bash — ECR push flow
#  123456789012.dkr.ecr.ap-south-1.amazonaws.com/myapp:1.4.2
#  └─ account ID ┘        └─ region ┘             └repo┘└tag┘

# 1. Repo must exist first (once):
aws ecr create-repository --repository-name myapp

# 2. Login (token valid 12h):
aws ecr get-login-password --region ap-south-1 \
  | docker login --username AWS --password-stdin 123456789012.dkr.ecr.ap-south-1.amazonaws.com

# 3. Tag with the FULL registry path, then push:
docker tag myapp:1.4.2 123456789012.dkr.ecr.ap-south-1.amazonaws.com/myapp:1.4.2
docker push 123456789012.dkr.ecr.ap-south-1.amazonaws.com/myapp:1.4.2
RegistryAuthBest For
Docker HubUsername + access tokenPublic images, learning
AWS ECRIAM + 12h token via CLIAnything deployed on AWS (the capstone)
Nexus (Module 06)Local users/rolesSelf-hosted, air-gapped, one warehouse for all artifacts
⚠️ "no basic auth credentials" in CI (but works locally)

Locally, docker login saved your token to ~/.docker/config.json — for your user. The pipeline runs as jenkins on another machine with no such file. Every pipeline run must do its own login inside the stage. This bites everyone exactly once.

4. Dynamic Versioning

Module 05 promised you'd always be able to answer "which code is running in prod?" This section keeps that promise. First, what NOT to do:

Tag StrategyExampleVerdict
latestmyapp:latest❌ Mutable — you can never prove what's deployed. Humans only.
Build numbermyapp:42🟡 Unique, but meaningless outside Jenkins
Git SHAmyapp:a1b2c3d🟢 Exact traceability to a commit
SemVermyapp:1.4.2🟢 Human-meaningful (Module 03)
Composite ⭐myapp:1.4.42-a1b2c3d✅ Version + build + commit in one tag
Jenkinsfile — computing the tag
environment {
    // GIT_COMMIT is provided by the checkout; take the short form
    GIT_SHORT = "${env.GIT_COMMIT.take(7)}"
    IMAGE_TAG = "1.4.${env.BUILD_NUMBER}-${GIT_SHORT}"
    IMAGE     = "123456789012.dkr.ecr.ap-south-1.amazonaws.com/myapp"
}
stages {
    stage('Build Image') {
        steps {
            sh 'docker build -t $IMAGE:$IMAGE_TAG .'
            // same image, second tag — 'latest' for humans browsing the registry
            sh 'docker tag $IMAGE:$IMAGE_TAG $IMAGE:latest'
        }
    }
}
💡 Tags Are Free

A tag is just a pointer to an image digest (Module 08 layers callback) — tagging the same image five ways costs zero bytes. Push the immutable composite tag for machines and latest for humans. Deploy scripts must only ever reference the composite.

5. Deploying via SSH from the Pipeline

Your Module 02/04 SSH skills, now non-interactive. The pipeline gets its own deploy key: a dedicated keypair whose public half lives only on the deploy server, under a dedicated deploy user.

Jenkinsfile — deploy stage
stage('Deploy') {
    steps {
        // sshagent loads the private key from the Jenkins credential store
        sshagent(['deploy-server-key']) {
            sh '''
                ssh -o StrictHostKeyChecking=accept-new deploy@10.0.1.50 \
                    "cd /opt/myapp && ./deploy.sh ${IMAGE_TAG}"
            '''
        }
    }
}
The Deploy Script Lives in the Repo

Notice the pipeline calls ./deploy.sh — it does not inline ten SSH'd commands in the Jenkinsfile. The script is versioned with the app (Module 03 discipline), reviewable in PRs, and testable by hand on the server.

deploy.sh — on the server, in the repo
#!/bin/bash
set -euo pipefail                    # Module 02 — always

TAG="${1:?Usage: deploy.sh <image-tag>}"   # refuse to run without a tag

echo "IMAGE_TAG=$TAG" > .env          # compose reads the tag from here
docker compose pull app              # fetch only the new image
docker compose up -d                 # recreate only what changed
docker image prune -f                # keep the disk alive (Incident 2!)
⚠️ StrictHostKeyChecking — don't just disable it

Host-key checking is what stops a man-in-the-middle from receiving your deploy commands (and secrets). accept-new trusts the key on first contact and pins it after — reasonable for a server you just provisioned. no disables protection entirely; better: pre-provision known_hosts on the agent (see Incident 3).

6. Compose Deployment & Zero-Downtime

The compose file references the tag through an environment variable — the pipeline writes .env, compose does the rest:

docker-compose.yml — on the server
services:
  app:
    image: 123456789012.dkr.ecr.ap-south-1.amazonaws.com/myapp:${IMAGE_TAG}
    restart: unless-stopped
    ports:
      - "3000:3000"
    env_file: app.env                # runtime secrets — NOT in the image (M08)
    healthcheck:                     # Module 08 payoff — gates the restart
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 10s
      timeout: 3s
      retries: 3

docker compose up -d compares desired vs running state and recreates only services whose image or config changed — your database container isn't touched by an app deploy.

Honest Talk: What "Zero-Downtime" Means Here
  • Compose up -d = stop old container → start new one. There is a gap of a few seconds.
  • You can shrink it (start-new-then-stop-old with two container names and an nginx flip — the Module 04 reverse proxy trick), but it's manual plumbing.
  • True rolling updates — new pods up, health-checked, old drained, automatic rollback — is exactly what Kubernetes does natively. That's Module 11, and now you know precisely why it exists.
Rollback: One Parameter Away
bash — because every deploy is an immutable tag…
# …rollback is just deploying the previous tag:
./deploy.sh 1.4.41-9f8e7d6          # the tag that worked an hour ago

In Jenkins: a parameterized job (IMAGE_TAG as a parameter, Module 07) that runs only the deploy stage. This is the capstone's rollback strategy in miniature — rollback = redeploy a known-good version, never "fix forward under pressure."

7. Post-Deploy Verification (Smoke Tests)

A pipeline isn't done when it deploys — it's done when it verifies. "Pipeline green, site down" (Incident 6) is what happens when this section is skipped.

Jenkinsfile — the stage most pipelines are missing
stage('Smoke Test') {
    steps {
        // -f: exit non-zero on HTTP errors → fails the stage (M02 exit codes!)
        // retry+sleep: give the container a moment to become healthy
        sh '''
            for i in $(seq 1 10); do
                curl -fs http://10.0.1.50:3000/health && exit 0
                echo "Attempt $i/10 failed, retrying in 5s..."
                sleep 5
            done
            echo "SMOKE TEST FAILED — app never became healthy"
            exit 1
        '''
    }
}
post {
    failure {
        // smoke test failed? redeploy the last known-good tag automatically
        sh 'ssh deploy@10.0.1.50 "cd /opt/myapp && ./deploy.sh $(cat last-good-tag)"'
    }
    success {
        sh 'ssh deploy@10.0.1.50 "echo $IMAGE_TAG > /opt/myapp/last-good-tag"'
    }
}
💡 Notify on State Change, Not Every Build

A Slack ping for every green build trains everyone to ignore Slack pings (alert fatigue — a core Module 16 theme). Notify when the build breaks and when it's fixed. Silence means healthy.

8. Pipeline Security

This pipeline now holds keys to GitHub, a registry, and a production server. Audit every credential against least privilege (the Module 06/07 theme):

CredentialScope It Should HaveScope It Should NOT Have
GitHub tokenRead this repo + webhooksOrg-wide, delete, admin
Registry credsPush/pull this one repositoryAccount-wide access token
SSH deploy keydeploy user on one serverroot, your personal key, password auth
AWS credentialsECR push to one repo (IAM policy)AdministratorAccess "just for now"
Secrets Never Bake Into Images
bash — the leak you can see
# If a secret was passed as a build ARG, it's in the layer history FOREVER:
docker history --no-trunc myapp:1.4.2 | grep -i token
# /bin/sh -c #(nop) ARG API_TOKEN=ghp_Xk29fPq...   ← anyone who pulls sees this

# Rule: build args for build-time config only (never secrets).
# Runtime secrets arrive via env_file / secret manager at 'docker run' time.
🚨 If a Secret Leaks

Same protocol as Modules 03 and 07, now for images: the token in a pushed layer is compromised the moment the image is pullable. Revoke → rotate → then clean up. Deleting the image tag doesn't delete pulled copies.

Image signing (awareness): Docker Content Trust / cosign let you cryptographically prove an image came from your pipeline and wasn't tampered with. We wire this up properly in Module 18 — for now, note the commented-out Trivy stage in the full pipeline below. You'll uncomment it there.

    options {
        timeout(time: 20, unit: 'MINUTES')           // M07 Incident 2

            steps {
                sh 'docker build -t $IMAGE:$IMAGE_TAG .'       // §2
                sh 'docker tag $IMAGE:$IMAGE_TAG $IMAGE:latest' // §4 — humans only
            }
        }
        // stage('Scan') {                                     // Module 18 — uncomment there
        //     steps { sh 'trivy image --exit-code 1 $IMAGE:$IMAGE_TAG' }
        // }
        stage('Push') {
            steps {
                sh '''aws ecr get-login-password --region ap-south-1 \

                sh 'docker push $IMAGE:$IMAGE_TAG && docker push $IMAGE:latest'

        stage('Deploy') {

                        "cd /opt/myapp && ./deploy.sh ${IMAGE_TAG}"'''
                }
            }
        }
        stage('Smoke Test') {
            when { branch 'main' }
            steps { sh './ci/smoke-test.sh http://10.0.1.50:3000/health' }  // §7
        }
    }

    post {
        fixed   { slackSend color: 'good',   message: "✅ ${env.JOB_NAME} is healthy again" }
        failure { slackSend color: 'danger', message: "🔴 ${env.JOB_NAME} #${env.BUILD_NUMBER} failed: ${env.BUILD_URL}" }

The Same Machine in GitHub Actions

Module 07's promise — concepts transfer 1:1:

.github/workflows/deploy.yml — equivalent pipeline
name: ci-cd
on: { push: { branches: [main] } }

jobs:
  build-deploy:
    runs-on: ubuntu-latest            # agent — but disposable, Docker pre-installed

      - uses: actions/checkout@v4

          echo "TAG=$TAG" >> $GITHUB_ENV
      - uses: aws-actions/amazon-ecr-login@v2   # login via OIDC — no stored AWS keys!

          ssh deploy@$SERVER "cd /opt/myapp && ./deploy.sh $TAG"
💡 Spot the Upgrade

That amazon-ecr-login step uses GitHub OIDC — no long-lived AWS keys stored anywhere (the capstone's authentication model, Module 13/18 territory). Jenkins can do this too, but Actions makes it one line. Each tool has its strengths.

10. Real-Time Incidents & Troubleshooting

Six incidents you will genuinely hit. Format: symptom → investigation → root cause → fix → prevention.

Incident 1: "permission denied … /var/run/docker.sock"
console log
+ docker build -t myapp:42 .
ERROR: permission denied while trying to connect to the Docker daemon socket
    at unix:///var/run/docker.sock
  • Investigate: sudo -u jenkins docker ps on the agent — same error confirms it's group membership, not the pipeline.
  • Root cause: jenkins user not in the docker group, or the agent process started before the group change.
  • Fix: sudo usermod -aG docker jenkins && sudo systemctl restart jenkins (the running session must be restarted to pick up the group).
  • Never: chmod 777 /var/run/docker.sock — that hands root to every user on the machine (§2 danger box).
Incident 2: "no space left on device" — the CI Disk-Full, Round 3
  • Symptom: Builds fail mid-docker build; the M02 reflex df -h shows /var/lib/docker at 100%.
  • Investigate: docker system df — hundreds of dangling images: every build made myapp:<n> and nothing ever cleaned up.
  • Fix now: docker system prune -af --filter "until=72h".
  • Prevention: docker image prune -f in post { always } + a nightly prune cron (Module 02). You've now solved disk-full on a server (M02), Jenkins (M07), and Docker (M08/09) — see the pattern? Everything that produces artifacts needs a retention policy.
Incident 3: Deploy Stage Hangs Forever
  • Symptom: Pipeline stuck at Deploy for 40 minutes, no output. (Your M07 timeout option eventually kills it — that's why it's there.)
  • Root cause: First-ever SSH to this server — The authenticity of host … can't be established. Continue? (yes/no) — and no human exists to type yes (Module 02's cron lesson, striking again in CI).
  • Fix: -o StrictHostKeyChecking=accept-new, or better: ssh-keyscan 10.0.1.50 >> ~jenkins/.ssh/known_hosts when provisioning the agent.
  • Prevention: Any prompt = a hang in CI. Audit every command for interactivity (-y, --password-stdin, --no-input).
Incident 4: Wrong Image Running in Production
  • Symptom: You deployed a fix; the bug is still there. docker inspect --format '{{.Image}}' app shows a digest that doesn't match your new build.
  • Root cause: Compose file said image: myapp:latest; the server's cached latest was three weeks old, and nobody ran pull.
  • Fix & prevention: Deploy scripts reference only immutable composite tags (§4) and always compose pull before up. latest is for humans browsing the registry — never for deploys.
Incident 5: Secret Leaked in an Image Layer
  • Symptom: Security review runs docker history --no-trunc on your image — an npm token is sitting in an ARG layer (§8's demo, but in prod).
  • Fix: The M03/M07 protocol: revoke the token first, rotate, then rebuild the image without the ARG and re-push. Deleting the tag does nothing for already-pulled copies.
  • Prevention: Secrets only at runtime (env_file), never at build time; a pipeline grep-gate for ARG.*(TOKEN|KEY|PASS) catches it before push.
Incident 6: Pipeline Green, Site Down
  • Symptom: Every stage passed; customers see 502 (Module 04: your upstream is dead). docker ps on the server: the app container is restart-looping.
  • Investigate: docker logs app → crash on a missing env var that exists in dev but not in the server's app.env.
  • Root cause (the real one): The pipeline's definition of "done" was "deploy command exited 0" — nobody asked the app if it was alive.
  • Fix & prevention: The §7 smoke-test stage with retry + automatic rollback to last-good-tag. This incident is why §7 exists.

🛠️ Real-Time Project: Complete CI/CD — Build → Dockerize → Deploy

Jenkins pipeline that checks out code from GitHub → runs tests in a container → builds a Docker image with a dynamic version tag → pushes to ECR (or Docker Hub) → SSHs into a server → deploys via docker-compose → verifies with a smoke test. The Phase 2 graduation project.

Prepare the Deploy Target

A second VM or free-tier EC2 Ubuntu instance. Run the full hardening + Docker setup — this is your Module 02/04/08 knowledge as one script:

bash — server-setup.sh (run once on the deploy server)
#!/bin/bash
set -euo pipefail

# --- 1. Docker + Compose (Module 08) ---
sudo apt update && sudo apt install -y docker.io docker-compose-v2
sudo systemctl enable --now docker

# --- 2. Dedicated deploy user (Module 02: never deploy as root) ---
sudo useradd -m -s /bin/bash deploy
sudo usermod -aG docker deploy          # deploy user drives Docker

# --- 3. Key-only SSH for the deploy user ---
sudo mkdir -p /home/deploy/.ssh
# paste the PUBLIC half of the pipeline's deploy key:
echo "ssh-ed25519 AAAA...jenkins-deploy-key" | sudo tee /home/deploy/.ssh/authorized_keys
sudo chmod 700 /home/deploy/.ssh && sudo chmod 600 /home/deploy/.ssh/authorized_keys
sudo chown -R deploy:deploy /home/deploy/.ssh

# --- 4. Firewall (Module 04: SSH first, then app port) ---
sudo ufw allow 22/tcp && sudo ufw allow 3000/tcp
sudo ufw --force enable

# --- 5. App directory owned by deploy ---
sudo mkdir -p /opt/myapp && sudo chown deploy:deploy /opt/myapp
echo "✔ Server ready. Now place docker-compose.yml, deploy.sh, app.env in /opt/myapp"

Then place the three runtime files in /opt/myapp (as the deploy user):

/opt/myapp/docker-compose.yml
services:
  app:
    image: ${ECR_REGISTRY}/myapp:${IMAGE_TAG}   # both come from .env, written by deploy.sh
    restart: unless-stopped
    ports:
      - "3000:3000"
    env_file: app.env                          # runtime secrets — never in the image
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 10s
      timeout: 3s
      retries: 3
      start_period: 15s
/opt/myapp/deploy.sh
#!/bin/bash
# deploy.sh <image-tag> — versioned in the app repo, synced here by the pipeline
set -euo pipefail

TAG="${1:?Usage: deploy.sh <image-tag>}"
REGISTRY="123456789012.dkr.ecr.ap-south-1.amazonaws.com"

cd /opt/myapp

# ECR login from the server (instance role or configured AWS CLI)
aws ecr get-login-password --region ap-south-1 \
    | docker login --username AWS --password-stdin "$REGISTRY"

# Point compose at the new tag
{
    echo "ECR_REGISTRY=$REGISTRY"
    echo "IMAGE_TAG=$TAG"
} > .env

docker compose pull app          # fetch only the new image
docker compose up -d             # recreate only what changed
docker image prune -f            # Incident 2 prevention

echo "✔ Deployed $TAG"

Stages 4–5: Deploy & Smoke Test

The deploy + verify pair, gated to main, with automatic rollback. First the smoke-test script (lives in the repo under ci/):

ci/smoke-test.sh
#!/bin/bash
# smoke-test.sh <health-url> — pipeline gate: exits 1 if app never becomes healthy
set -euo pipefail
URL="${1:?Usage: smoke-test.sh <health-url>}"

for i in $(seq 1 10); do
    if curl -fs --max-time 5 "$URL" > /dev/null; then
        echo "✔ Healthy on attempt $i"
        exit 0
    fi
    echo "Attempt $i/10 failed — retrying in 5s"
    sleep 5
done

echo "✖ SMOKE TEST FAILED — $URL never became healthy"
exit 1
Jenkinsfile — deploy, verify, rollback, notify
stage('Deploy') {
    when { branch 'main' }
    steps {
        sshagent(['deploy-server-key']) {
            sh '''
                # sync the versioned deploy files, then run the deploy
                scp -o StrictHostKeyChecking=accept-new \
                    deploy.sh docker-compose.yml deploy@10.0.1.50:/opt/myapp/
                ssh deploy@10.0.1.50 "cd /opt/myapp && ./deploy.sh ${IMAGE_TAG}"
            '''
        }
    }
}
stage('Smoke Test') {
    when { branch 'main' }
    steps {
        sh './ci/smoke-test.sh http://10.0.1.50:3000/health'
    }
}

post {
    success {
        // only a verified deploy becomes the rollback target
        sshagent(['deploy-server-key']) {
            sh 'ssh deploy@10.0.1.50 "echo ${IMAGE_TAG} > /opt/myapp/last-good-tag"'
        }
    }
    failure {
        // smoke test failed → put the last verified version back
        sshagent(['deploy-server-key']) {
            sh '''ssh deploy@10.0.1.50 \
                "cd /opt/myapp && ./deploy.sh \$(cat last-good-tag)" || true'''
        }
        withCredentials([string(credentialsId: 'slack-webhook', variable: 'HOOK')]) {
            sh '''curl -s -X POST -H 'Content-type: application/json' \
                --data "{\\"text\\":\\"🔴 ${JOB_NAME} #${BUILD_NUMBER} failed — rolled back. ${BUILD_URL}\\"}" $HOOK'''
        }
    }
    fixed {
        withCredentials([string(credentialsId: 'slack-webhook', variable: 'HOOK')]) {
            sh '''curl -s -X POST -H 'Content-type: application/json' \
                --data "{\\"text\\":\\"✅ ${JOB_NAME} is healthy again\\"}" $HOOK'''
        }
    }
    always { sh 'docker image prune -f' }
}

💬 Real DevOps Interview Questions

"Walk me through your CI/CD pipeline" is the single most common DevOps interview request. This module is your answer — practice saying it out loud.

Q: Walk me through your CI/CD pipeline end-to-end.

Narrate §1's diagram with tool names: push to GitHub fires a webhook to Jenkins; a multibranch pipeline runs tests inside a pinned container image; on main, we build a Docker image tagged version.build-sha, push to ECR with short-lived credentials, SSH to the server with a scoped deploy key, and docker compose up -d with the new tag. A smoke-test stage curls the health endpoint with retries — on failure it auto-rolls-back to the last known-good tag and alerts the channel. Bonus points: mention what you'd upgrade (Kubernetes rolling updates, GitOps pull-based deploys).

Q: How do you version Docker images, and why not just use latest?

latest is a mutable pointer — it can silently mean different images on different machines, so you can never prove what's deployed, and cached stale copies cause "I deployed but nothing changed" (Incident 4). We use an immutable composite tag — semantic version + build number + short git SHA (e.g. 1.4.42-a1b2c3d) — giving exact commit traceability. latest is additionally tagged for humans browsing the registry, but deploy scripts never reference it.

Q: How does your pipeline authenticate to the registry safely?

Credentials live in the Jenkins credential store, injected per-stage with withCredentials, and login uses --password-stdin so the secret never appears in the process list or console log. For ECR, the AWS CLI issues a 12-hour token — no static registry password at all. The credential is scoped to push/pull on one repository, not account-wide. Best answer mentions the next level: OIDC federation (GitHub Actions → AWS) with no stored cloud keys whatsoever.

Q: Your build fails with "permission denied on /var/run/docker.sock". Fix it.

The CI user can't reach the Docker daemon socket. Fix: add it to the docker group (usermod -aG docker jenkins) and restart the agent service so the session picks up the group. Crucially, say what you wouldn't do: chmod 777 on the socket — docker-group access is root-equivalent, which is also why builds belong on agents, never on the controller that holds all the credentials.

Q: How do you deploy with minimal downtime using Compose — and where's the ceiling?

compose pull then up -d recreates only changed services, and healthchecks gate readiness — but there's still a stop-old/start-new gap of seconds. You can DIY blue-green with two container names and an nginx flip, but that's plumbing. The honest answer: true rolling updates with surge, readiness gating, and automatic rollback are native Kubernetes features — Compose is fine for a single host; past that, you've outgrown it.

Q: The pipeline is green but production is down. What's missing?

Post-deploy verification. The pipeline's definition of done was "deploy command exited 0" — nobody asked the application if it's alive. The fix is a smoke-test stage: curl the health endpoint with a retry loop, fail the pipeline if it never becomes healthy, and auto-rollback to the last known-good tag. A pipeline isn't done when it deploys; it's done when it verifies.

Q: How do you roll back a bad deployment?

Because every deploy is an immutable tag, rollback is just deploying the previous tag — a parameterized Jenkins job that runs only the deploy stage with IMAGE_TAG as input, taking under a minute. We track last-good-tag on the server (updated only after the smoke test passes) so rollback can even be automatic. The anti-pattern: "fixing forward" under pressure — debugging in production while customers wait.

Q: Why run build stages inside Docker containers?

It eliminates toolchain drift between agents: agent { docker { image 'node:20-alpine' } } means the exact Node version ships with the pipeline definition, in Git, instead of being hand-installed on each agent ("works on this agent, fails on that one"). Agents then need only Docker itself, making them interchangeable and trivially replaceable — the same reasoning that later makes Kubernetes build agents attractive.
