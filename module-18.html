1. Shift Left & the DevSecOps Mindset

"Shift Left" means moving security checks earlier in the SDLC — from a quarterly pen-test to a sub-minute CI stage. The cost to fix a vulnerability multiplies at each stage: $1 in dev code → $10 in CI → $100 in staging → $10,000 in production (breach).

2. Secret Detection — Stop Secrets Before They Hit Git

⚠️ The Golden Rule

Once a secret is pushed to a remote Git repo, assume it is compromised — even if you immediately delete it. GitHub's secret scanning, third-party bots, and forks may have already seen it. Rotate immediately; don't just remove the commit.

bash — Gitleaks: install and scan
# Install Gitleaks
brew install gitleaks          # macOS
# or download binary from GitHub releases for Linux

# Scan entire repo history for secrets
gitleaks detect --source . --verbose

# Scan specific commit range (use in CI)
gitleaks detect --source . --log-opts "origin/main..HEAD"

# As a pre-commit hook (module-03 Git payoff)
# .pre-commit-config.yaml:
# repos:
#   - repo: https://github.com/gitleaks/gitleaks
#     hooks: [{ id: gitleaks }]
groovy — Jenkins: Gitleaks stage
// Add to the Jenkins pipeline from Module 09:
stage('Secret Scan') {
    steps {
        sh '''
            docker run --rm -v $(pwd):/path \
                zricethezav/gitleaks:latest detect \
                --source /path \
                --log-opts "origin/main..HEAD" \
                --exit-code 1
        '''
    }
}

3. SAST — Semgrep Static Analysis

SAST (Static Application Security Testing) analyzes source code without executing it, looking for patterns that indicate security bugs — SQL injection, hardcoded creds, insecure deserialization. Semgrep is the OSS leader: fast, rules-as-code (YAML), and language-agnostic.

bash — Semgrep scan
# Install
pip install semgrep --break-system-packages

# Run OWASP Top 10 rules against a Node.js app
semgrep --config p/owasp-top-ten .

# Python-specific security rules
semgrep --config p/python-security .

# Output results as SARIF (for GitHub Code Scanning upload)
semgrep --config p/owasp-top-ten --sarif > semgrep.sarif

# Custom rule example (catch subprocess with shell=True)
# rules:
#   - id: subprocess-shell-true
#     pattern: subprocess.run(..., shell=True, ...)
#     message: "shell=True with subprocess can allow command injection"
#     severity: ERROR

4. Container Scanning — Trivy

A Docker image is a filesystem containing OS packages, runtime, and your app. Every installed package is a potential CVE. Trivy scans the image layers and reports vulnerabilities with severity (CRITICAL/HIGH/MEDIUM/LOW) and the fix version.

bash — Trivy: scan image + fail on CRITICAL ⭐
# Install Trivy
brew install aquasecurity/trivy/trivy     # macOS
apt-get install -y trivy                  # Debian/Ubuntu

# Scan an image — full report
trivy image techitfactory/api:latest       # ⭐ daily use

# CI mode: fail build on CRITICAL CVEs only (avoid alert fatigue)
trivy image \
    --exit-code 1 \
    --severity CRITICAL \
    --no-progress \
    techitfactory/api:${IMAGE_TAG}

# Scan a Dockerfile before building
trivy config Dockerfile

# Scan Terraform/K8s manifests for misconfigurations
trivy config ./terraform
trivy config ./k8s-manifests/
💡 Reduce image attack surface

Use FROM node:20-alpine instead of FROM node:20. Alpine-based images have ~50% fewer CVEs (smaller OS footprint). Go and Rust can use FROM scratch — a completely empty base, zero OS CVEs. Multi-stage builds (Module 09) help further: the final image contains only the compiled binary, not the build toolchain.

5. SBOM — Software Bill of Materials

An SBOM is a machine-readable list of every component in a software artifact — OS packages, runtime libraries, direct and transitive dependencies. US Executive Order 14028 (2021) mandated SBOMs for software sold to the federal government. Even for private companies, SBOMs enable rapid CVE triage: when Log4Shell dropped, teams with SBOMs found affected services in minutes; others took days.

bash — generate and attach SBOM
# Generate SBOM with Trivy (SPDX format)
trivy image \
    --format spdx-json \
    --output sbom.spdx.json \
    techitfactory/api:${IMAGE_TAG}

# Or with Syft (CycloneDX format)
syft techitfactory/api:${IMAGE_TAG} -o cyclonedx-json > sbom.cyclonedx.json

# Attach SBOM to image as OCI attestation (cosign)
cosign attach sbom --sbom sbom.spdx.json techitfactory/api:${IMAGE_TAG}

# In CI: archive as artifact so auditors can access it

6. Policy as Code — OPA Gatekeeper

OPA (Open Policy Agent) Gatekeeper is a Kubernetes admission webhook: it intercepts every kubectl apply and evaluates your Rego policies. Non-compliant resources are rejected before they're persisted — cluster-level enforcement that no amount of CI discipline can replace.

bash — install Gatekeeper
kubectl apply -f \
    https://raw.githubusercontent.com/open-policy-agent/gatekeeper/release-3.15/deploy/gatekeeper.yaml
yaml — ConstraintTemplate: require resource limits
# ConstraintTemplate defines the Rego logic
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8srequiredlimits
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredLimits
  targets:
  - target: admission.k8s.gatekeeper.sh
    rego: |
      package k8srequiredlimits
      violation[{"msg": msg}] {
        container := input.review.object.spec.containers[_]
        not container.resources.limits.memory
        msg := sprintf("Container %v must set memory limit", [container.name])
      }
---
# Constraint applies the template to specific resources
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLimits
metadata:
  name: must-have-limits
spec:
  match:
    kinds: [{ apiGroups: [""], kinds: ["Pod"] }]
    namespaces: ["production", "staging"]
Common policiesWhat they enforce
Require resource limitsNo pod can skip memory/CPU limits (prevents OOMKilled cascades — M11 incident payoff)
No privileged containersBlocks securityContext.privileged: true
Require specific image registryOnly images from your ECR allowed (prevents supply-chain attacks)
Read-only root filesystemContainer filesystem immutable at runtime
No latest tagBlocks image: nginx:latest (non-reproducible)

7. Secrets Management in Kubernetes

Module 11 created K8s Secrets — but native K8s Secrets are base64-encoded, not encrypted, and cannot be safely committed to Git. Two solutions for GitOps (M17) + secrets:

ApproachHow it worksBest for
Sealed Secretsbitnami/sealed-secrets controller in the cluster holds a private key. kubeseal encrypts your secret → SealedSecret CRD safe to commit to Git. Controller decrypts at apply time.Simple self-contained clusters
External Secrets OperatorWatches ExternalSecret CRDs. Fetches real secrets from AWS Secrets Manager / Parameter Store at sync time. Secret never lives in Git at all.AWS-native, team already using Secrets Manager
bash — Sealed Secrets workflow
# Install Sealed Secrets controller
helm install sealed-secrets \
    sealed-secrets/sealed-secrets \
    -n kube-system

# Install kubeseal CLI
brew install kubeseal

# Encrypt a secret (outputs SealedSecret — safe to commit)
kubectl create secret generic db-pass \
    --from-literal=password=s3cr3t \
    --dry-run=client -o yaml | \
kubeseal --format yaml > sealed-db-pass.yaml

# Commit sealed-db-pass.yaml to the gitops config repo (Module 17)
# ArgoCD applies it; the controller decrypts it into a real K8s Secret
yaml — External Secrets Operator with AWS Secrets Manager
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
  namespace: production
spec:
  refreshInterval: 1h
  secretStoreRef:
    kind: ClusterSecretStore   # uses IRSA from Module 12
    name: aws-secretsmanager
  target:
    name: db-credentials       # K8s Secret created in-cluster
  data:
  - secretKey: password
    remoteRef:
      key: techitfactory/prod/db  # AWS Secrets Manager key name
      property: password

8. Runtime Security — Falco

Container scanning finds known CVEs in images at build time. Falco detects anomalous behavior at runtime — a container executing a shell, reading /etc/shadow, making unexpected network connections. It uses Linux eBPF/kprobes to hook kernel syscalls with near-zero overhead.

bash — install Falco with Helm
helm repo add falcosecurity https://falcosecurity.github.io/charts
helm install falco falcosecurity/falco \
    --namespace falco --create-namespace \
    --set falco.json_output=true \
    --set falco.log_stderr=true

# Watch Falco alerts in real time
kubectl logs -n falco -l app.kubernetes.io/name=falco -f
yaml — custom Falco rule: shell spawned in container
- rule: Shell Spawned in Container
  desc: Detect any shell executed in a running container
  condition: >
    container
    and proc.name in (bash, sh, zsh, dash)
    and not proc.pname in (runc, containerd-shim)
  output: >
    Shell spawned in container
    (user=%user.name container=%container.name
     image=%container.image.repository:%container.image.tag
     cmd=%proc.cmdline)
  priority: WARNING
  tags: [container, shell, mitre_execution]
⚠️ Falco is detection, not prevention

Falco alerts after the syscall happens — it's an IDS (Intrusion Detection System), not IPS. For prevention, combine it with: OPA Gatekeeper (block privileged containers before they start) + K8s NetworkPolicies (restrict lateral movement). Falco's alerts can trigger a Prometheus alert → AlertManager → PagerDuty chain (Module 16 integration).

9. Secure Pipeline Integration — Full Jenkinsfile

groovy — Jenkinsfile with full DevSecOps stages
// Module 09 pipeline + DevSecOps layers
pipeline {
    agent any
    environment {
        IMAGE = "123456789.dkr.ecr.us-east-1.amazonaws.com/techitfactory-api"
        TAG   = "${BUILD_NUMBER}-${GIT_COMMIT[0..6]}"
    }
    stages {
        stage('Secret Scan') {         // NEW: catch secrets before anything else
            steps {
                sh 'gitleaks detect --source . --log-opts "origin/main..HEAD" --exit-code 1'
            }
        }
        stage('SAST') {                 // NEW: static analysis
            steps {
                sh 'semgrep --config p/owasp-top-ten --error .'
            }
        }
        stage('Build') {
            steps {
                sh 'docker build -t $IMAGE:$TAG .'
            }
        }
        stage('Container Scan') {       // NEW: CVE scan before push
            steps {
                sh 'trivy image --exit-code 1 --severity CRITICAL --no-progress $IMAGE:$TAG'
            }
        }
        stage('Generate SBOM') {        // NEW: compliance artifact
            steps {
                sh 'trivy image --format spdx-json --output sbom-${TAG}.json $IMAGE:$TAG'
                archiveArtifacts 'sbom-*.json'
            }
        }
        stage('Push to ECR') {
            steps {
                sh 'docker push $IMAGE:$TAG'
            }
        }
        stage('Update Config Repo') {    // Module 17 GitOps handoff
            steps {
                sh '''
                    cd /tmp/gitops
                    yq eval ".image.tag = env(TAG)" -i environments/dev/values-dev.yaml
                    git commit -am "ci: bump to ${TAG} [skip ci]" && git push
                '''
            }
        }
    }
    post {
        always { cleanWs() }
        failure {
            slackSend(channel: '#alerts', message: "Security gate FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}")
        }
    }
}

10. Real-Time Incidents & Troubleshooting

Incident 1 — AWS Key Committed to GitHub

StageDetails
TimelineDeveloper commits hardcoded aws_secret_access_key in a config file. GitHub's secret scanning triggers within minutes. But third-party bots also scan GitHub continuously — compromise can happen in <5 minutes.
Immediate(1) Deactivate the key in IAM console — NOW, before anything else. (2) Check CloudTrail for usage of that key. (3) git filter-repo to rewrite history. (4) Force push with git push --force.
PreventionGitleaks pre-commit hook + CI secret scan. Never use long-lived access keys — use IAM roles (EC2/EKS IRSA — Modules 10/12). No access keys in developer machines for production; use AWS SSO.

Incident 2 — Trivy Blocks CI: CRITICAL CVE in Base Image

bash — handle base image CVEs
# Trivy fails: CRITICAL CVE in node:18-alpine base image
# Option 1: Update base image to latest patch
# FROM node:18-alpine → FROM node:20-alpine (or latest 18 patch)
docker pull node:20-alpine
trivy image node:20-alpine      # verify fewer CVEs

# Option 2: Suppress known-acceptable CVE with .trivyignore
# .trivyignore (commit to repo, document why each one is acceptable):
# CVE-2024-XXXX  # Not exploitable: we don't use this code path

# Option 3: Scope the severity threshold (avoid blocking on MEDIUM if noise is high)
trivy image --exit-code 1 --severity CRITICAL,HIGH $IMAGE:$TAG

Incident 3 — OPA Blocks a Legitimate Deploy

StageDetails
Symptomkubectl apply fails with: "Container api must set memory limit"
FixAdd resource limits to the Deployment manifest. This is actually correct behavior — Gatekeeper doing its job. If the policy is wrong, fix the policy (not by bypassing Gatekeeper).
Development workflowRun trivy config ./k8s-manifests/ locally before pushing (Section 4). This catches policy violations before hitting the admission webhook.

Incident 4 — Falco Alert: Shell in Production Container

StageDetails
AlertFalco: "Shell spawned in container — user=root container=api cmd=bash" at 2:17 AM
TriageWas this an authorized operator doing emergency debugging, or an attacker who exploited RCE? Check: who exec'd into the pod? Was there an on-call incident at that time? Check CloudTrail and kubectl audit logs.
PreventionDistroless images have no shell — attackers can't exec into them even if they gain code execution. Module 18 best practice: production images use FROM gcr.io/distroless/nodejs20. Debugging via ephemeral debug containers (kubectl debug), not exec.

Interview Questions

Click a card to reveal a strong candidate answer.

Q: What does "Shift Left" mean in DevSecOps?

Moving security checks earlier (leftward on the timeline) in the SDLC — from production pen-tests to pre-commit hooks and CI stages. The economics: a vulnerability costs ~$1 to fix in development, ~$10 in CI, ~$100 in staging, and can cost $10K+ in a production breach. Shift Left tools include: pre-commit hooks (Gitleaks), SAST in CI (Semgrep), container scanning before push (Trivy), policy admission control at deploy time (OPA Gatekeeper). The developer gets fast feedback on security issues without waiting for a security team review — security becomes self-service.

Q: How do you handle secrets in a GitOps setup?

Native K8s Secrets are base64 — not encrypted — and can't be safely committed to Git. Two main approaches: (1) Sealed Secrets: kubeseal encrypts with the cluster's public key → SealedSecret CRD is safe to commit → controller decrypts in-cluster. Simple but tied to one cluster. (2) External Secrets Operator: commits ExternalSecret manifests to Git (not the secret itself), ESO fetches live values from AWS Secrets Manager using IRSA. Rotation is automatic. For a GitOps + AWS setup, External Secrets is preferred because secrets never touch Git at all and rotation is just a Secrets Manager update — no re-sealing required.

Q: A developer pushed an AWS key to GitHub. What do you do?

Immediate actions (in order): (1) Deactivate the IAM key NOW in the console — before doing anything else. Bots scan GitHub continuously; assume compromise already happened. (2) Check CloudTrail for any API calls using that key after the commit timestamp. (3) Notify the security team and begin an incident. (4) Clean the Git history with git filter-repo and force-push. (5) Root cause and prevent: add Gitleaks as a pre-commit hook and CI stage, adopt IAM roles (IRSA, EC2 instance profiles) to eliminate long-lived access keys for developers. Key insight: deleting the commit doesn't help if the repo is public or was cloned — rotation is the only reliable fix.

Q: What is OPA Gatekeeper and how does it work?

OPA Gatekeeper is a Kubernetes admission webhook that enforces policies on every API server request. When you run kubectl apply, the API server sends the manifest to Gatekeeper before persisting it. Gatekeeper evaluates Rego policies (ConstraintTemplates) and returns ALLOW or DENY with a message. Examples: require all containers to have memory limits, block privileged containers, allow images only from your ECR. Unlike RBAC (which controls who can do what), Gatekeeper controls what can be created/updated. It's cluster-level enforcement that no amount of CI policy can replace — even if CI is bypassed, Gatekeeper still blocks non-compliant workloads from running.

Q: What's the difference between SAST and container scanning?

SAST (Static Application Security Testing) analyzes your source code for security bugs — patterns like SQL injection, hardcoded credentials, insecure deserialization, prototype pollution. It runs against your code before it's compiled or containerized. Container scanning (Trivy, Grype) analyzes the built Docker image — it inventories every OS package, runtime, and library in the image layers and checks them against CVE databases. SAST finds bugs you wrote; container scanning finds vulnerabilities in your dependencies. Both are needed: a perfectly written Node.js app can still have a CRITICAL CVE in node:18 that container scanning catches, while SAST catches a SQL injection that container scanning would miss entirely.

Q: How does Falco complement static security tools?

Static tools (Trivy, Semgrep, Gatekeeper) catch known issues at build time and deploy time. Falco addresses a different threat model: zero-day exploits, supply-chain attacks where a compromised npm package later phones home, or legitimate credentials stolen and used to exec into running containers. Falco monitors Linux kernel syscalls at runtime via eBPF — if a container suddenly spawns a shell, reads /etc/shadow, or opens a connection to an unexpected IP, Falco alerts in real time. The combination is: prevent known issues at build/deploy (static), detect anomalous behavior at runtime (Falco). Neither alone is sufficient.
