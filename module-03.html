1. What is Version Control & Why It Matters

Without a Version Control System (VCS), collaboration descends into chaos. We've all lived through the horror of folder structures like project_v1.zip, project_final.zip, and project_REALLY_FINAL_USE_THIS.zip. VCS solves this by tracking changes over time, allowing teams to collaborate concurrently, and providing an exact audit trail of who changed what, when, and why.

FeatureCentralized (e.g., SVN)Distributed (e.g., Git)
Repository LocationSingle central serverEvery developer has a full local copy
Offline WorkImpossible (needs server connection)Full support for commits, branches, logs offline
Single Point of FailureYes — if server dies, nobody can workNo — any dev's clone can restore the project
Why DevOps Runs on Git
  • Git won the VCS war: It is fast, branching is incredibly cheap, and it scales effortlessly.
  • Everything as Code: DevOps treats infrastructure (Terraform), configuration (Ansible), and pipelines (Jenkinsfile) exactly like application source code. All of it lives in Git.
  • GitOps (Preview): In modern workflows (Module 17), Git isn't just a storage location — it's the operational dashboard. If it's not in Git, it doesn't exist in production.

2. Git Architecture

Git doesn't just save files; it manages files moving across four distinct zones. Understanding this flow is the key to mastering Git and diagnosing issues.

The Four Zones of Git
Working Tree
(Your local files) →
git add Staging Area
(The index/prep zone) →
git commit Local Repo
(.git/ directory) →
git push Remote Repo
(GitHub/GitLab)
  • The .git/ Directory: When you run git init, Git creates a hidden .git/ folder. This is the repository. If you delete this folder, your files remain, but all version history is permanently destroyed.
  • Commits are Snapshots: Git doesn't store differences (deltas). It stores complete snapshots of your project at the exact moment of the commit, identified by a unique 40-character SHA-1 hash (e.g., f8d9a2c...).

Step 2 — Decoding git status

git status is your map. Learn to read its three main categories:

Understanding State
# 1. Untracked (Git doesn't know about this file yet)

        new_script.sh

# 2. Modified (Known to Git, changed, but NOT ready to commit)

Changes to be committed:
        modified:   readme.txt
        new file:   new_script.sh
Step 3 — Reading History

You spend more time reading code history than writing it.

History Commands
git log --oneline --graph
* a1b2c3d (HEAD -> main) Fix typo in config
* 9f8e7d6 Add database connection
* 1a2b3c4 Initial commit

git diff                  # What did I change right now? (Working vs Staged)
git diff --staged         # What am I about to commit? (Staged vs Last Commit)
git diff 1a2b3c4 9f8e7d6  # What changed between these two commits?

git show a1b2c3d          # Show me exactly what was changed in this commit
git blame file.txt        # Who wrote line 42? (Shows commit hash and author per line)
Step 4 — Remotes Explained

A "remote" is just a URL to a Git repo hosted elsewhere (like GitHub). By default, it's named origin.

Working with Remotes
git remote -v
origin  https://github.com/user/repo.git (fetch)
origin  https://github.com/user/repo.git (push)

# Pushing for the first time sets the "upstream tracking" (-u)
# This links your local 'main' to origin's 'main' so future pushes just need 'git push'
git push -u origin main

4. Branching & Merging

The Pointer Concept

A branch is not a physical copy of your files. It is simply a lightweight, movable pointer to a commit. In fact, a branch is literally just a 41-byte text file containing a commit hash. Because they are just pointers, branching in Git is instantaneous and takes almost no disk space. You should branch for every new feature or bugfix.

Branching Basics
git branch feature-login          # Create branch
git switch feature-login          # Switch to branch
git switch -c feature-login       # Create AND switch (shortcut)
git branch -d feature-login       # Delete branch
Fast-Forward vs 3-Way Merge

When you merge feature into main, Git does one of two things:

Merge Types
# 1. Fast-Forward Merge: If 'main' hasn't changed since you branched off.
# Git simply moves the 'main' pointer forward to point to 'feature'. No new commit needed.
Before: main(A) --- feature(B)
After:  main,feature(B)

# 2. 3-Way Merge: If 'main' HAS changed since you branched off.
# Git must combine the two divergent paths, creating a brand new "Merge Commit" (F).
main:      A --- B --- C ------- F (Merge Commit)
                        \       /
feature:                 D --- E
The Merge Conflict Deep-Dive

Merge conflicts happen when Git can't figure out how to combine changes—usually because two people modified the exact same lines. Resolving them is just text editing.

Resolving a Conflict
git merge feature-branch
Auto-merging app.py
CONFLICT (content): Merge conflict in app.py
Automatic merge failed; fix conflicts and then commit the result.

# You panic? You can always abort:
git merge --abort

# Or you open app.py and find the Conflict Markers:
<<<<<<< HEAD (Current Branch)
print("Hello from main branch")
=======
print("Hello from feature branch")
>>>>>>> feature-branch

# Edit the file to keep the correct line, delete the <<< === >>> markers, then:
git add app.py
git commit -m "Merge feature-branch, resolved conflict in app"

5. Git Workflows (Deep Dive)

How should your team organize branches? The industry relies on three main patterns. Choosing the wrong workflow can paralyze a team's delivery speed.

1. GitFlow (The Legacy Standard)

GitFlow uses strict, heavily separated branches. It separates day-to-day development (develop) from production (main), and uses specific branches for features, releases, and hotfixes. Best for apps with slow, scheduled release cycles (e.g. enterprise software released every 3 months).

GitFlow Example
main:    v1.0.0 ------------------------------------> v1.1.0
                 \                                     /
release:          \                                   / (Release QA)
                   \                                 /
develop:            --- A --- B --- C --- D --- E ---
                             /           \
feature:                    /             \
                   feature/login       feature/cart
2. GitHub Flow (The Modern Standard)

Simple and agile. The main branch is considered highly sacred and is always deployable to production. All new work happens in short-lived feature branches and is merged back into main via Pull Requests (PRs). Best for Web apps, SaaS, and continuous delivery.

GitHub Flow Example
# Every feature branch merges directly to main via PR
main:      A --- B --- C -------------- F (Deployed to Prod)
                        \              /
feature:                 D --- E (PR) -
3. Trunk-Based Development (Elite Speed)

Extreme continuous integration. Everyone commits directly to the main branch (the trunk) multiple times a day, or uses extremely short-lived branches (hours, not days). Instead of isolating features in branches, teams isolate features in code using Feature Flags (e.g., if (feature_enabled)). Best for high-maturity DevOps teams (Google, Netflix).

Trunk-Based Example
# No long branches. Everyone pushes to main constantly.
# Broken or unfinished code is hidden behind feature flags.
main (Trunk):  A --- B (Dev 1) --- C (Dev 2) --- D (Dev 1) --- E (Dev 3)
💡 The DevOps Verdict

GitHub Flow and Trunk-Based Development dominate modern DevOps. GitFlow is considered an anti-pattern for continuous delivery because long-lived develop and release branches delay integration and cause massive, painful merge conflicts. Merge small, merge often.

6. Pull Requests & Code Reviews

A Pull Request (PR) (or Merge Request in GitLab) is a request to merge your feature branch into the main codebase. It acts as the gatekeeper for code quality.

The PR Lifecycle
Branch
(checkout -b) → Commit & Push
(to remote) → Open PR
(on GitHub) → CI Checks Run
(tests pass) → Peer Review
(approved) → Merge
(to main)
  • Branch Protection Rules: GitHub allows you to enforce rules on main: e.g., "Require at least 1 approval," "Require CI tests to pass," or "Block force pushes."
  • CODEOWNERS: A file that automatically assigns PR reviews to specific people based on which files were changed (e.g., frontend team reviews UI files).
  • What Reviewers Look For: Logic errors, missing tests, poor performance, security flaws, and readability.

7. Rebase vs Merge

The ultimate interview question. Both integrate changes from one branch into another, but they do it differently.

Merge vs Rebase Visualized
# BEFORE (You are on feature, and main has moved forward)
main:      A --- B --- C
                        \
feature:                 D --- E

# 1. MERGE (Railroad Tracks) - Preserves exact history, adds Merge Commit (F)
main:      A --- B --- C ------- F
                        \       /
feature:                 D --- E

# 2. REBASE (Straight Line) - Rewrites history, replaying D and E onto C
main:      A --- B --- C
                        \
feature:                 D' --- E'
Merge

Takes the two endpoints and joins them with a new merge commit. Preserves absolute historical timeline, but creates messy "railroad track" graphs.

Rebase

Takes your feature branch commits and replays them, one by one, on top of the latest main. Creates a perfectly linear, clean history, but rewrites the commit hashes.

⚠️ The Golden Rule of Rebase

NEVER rebase a public/shared branch. If you rebase a branch that others have already pulled, you rewrite history. When they try to pull again, Git will fail spectacularly. Only rebase your local, private feature branches before opening a PR.

Squash Merges: GitHub allows you to "Squash and Merge" a PR. This takes all 20 messy WIP commits from your feature branch and squashes them into a single, clean commit on main. This is the preferred strategy for most teams.

8. Stash, Cherry-pick, Reset, Revert (The "Oops" Toolkit)

Stashing

Temporarily shelves your uncommitted changes so you can quickly switch branches.

Stash Workflow
git stash
Saved working directory and index state WIP on main: a1b2c3d Update README

# ... switch branches, do emergency work ...

git stash pop
On branch main
Changes not staged for commit:
  modified:   app.py
Dropped refs/stash@{0}
Modern Git: Switch and Restore

git checkout used to do too much. Modern Git split it into two safer commands:

Switch & Restore vs Checkout
# Switching branches
git switch feature-login      # Modern replacement for git checkout feature-login
git switch -c new-branch      # Modern replacement for git checkout -b new-branch

# Restoring files
git restore file.txt          # Discard uncommitted changes (replaces git checkout -- file.txt)
git restore --staged file.txt # Unstage a file (replaces git reset HEAD file.txt)
Reset vs Revert
CommandWhat it doesSafe for Shared Branches?
git revert Creates a new commit that does the exact opposite of an old commit. Yes. History only moves forward. Best for reverting production bugs.
git reset Moves the branch pointer backward in time, effectively erasing recent history. No. Rewrites history. Only use locally before you push.
Reset Modes
# --soft: Moves branch pointer, keeps changes STAGED
git reset --soft HEAD~1

# --mixed (default): Moves branch pointer, keeps changes UNSTAGED
git reset --mixed HEAD~1

# --hard: DANGER. Moves branch pointer, DESTROYS all changes
git reset --hard HEAD~1
The "Undo Anything" Decision Table
Oops...How to fix it
Staged the wrong filegit restore --staged <file>
Bad commit message (not pushed)git commit --amend
Committed to wrong branch (not pushed)git cherry-pick <hash> to new branch, then git reset --hard HEAD~1 on old branch
Deleted everything with a hard resetgit reflog to find the lost hash, then git reset --hard <hash>

⚠️ Leaking Secrets in Git

If you commit an AWS access key or a password to Git, it is forever recorded in the history. Even if you delete the file in the next commit, hackers can just look at the previous commit. If you leak a secret, consider it compromised immediately. You must revoke the key in AWS/Azure immediately. Do not try to clean the Git history (tools like BFG exist, but revocation is the only secure path).

Conventional Commits

Use a standard format for commit messages to enable automated changelogs and semantic versioning:

  • feat: add login page (New feature)
  • fix: resolve null pointer in payment gateway (Bug fix)
  • docs: update README deployment steps
  • ci: update GitHub Actions node version

10. Tags & Semantic Versioning

A tag is a permanent, immutable bookmark to a specific commit. While branches move forward, tags stay exactly where they were created. They are universally used to mark releases.

Semantic Versioning (SemVer)
v 2 . 4 . 1
  • MAJOR (2): Breaking changes, incompatible API updates.
  • MINOR (4): New features, but backwards-compatible.
  • PATCH (1): Bug fixes only.
  • Lightweight tag: Just a pointer (git tag v1.0.0)
  • Annotated tag: Stored as a full object with creator info, date, and a message (git tag -a v1.0.0 -m "Release v1.0.0"). Use these for official releases.
  • CI/CD Tie-in: In Jenkins or GitHub Actions (Modules 07/09), pushing a tag matching v* is usually the trigger that tells the pipeline to deploy to Production.

11. GitHub Platform Intro

Git is the tool; GitHub is the hosting platform and collaboration ecosystem. (GitLab and Bitbucket are the main competitors, providing nearly identical features).

  • SSH Key Setup: Reuse your knowledge from Module 02. Generate a key (ssh-keygen -t ed25519), add the public key to your GitHub Settings > SSH Keys, and you'll never need to type a password to push again.
  • Issues & Projects: GitHub provides Agile tracking. You can link a commit to an issue simply by typing Fixes #42 in your commit message (this auto-closes the issue when merged).
  • GitHub Actions (Teaser): Built-in CI/CD. You can run automated tests every time someone opens a Pull Request. We will cover this extensively in later CI/CD modules.

12. Advanced Real-Time DevOps Scenarios

As a DevOps engineer, you will face complex Git challenges beyond standard branching. Here are real-world scenarios and how to solve them.

Scenario 1: The "Split Brain" Monorepo Refactor

Context: Your company has a massive 50GB monorepo. The architecture team decided to extract the payments-microservice/ folder into its own standalone Git repository, but you must preserve all commit history for auditing purposes.

Solution: git subtree or filter-repo
# Using the modern git filter-repo tool (requires python)
git clone https://github.com/org/monorepo.git payments-service
cd payments-service
git filter-repo --path payments-microservice/ --to-subdirectory-filter .
# Now, this repo ONLY contains the payments microservice and its full commit history.
Scenario 2: The "Needle in a Haystack" Bug

Context: The CI pipeline is failing on the main branch. The last known good release was a week ago (commit a1b2c3d), and 50 commits have been pushed since then. You need to find exactly which commit broke the build.

Solution: git bisect
git bisect start
git bisect bad                 # The current commit is broken
git bisect good a1b2c3d        # The commit from last week worked
# Git will checkout a commit exactly in the middle. Run your test:
npm run test
# If it fails, type `git bisect bad`. If it passes, type `git bisect good`.
# Git will binary-search the 50 commits and find the exact culprit in ~6 steps.
git bisect reset               # Exit when done
Scenario 3: Submodules for Shared Infrastructure

Context: You have a highly optimized Terraform module for deploying AWS EKS. You want 10 different application teams to use this exact code in their own repositories without copy-pasting it.

Solution: git submodules
# In the application repository:
git submodule add https://github.com/org/terraform-eks-module.git infrastructure/eks
# This creates a pointer to a specific commit of the EKS module repo.
# When the app team clones their repo, they run:
git clone --recurse-submodules https://github.com/org/app-repo.git

13. DevOps Git Gauntlet (Practical Exams)

Test your mastery with these hands-on challenges. Try them locally in a dummy repository.

Exam 1

The Rebase Rescue

Setup: Create a branch feature-login. Make 5 separate commits, each adding one word to a file. The commits have terrible messages like "wip", "typo fix", "more stuff".

Challenge: Before merging to main, use Interactive Rebase (git rebase -i HEAD~5) to:
1. Squash all 5 commits into a single commit.
2. Rename the final commit message to feat: implement secure login module.

Success Criteria: git log on your branch should show only one clean commit instead of five messy ones.
Exam 2

The P0 Incident Interrupt

Setup: You are on feature-infra editing main.tf. Suddenly, production goes down. You must fix it immediately on the main branch, but your current Terraform work is halfway done and broken, so you can't commit it.

Challenge:
1. Use git stash to save your broken main.tf.
2. git checkout main and create a new branch hotfix-p0.
3. Edit a file to fix the bug, commit, and simulate merging to main.
4. Go back to feature-infra and use git stash pop to retrieve your broken Terraform code and continue working.

Success Criteria: You successfully deployed a hotfix without losing your in-progress feature work.
Exam 3

The Reflog Resurrection

Setup: Create a commit on main that adds a file called crucial_data.txt. Now, simulate a catastrophic mistake by running git reset --hard HEAD~1. The file is gone, the commit is gone. You never pushed this to GitHub.

Challenge:
1. Run git reflog to view the chronological history of your HEAD pointer.
2. Find the SHA hash of the commit right before you ran the hard reset.
3. Run git reset --hard <hash> to bring the lost commit (and crucial_data.txt) back from the dead.

Success Criteria: crucial_data.txt exists in your directory again. You have survived the worst Git disaster.

14. Daily Git Cheat Sheet & Pro Tips

The 15 commands you will type every single day as a DevOps Engineer.

The Daily 15
# === THE BASICS ===
git status                  # ⭐ The map. Where am I and what changed?
git pull                    # ⭐ Download and merge latest changes from remote
git add .                   # ⭐ Stage all changes in current directory
git commit -m "msg"         # ⭐ Save staged changes with a message
git push                    # ⭐ Upload commits to remote

# === BRANCHING ===
git switch -c feature-x     # ⭐ Create a new branch and switch to it
git switch main             # ⭐ Switch back to main
git branch                  # ⭐ List all local branches
git branch -d feature-x     # ⭐ Delete a local branch (safely)

# === HISTORY & RECOVERY ===
git log --oneline --graph   # ⭐ View history compactly
git diff                    # ⭐ Show unstaged changes
git stash                   # ⭐ Temporarily shelve changes
git stash pop               # ⭐ Bring shelved changes back
git restore file.txt        # ⭐ Discard local unstaged changes to a file
git reset --soft HEAD~1     # ⭐ Undo the last commit (keep files staged)
Pro Tip 1: Git Aliases

Stop typing full commands. Set up global aliases in your ~/.gitconfig.

Setup Aliases
git config --global alias.co checkout
git config --global alias.sw switch
git config --global alias.st status
git config --global alias.cm commit
git config --global alias.lg "log --oneline --graph"

# Now you can just type:
git st
git sw -c new-branch
Pro Tip 2: git pull --rebase

A standard git pull executes a fetch followed by a merge. If you and a coworker both made commits to the same branch, this creates an ugly "Merge branch 'main' into main" commit. Instead, use --rebase to pull their changes and cleanly apply yours on top.

Pull with Rebase
# Keeps history linear and clean:
git pull --rebase

# Make it the default behavior for all pulls:
git config --global pull.rebase true

🛠️ Real-Time Project: Team Collaboration Git Workflow

Create a multi-branch project on GitHub. Simulate a team workflow: create feature branches, raise pull requests, resolve merge conflicts, use git rebase, tag releases with semantic versioning, and set up branch protection rules. Portfolio-ready GitHub repo as the artifact.

Repo Setup

Create a new repository on GitHub. Clone it locally. Add a README.md, an app.py script (just a print statement), and a DevOps .gitignore. Commit and push to main.

Terminal
git init my-project
cd my-project
echo "# My Project" > README.md
echo 'print("v1.0")' > app.py
echo "node_modules/" > .gitignore
git add .
git commit -m "Initial commit"
# Link to GitHub and push
git remote add origin https://github.com/your-username/my-project.git
git push -u origin main

Branch Protection (GitHub UI)

We must lock down main so nobody can push directly to production.

  • Go to your repository on GitHub.
  • Click Settings > Branches.
  • Click Add branch ruleset. Name it "Protect Main".
  • Set Target branches to include the default branch (main).
  • Check Require a pull request before merging.
  • Check Do not allow bypassing the above settings.
  • Save changes. You are now locked out of pushing directly to main.

Feature Branches (The Simulation)

Create feature-A and feature-B locally. Make conflicting changes to the same line in app.py.

Terminal
# Simulate Developer 1 (Feature A)
git switch -c feature-A
echo 'print("Feature A active")' > app.py
git add app.py
git commit -m "Add feature A"
git push -u origin feature-A

# Simulate Developer 2 (Feature B) starting from main
git switch main
git switch -c feature-B
echo 'print("Feature B active")' > app.py
git add app.py
git commit -m "Add feature B"

Pull Request & Self-Review

Push feature-A to GitHub and merge it via PR.

  • Go to GitHub, you will see a prompt to "Compare & pull request" for feature-A.
  • Click it. Since it's a solo project, you may need to temporarily disable the "Require approvals" rule to merge it yourself.
  • Click Merge pull request. Delete the branch on GitHub.

Engineered Merge Conflict

Switch to feature-B. Pull the updated main to trigger a conflict.

Terminal
git switch feature-B
git pull origin main
CONFLICT (content): Merge conflict in app.py
Automatic merge failed; fix conflicts and then commit the result.

# Open app.py, resolve the markers to keep BOTH features:
print("Feature A active")
print("Feature B active")

git add app.py
git commit -m "Resolve merge conflict with feature A"
git push -u origin feature-B

Now open a PR for feature-B and merge it on GitHub.

Rebase Practice

Create feature-C off main. Add 3 messy WIP commits and squash them into one clean commit before pushing.

Terminal
git switch main
git pull origin main
git switch -c feature-C

echo 'print("WIP 1")' >> app.py && git commit -am "WIP"
echo 'print("WIP 2")' >> app.py && git commit -am "more WIP"
echo 'print("WIP 3")' >> app.py && git commit -am "done"

# Interactive rebase the last 3 commits
git rebase -i HEAD~3
# Change 'pick' to 'squash' (or 's') for the bottom two commits.
# Save and close the editor. Then write a clean commit message.

git push -u origin feature-C

Tag Release v1.0.0

Get all merged features and create a release tag.

Terminal
git switch main
git pull origin main

# Create annotated tag
git tag -a v1.0.0 -m "Initial Production Release"

# Push tag to GitHub
git push origin v1.0.0

Go to your GitHub repository, click Tags, and verify v1.0.0 is listed as a Release.

Interview Questions

Q1

Merge vs Rebase. Which do you prefer and why?

Merge preserves the exact historical timeline and creates a merge commit, which is safe for shared branches but can create a messy graph. Rebase replays your commits on top of the target branch, creating a perfectly linear history, but rewrites commit hashes. I prefer rebasing local, unshared feature branches against main to keep my PR clean, but rely on Squash & Merge on GitHub to actually integrate into main. I never rebase shared/public branches.
Q2

What is the difference between git fetch and git pull?

git fetch only downloads new data from the remote repository to your local .git/ directory; it does not change any of your files in the working directory. git pull executes a fetch and then immediately tries to merge those incoming changes into your current active branch.
Q3

A bad commit was pushed to production (main). How do you undo it? Reset or Revert?

Always use git revert <commit_hash>. This creates a new commit that safely negates the changes of the bad commit, moving the history forward. If you used git reset, you would rewrite history on a shared branch, breaking the repository for all other developers on the team when they next try to push or pull.
Q4

How exactly do you resolve a merge conflict?

I run git status to find files marked "both modified". I open each file in my editor and look for the conflict markers (<<<<<<< HEAD, =======, >>>>>>> branch-name). I review the two competing code blocks, manually edit the code to the correct final state, and delete the marker lines. Then, I git add the resolved files and run git commit to complete the merge.
Q5

Explain GitFlow vs Trunk-Based Development.

GitFlow is a heavy branching model with separate branches for master, develop, features, releases, and hotfixes—useful for software with strict, infrequent release cycles. Trunk-Based Development has developers committing to a single trunk (main) multiple times a day (or using very short-lived feature branches). Trunk-based relies heavily on automated testing and feature flags and is the standard for modern CI/CD and DevOps.
Q6

You accidentally committed an AWS access key and pushed it to GitHub. What happens, and what do you do?

What happens: Bots scrape public GitHub repos 24/7; my key is compromised within seconds, potentially costing the company thousands in spun-up crypto miners. What to do: Do NOT just make a new commit deleting the key—the key is still in the history! Step 1: Immediately revoke/delete the compromised key in the AWS Console. Step 2: Generate a new key and handle it properly using secrets management or environment variables.
Q7

What does it mean to be in a "detached HEAD" state?

Normally, HEAD points to a branch name (like main), which in turn points to the latest commit. If you check out a specific commit hash directly (e.g., git checkout abc1234), HEAD points directly to that commit, not a branch. You are now "detached". Any new commits made here will be orphaned and lost when you switch away, unless you create a new branch from this state.
