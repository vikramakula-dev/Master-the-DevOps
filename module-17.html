Module 17 — GitOps with ArgoCD

1. GitOps Principles

PrincipleWhat it meansTraditional CD contrast
DeclarativeDesired cluster state is in Git as YAML/HelmImperative kubectl apply in CI scripts
VersionedEvery state change = a Git commit with author, PR, reviewDeployment is a CI job with no persistent audit trail
PulledCluster agent pulls from Git — CI has no cluster credentialsCI pushes (holds kubeconfig, broad permissions)
ReconciledArgoCD continuously diffs Git vs cluster, auto-heals driftDrift undetected until next deploy or manual inspection
Push CD vs GitOps Pull
Push CD:
CI builds → CI deploys
(CI holds kubeconfig) vs GitOps:
CI builds → commits tag to config repo
→ ArgoCD detects → pulls → deploys
💡 Security argument for pull

In push CD, every CI runner holds a kubeconfig (cluster-admin in most naive setups). Compromise one CI machine and you own the cluster. In GitOps, CI only writes to a Git repo. ArgoCD, running inside the cluster, is the only thing that applies manifests — and its credentials never leave the cluster. The attack surface is dramatically smaller.

3. Install ArgoCD

bash — install + first login
# Install ArgoCD into its own namespace
kubectl create namespace argocd
kubectl apply -n argocd \
    -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# Install the argocd CLI
curl -sSL -o argocd-linux-amd64 \
    https://github.com/argoproj/argo-cd/releases/latest/download/argocd-linux-amd64
sudo install -m 555 argocd-linux-amd64 /usr/local/bin/argocd

# Wait for ArgoCD to be ready
kubectl wait --for=condition=available deployment/argocd-server \
    -n argocd --timeout=120s

# Port-forward the UI
kubectl port-forward svc/argocd-server -n argocd 8080:443

# Get the initial admin password
argocd admin initial-password -n argocd
# Login
argocd login localhost:8080 --username admin --insecure

# Connect the Git repo (use SSH key or HTTPS token)
argocd repo add git@github.com:techitfactory/gitops.git \
    --ssh-private-key-path ~/.ssh/argocd-deploy-key

4. ArgoCD Applications & Sync Policy

yaml — annotated Application manifest
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: techitfactory-dev
  namespace: argocd              # ArgoCD always lives in argocd namespace
spec:
  project: default               # ArgoCD project (RBAC boundary — Section 8)
  source:
    repoURL: git@github.com:techitfactory/gitops.git
    targetRevision: HEAD          # or a branch name, tag, or commit SHA
    path: environments/dev        # directory inside the repo
    helm:
      valueFiles: [values-dev.yaml]
  destination:
    server: https://kubernetes.default.svc  # the cluster ArgoCD runs in
    namespace: development
  syncPolicy:
    automated:                    # auto-sync: ArgoCD applies whenever Git changes
      selfHeal: true              # revert manual kubectl changes (drift fix)
      prune: true                 # delete resources removed from Git
    syncOptions:
    - CreateNamespace=true
Sync modeBehaviorUse case
ManualArgoCD detects diff but waits for human to click SyncProduction (governance gate)
Auto-syncApplies changes immediately when Git changesDev/staging
Auto-sync + selfHealAlso reverts any manual cluster changesStrict staging where drift = bug

5. ArgoCD + Helm + Config Repo — The CI Handoff

The capstone uses a three-repo model (app code, Helm chart, config repo). CI builds the image and commits the new tag to the config repo. ArgoCD detects the commit and syncs. Jenkins never touches the cluster.

The CI/CD Handoff — Module 09 Pipeline's Final Form
Dev pushes code
to app repo → Jenkins:
Build + Test + Push ECR → Jenkins commits
new image.tag to
config repo → ArgoCD detects
config change → Cluster pulls +
applies new tag
bash — Jenkins stage: commit image tag to config repo (yq)
# In the Jenkins pipeline (after docker push to ECR):
IMAGE_TAG="${BUILD_NUMBER}-${GIT_COMMIT[0..6]}"

# Clone the config repo
git clone git@github.com:techitfactory/gitops.git /tmp/gitops
cd /tmp/gitops

# Update the image tag using yq (Module 02/14 YAML-patching payoff)
yq eval ".image.tag = \"${IMAGE_TAG}\"" -i environments/dev/values-dev.yaml

# Commit and push
git add environments/dev/values-dev.yaml
git commit -m "ci: bump api image to ${IMAGE_TAG} [skip ci]"
git push origin main

# ArgoCD detects the commit within ~3 minutes (default poll interval)
# Or configure a webhook for instant detection

6. Multi-Environment — Dev Auto, Prod Manual

text — config repo structure
techitfactory-gitops/
├── apps/                        # ArgoCD Application manifests
│   ├── techitfactory-dev.yaml   # auto-sync, selfHeal
│   └── techitfactory-prod.yaml  # manual sync (governance)
├── base/                        # shared Helm chart
│   └── chart/
└── environments/
    ├── dev/
    │   └── values-dev.yaml      # image.tag updated by CI
    └── prod/
        └── values-prod.yaml     # updated via PR from dev (promotion)

Promotion to production is a PR: update values-prod.yaml with the tag that passed in dev, get a code review, merge. ArgoCD detects the merge, shows the diff in the UI, and a human clicks Sync. Deployment review = code review.

7. App of Apps — Bootstrap an Entire Cluster

Instead of applying each ArgoCD Application manually, create a root Application that manages the other Applications. Apply one manifest → ArgoCD bootstraps everything. This is the capstone disaster recovery story: new cluster + one apply = everything back.

yaml — root App of Apps
# root-app.yaml — apply this once to bootstrap the whole cluster
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: root
  namespace: argocd
spec:
  project: default
  source:
    repoURL: git@github.com:techitfactory/gitops.git
    targetRevision: HEAD
    path: apps/                     # directory containing child Application manifests
  destination:
    server: https://kubernetes.default.svc
    namespace: argocd
  syncPolicy:
    automated: { selfHeal: true, prune: true }
bash — disaster recovery: new cluster, one command
# 1. New EKS cluster (terraform apply — Module 13)
# 2. Install ArgoCD
kubectl apply -n argocd -f https://...argocd/stable/manifests/install.yaml

# 3. Connect repo and apply the root app
kubectl apply -f root-app.yaml

# Result: ArgoCD discovers all child apps, syncs them all
# All namespaces, monitoring, the app — everything converges from Git
# This is Modules 13 + 17 combined: full stack from code

8. RBAC & SSO

ArgoCD Projects are RBAC tenancy boundaries — a project restricts which repos and clusters an Application can use. Roles within projects control who can sync.

yaml — ArgoCD Project with RBAC
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: techitfactory
  namespace: argocd
spec:
  description: techitfactory applications
  sourceRepos:
  - git@github.com:techitfactory/gitops.git    # only this repo allowed
  destinations:
  - namespace: production
    server: https://kubernetes.default.svc
  roles:
  - name: read-only
    description: Read-only for developers
    policies:
    - p, proj:techitfactory:read-only, applications, get, techitfactory/*, allow
  - name: deployer
    description: Can sync apps
    policies:
    - p, proj:techitfactory:deployer, applications, sync, techitfactory/*, allow

9. Real-Time Incidents & Troubleshooting

Incident 1 — App Stuck "OutOfSync" (Drift)

StageDetails
SymptomArgoCD shows application as OutOfSync even though no recent Git commit. A team member ran kubectl edit deployment api to hotfix a bug.
Root CauseManual kubectl change created drift. With selfHeal: true, ArgoCD reverts it within 3 minutes automatically — demonstrating the power and the danger of GitOps: the hotfix disappears unless it's committed to Git first.
FixCommit the intended change to Git (open a PR). ArgoCD syncs it properly. The lesson: there are no more "quick kubectl fixes" in GitOps — all changes go through Git.
PreventionK8s RBAC in production: deny direct kubectl patch/edit to most developers. Only ArgoCD's service account has cluster-write. Engineers must go through Git.

Incident 2 — Sync Fails with Immutable Field Error

bash — fix immutable field conflict
# Error: "field is immutable" — e.g., changed a Job's selector or Service's clusterIP
# Strategy 1: Replace=true annotation (ArgoCD deletes and recreates)
# Add to the Application's syncPolicy:
# syncOptions: [Replace=true]

# Strategy 2: Manual delete and let ArgoCD recreate
kubectl delete job my-migration-job -n production
# ArgoCD next sync creates it fresh

# Note: Replace=true causes downtime — use only for immutable-field updates

Incident 3 — Secret in Config Repo

StageDetails
SymptomA developer committed a real database password to values-prod.yaml in the config repo.
Immediate actions(1) Rotate the credential immediately. (2) Remove from Git history (git filter-repo). (3) Force-push (coordinate with team). (4) Assume compromised.
Long-term fixUse Sealed Secrets or External Secrets Operator. Sealed Secrets encrypt the secret with a cluster-specific key — the SealedSecret CRD is safe to commit. ESO fetches secrets from AWS Secrets Manager at sync time (Module 18 bridge).

Incident 4 — ArgoCD Shows Healthy, App is Broken

StageDetails
Root CauseArgoCD's Healthy status = Deployment has desired replicas running. It doesn't know if the app is actually serving requests correctly. A misconfigured env var causes 500 errors on every request — pods are Running, ArgoCD is happy, users are not.
FixCombine ArgoCD with the Module 16 Prometheus alerts. ArgoCD handles deployment lifecycle; Prometheus/AlertManager handles runtime correctness. Both are needed.

Interview Questions

Click a card to reveal a strong candidate answer.

Q: What is GitOps and how does it differ from traditional CD?

GitOps is a CD approach where Git is the single source of truth for the desired cluster state. A reconciliation agent (ArgoCD) continuously compares Git to the cluster and applies any diff. Traditional CD (push): a CI pipeline holds cluster credentials and applies manifests during a build job — the cluster state after the pipeline is unknown until the next plan. GitOps advantages: all changes have a Git audit trail, CI never holds cluster credentials (pull reduces attack surface), drift is detected and auto-healed, rollback is a git revert. The tradeoff: more tooling, and engineers must change the habit of kubectl apply.

Q: What's the security argument for the pull model?

In push CD, every CI runner needs cluster credentials (typically kubeconfig with broad permissions). Compromise one Jenkins agent and you can deploy arbitrary code or delete everything. In GitOps pull, CI runners only need write access to a Git repository — a far narrower blast radius. ArgoCD, running inside the cluster, is the only system that applies manifests, and its service account credentials never leave the cluster. Even a full compromise of the CI system can't directly deploy to the cluster — an attacker would need to also control the Git repo (which has branch protection and PR reviews).

Q: What happens when someone does a kubectl edit in a GitOps setup?

ArgoCD detects the drift immediately and marks the application OutOfSync. If selfHeal is enabled (typical for staging), ArgoCD reverts the change within its sync interval (default 3 minutes). In production with manual sync, the app stays OutOfSync until a human reviews and decides whether to sync (reverting the kubectl change) or commits the change to Git (codifying it). This is the key cultural shift in GitOps: the cluster state always mirrors Git, so any change not in Git is temporary at best.

Q: Why split app repo from config repo?

Separation of concerns. The app repo contains source code — it changes frequently and triggers builds. The config repo contains deployment state (image tags, replica counts, env-specific values) — it changes when you want to deploy a specific version. Separation means: CI commits image tags to the config repo without triggering a new CI build (no infinite loop). Different teams can have different permissions — developers commit to the app repo; platform team controls config repo merge rules. Multiple microservices can share one config repo (a monorepo for infra state). ArgoCD watches the config repo, not the app repo.

Q: How do rollbacks work in GitOps?

A rollback is a git operation — git revert HEAD or git checkout HEAD~1 -- values-prod.yaml && git commit. This produces a new commit in the config repo that sets the image.tag back to the previous value. ArgoCD detects the commit, syncs, and the cluster deploys the previous image. The entire operation is audited in Git: who decided to roll back, when, and why (the commit message). No special rollback tool needed. This is cleaner than helm rollback because the rollback state is recorded in Git history.

Q: What is App of Apps and when would you use it?

App of Apps is a pattern where one ArgoCD Application manages a directory of other Application manifests. Instead of manually creating each app via CLI or UI, you apply one root app and ArgoCD recursively manages all the children. Use cases: bootstrapping a new cluster (apply one manifest, everything converges), managing many microservices across environments (the root app watches apps/), disaster recovery (new cluster + one apply = full restore from Git). The tradeoff: delete the root app and ArgoCD tries to prune all children — be careful with prune settings on the root app.
