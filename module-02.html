1. Introduction to Operating Systems

An Operating System (OS) is the software layer between hardware and applications. It manages the CPU, memory, storage, and network, and provides the interfaces (shell, system calls) that programs use. The three components you must understand:

  • Kernel — the core that talks to hardware, schedules processes, and manages memory
  • Shell — the command interpreter (Bash, Zsh, PowerShell) between you and the kernel
  • File System — how data is organized and stored on disk
OSKernelTypical UseDevOps Relevance
WindowsWindows NTDesktops, enterprise apps, .NETSome enterprise servers; WSL2 for local Linux
macOSXNU (Unix-based)Developer workstationsUnix-like terminal, great DevOps dev machine
LinuxLinuxServers, cloud, containers, embeddedThe platform of DevOps — servers, containers, K8s nodes
Why Linux Dominates DevOps
  • Open source and free — no licensing at scale
  • Runs ~96% of the top million web servers and nearly all cloud instances
  • Docker containers share the Linux kernel — containers ARE Linux processes
  • Every Kubernetes node runs Linux; every CI runner defaults to Linux
  • Scriptable end-to-end — everything is a file, everything can be automated
💡 Popular Linux Distributions

Ubuntu/Debian (apt) — most common on cloud & this roadmap's choice. RHEL/CentOS/Amazon Linux/Fedora (yum/dnf) — enterprise standard. Alpine — tiny (~5MB), the default base image for Docker containers.

4. Linux File System & Directory Structure

Linux organizes everything under a single root / — there are no drive letters. The layout follows the Filesystem Hierarchy Standard (FHS), so every Linux machine feels familiar:

Filesystem Hierarchy
/
├── bin/     → essential command binaries (ls, cp, bash)
├── boot/    → kernel & bootloader files
├── dev/     → device files (disks: /dev/sda, terminals)
├── etc/     → system-wide CONFIGURATION files
├── home/    → user home directories (/home/vikram)
├── lib/     → shared libraries
├── opt/     → optional third-party software
├── proc/    → virtual files exposing kernel/process info
├── root/    → home directory of the root user
├── tmp/     → temporary files (cleared on reboot)
├── usr/     → user programs (/usr/bin, /usr/local)
└── var/     → variable data — LOGS (/var/log), web roots, spools
💡 The Three You'll Live In

/etc — every config you'll ever edit (nginx, ssh, cron). /var/log — the first place you look when something breaks. /home — your files and dotfiles. Also remember: in Linux everything is a file — devices, sockets, even running processes (/proc).

Absolute vs Relative Paths
bash
cd /var/log/nginx     # absolute — starts from root /
cd ../backups          # relative — .. means parent directory
cd ~                   # ~ = your home directory
cd -                   # jump back to previous directory
pwd                    # print where you are

The Search & Text Power Trio — find, grep, awk/sed

cat access.log | grep " 500 " | awk '{print $1}' | sort | uniq -c | sort -rn | head
# → top IPs causing 500 errors. This one-liner is a real interview question.

6. File Permissions & Ownership — chmod, chown, umask

Every file has an owner, a group, and permission bits for three audiences: user (owner), group, and others. Read ls -l output like this:

Decoding ls -l
-rwxr-x r-- 1 vikram devops 4096 Jul 13 10:30 deploy.sh
│└┬┘└┬┘└┬┘   └──┬──┘ └──┬──┘
│ │  │  │     owner    group
│ │  │  └── others: r-- (read only)
│ │  └───── group:  r-x (read + execute)
│ └──────── user:   rwx (read + write + execute)
└────────── type: - file, d directory, l symlink
Numeric (Octal) Notation — r=4, w=2, x=1
NumberMeaningCommon Use
755rwx r-x r-xScripts & directories — owner full, everyone can run/enter
644rw- r-- r--Regular files — owner edits, everyone reads
600rw- --- ---Secrets — SSH private keys MUST be 600
777rwx rwx rwx⚠️ Everyone everything — almost always wrong
bash
chmod 755 deploy.sh            # numeric mode
chmod +x deploy.sh             # symbolic — add execute for all
chmod u+x,g-w app.sh           # fine-grained symbolic
chown vikram:devops app.log    # change owner AND group
chown -R www-data:www-data /var/www   # recursive — web server files
umask                          # default mask: 022 → new files 644, dirs 755
⚠️ sudo & root

root (UID 0) can do anything. Never work as root directly — use sudo command for individual privileged actions. Users allowed to sudo are configured in /etc/sudoers (edit only with visudo) or by membership of the sudo group.

7. User & Group Management

Linux is multi-user by design. Users are defined in /etc/passwd, password hashes in /etc/shadow, groups in /etc/group. Services get their own non-login users (e.g. www-data for nginx) — a core security practice.

bash
# Create a user with home directory and bash shell
sudo useradd -m -s /bin/bash devuser
sudo passwd devuser                 # set password

# Groups
sudo groupadd deployers
sudo usermod -aG deployers devuser  # -aG = APPEND to group (never forget -a!)
sudo usermod -aG sudo devuser       # grant sudo rights (Ubuntu)
groups devuser                      # check memberships

# Inspect & switch
id devuser                          # uid, gid, groups
whoami                              # current user
su - devuser                        # switch user (login shell)

# Remove
sudo userdel -r devuser             # -r also removes home dir
🚨 Classic Mistake

usermod -G group user (without -a) replaces all existing group memberships — you can lock yourself out of sudo. Always usermod -aG.

9. Process Management — ps, top, kill, systemctl

Every running program is a process with a PID. Long-running background processes (web servers, databases) are services/daemons, managed on modern Linux by systemd.

bash — inspecting processes
ps aux                          # all processes, full detail
ps aux | grep nginx             # find a specific process
top                             # live view — CPU/mem per process (q quits)
htop                            # nicer top (sudo apt install htop)

# Signals
kill 1234                       # SIGTERM — polite "please stop"
kill -9 1234                    # SIGKILL — force kill, last resort
pkill -f myapp                  # kill by name/pattern

# Background jobs
./long-task.sh &                # run in background
nohup ./server.sh &             # keep running after logout
jobs / fg / bg                  # manage shell jobs
systemctl — Managing Services (systemd)
bash — the service lifecycle you'll use daily
sudo systemctl start nginx      # start now
sudo systemctl stop nginx
sudo systemctl restart nginx    # after config changes
sudo systemctl reload nginx     # re-read config without dropping connections
sudo systemctl enable nginx     # start automatically on boot ⭐
systemctl status nginx          # running? failed? recent log lines
journalctl -u nginx -f          # follow a service's logs live
💡 Troubleshooting Pattern

Service down? The sequence is always: systemctl status app → journalctl -u app -n 50 → check config in /etc/ → systemctl restart app. Memorize this flow — it's half of on-call life.

10. Networking Basics — IP, DNS, ports, curl, netstat

Servers talk over networks; DevOps engineers debug those conversations. The essentials: every machine has an IP address, services listen on ports (nginx: 80/443, SSH: 22, Postgres: 5432), and DNS translates names to IPs.

PortServicePortService
22SSH443HTTPS
80HTTP3306MySQL
53DNS5432PostgreSQL
8080HTTP alt (Jenkins!)6379Redis
bash — the network debugging toolkit
# Who am I / can I reach the world?
ip a                            # my interfaces & IPs (replaces ifconfig)
ping -c 4 google.com            # basic reachability
nslookup example.com            # DNS resolution (also: dig example.com)

# HTTP requests from the terminal
curl -I https://example.com     # headers only — status code check
curl -s api.example.com/health | jq   # API call, pretty JSON
wget https://example.com/file.tar.gz  # download files

# What is listening on this server?
sudo ss -tulpn                  # modern netstat — TCP/UDP listeners + PIDs ⭐
sudo netstat -tulpn             # classic version, same idea
sudo lsof -i :8080              # which process holds port 8080?

# Firewall (Ubuntu ufw)
sudo ufw allow 22/tcp           # ALWAYS allow SSH before enabling!
sudo ufw allow 80,443/tcp
sudo ufw enable && sudo ufw status
⚠️ "Port already in use"

The most common deployment error. Fix: sudo lsof -i :PORT to find the process, then stop it properly (systemctl stop) or choose another port. Never blind-kill -9 a database.

11. Shell Scripting with Bash

Bash scripts turn command sequences into reusable automation — the direct ancestor of every CI/CD pipeline you'll ever write. A script is just a text file of commands with a shebang line and execute permission. We'll build up from zero.

Step 1 — Your Very First Script
bash — hello.sh
#!/bin/bash
# ↑ The "shebang" — MUST be line 1. Tells Linux which interpreter runs this file.

echo "Hello, DevOps World!"     # echo prints text to the screen
echo "Today is: $(date)"        # $(...) runs a command and inserts its output
bash — make it executable and run it
chmod +x hello.sh    # give the file execute permission (one time only)
./hello.sh           # ./ means "run from the current directory"

# Output:
# Hello, DevOps World!
# Today is: Sun Jul 13 10:30:00 IST 2026
Step 2 — Variables (Storing Values)

A variable stores a value you want to reuse. Rule 1: no spaces around =. Rule 2: read a variable with $name. Rule 3: always wrap in double quotes "$name" so values with spaces don't break your script.

bash — variables.sh
#!/bin/bash

# Creating variables (NO spaces around =)
NAME="Vikram"                    # text value
APP_PORT=8080                    # number value
TODAY=$(date +%F)                # command output → variable (e.g. 2026-07-13)
SERVER_IP=$(hostname -I | awk '{print $1}')  # pipes work inside $() too

# Using variables — $ in front, quotes around
echo "User      : $NAME"
echo "Port      : $APP_PORT"
echo "Date      : $TODAY"
echo "Server IP : $SERVER_IP"

# Curly braces {} when text touches the variable name
echo "Log file  : ${NAME}_backup_${TODAY}.log"   # Vikram_backup_2026-07-13.log

# Read input from the user
read -p "Which environment (dev/prod)? " ENV
echo "You chose: $ENV"
Step 3 — Script Arguments ($1, $2 …)

Arguments are values you pass when running the script: ./deploy.sh prod v2.1. Inside the script, $1 is prod, $2 is v2.1.

bash — args.sh
#!/bin/bash
# Run as:  ./args.sh prod v2.1

echo "Script name    : $0"      # ./args.sh
echo "First argument : $1"      # prod
echo "Second argument: $2"      # v2.1
echo "Argument count : $#"      # 2
echo "All arguments  : $@"      # prod v2.1

# Default value if the argument was not given
ENVIRONMENT=${1:-dev}            # use $1, or "dev" if $1 is empty
echo "Deploying to   : $ENVIRONMENT"

# Fail fast if a required argument is missing
if [ $# -lt 1 ]; then
    echo "Usage: $0 <environment> [version]"
    exit 1                       # non-zero exit = failure
fi
Step 4 — If / Else & Test Operators

Conditions live inside [ ] (spaces required inside the brackets!). These are the operators you'll use constantly:

OperatorTrue When…OperatorTrue When…
-f fileFile exists-eqNumbers equal
-d dirDirectory exists-neNumbers not equal
-z "$var"String is empty-gt / -ltGreater / less than
-n "$var"String is NOT empty= / !=Strings equal / not equal
-x fileFile is executable&& / ||AND / OR
bash — conditions.sh
#!/bin/bash

CONFIG="/etc/nginx/nginx.conf"

# if / elif / else — mind the spaces inside [ ]
if [ -f "$CONFIG" ]; then
    echo "✔ Config file exists"
elif [ -d "/etc/nginx" ]; then
    echo "Directory exists but config is missing"
else
    echo "✖ Nginx is not installed"
fi

# Comparing numbers — disk usage alert
USAGE=$(df / | awk 'NR==2 {print $5}' | tr -d '%')   # e.g. 82
if [ "$USAGE" -gt 80 ]; then
    echo "⚠ Disk is ${USAGE}% full — clean up!"
fi

# Comparing strings
if [ "$1" = "prod" ]; then
    echo "Production deploy — extra careful!"
fi

# One-line shortcuts with && (and) / || (or)
[ -f app.pid ] && echo "App is running" || echo "App is stopped"
Step 5 — Loops (for & while)
bash — loops.sh
#!/bin/bash

# FOR over a list — check several services
for service in nginx docker sshd; do
    echo "Checking $service..."
    systemctl is-active --quiet "$service" && echo "  ✔ running" || echo "  ✖ down"
done

# FOR over files — compress every .log file
for f in /var/log/myapp/*.log; do
    gzip "$f"
    echo "Compressed: $f"
done

# FOR over numbers
for i in {1..5}; do
    echo "Attempt $i of 5"
done

# WHILE — retry until a service responds (real deployment pattern ⭐)
RETRIES=0
while ! curl -s http://localhost:8080/health > /dev/null; do
    RETRIES=$((RETRIES + 1))            # $(( )) = arithmetic
    [ "$RETRIES" -ge 10 ] && { echo "✖ App never came up"; exit 1; }
    echo "Waiting for app... ($RETRIES/10)"
    sleep 3
done
echo "✔ App is healthy"

# WHILE reading a file line by line
while IFS= read -r line; do
    echo "Server: $line"
done < servers.txt
Step 6 — Functions & Case
bash — functions.sh
#!/bin/bash

# A function = named block you can call many times
log() {
    # $1 here is the FUNCTION's first argument, not the script's
    echo "[$(date '+%H:%M:%S')] $1"
}

check_service() {
    local name="$1"                 # local = variable exists only inside function
    if systemctl is-active --quiet "$name"; then
        log "$name is UP"
        return 0                    # function exit code: 0 = success
    else
        log "$name is DOWN"
        return 1
    fi
}

check_service nginx               # call it like a command
check_service docker

# CASE — clean menus instead of long if/elif chains
case "$1" in
    start)   log "Starting app";  systemctl start myapp ;;
    stop)    log "Stopping app";  systemctl stop myapp ;;
    restart) log "Restarting";    systemctl restart myapp ;;
    status)  systemctl status myapp ;;
    *)       echo "Usage: $0 {start|stop|restart|status}"; exit 1 ;;
esac
Step 7 — Putting It All Together: A Real Script
bash — anatomy of a real script
#!/bin/bash
# backup.sh — everything below is genuine daily-use syntax
set -euo pipefail          # exit on error/unset var/pipe failure — ALWAYS ⭐

# Variables (no spaces around =)
APP_NAME="myapp"
BACKUP_DIR="/backups"
TIMESTAMP=$(date +%F_%H-%M)          # command substitution

# Arguments: $1 = first arg, $# = count, $@ = all
ENVIRONMENT=${1:-dev}                 # default value if not passed

# Conditionals
if [ ! -d "$BACKUP_DIR" ]; then
    mkdir -p "$BACKUP_DIR"
    echo "Created $BACKUP_DIR"
fi

# Functions
log() {
    echo "[$(date '+%H:%M:%S')] $1"
}

# Loops
for service in nginx postgresql redis; do
    if systemctl is-active --quiet "$service"; then
        log "$service is running ✔"
    else
        log "$service is DOWN ✖"
    fi
done

# Exit codes: 0 = success, anything else = failure
tar -czf "$BACKUP_DIR/${APP_NAME}_${TIMESTAMP}.tar.gz" /var/www/$APP_NAME \
    && log "Backup complete" \
    || { log "Backup FAILED"; exit 1; }
bash — run it
chmod +x backup.sh
./backup.sh prod                # $1 = "prod"
echo $?                         # exit code of last command
Step 8 — Level-Up Extras (Arrays, Strings, trap, Debugging)
bash — advanced.sh
#!/bin/bash

# ARRAYS — lists of values
SERVERS=("web1" "web2" "db1")
echo "First   : ${SERVERS[0]}"        # web1
echo "All     : ${SERVERS[@]}"        # web1 web2 db1
echo "Count   : ${#SERVERS[@]}"       # 3
for s in "${SERVERS[@]}"; do
    echo "Deploying to $s..."
done

# STRING MANIPULATION — built into ${}, no external tools needed
FILE="/backups/app_backup_2026-07-13.tar.gz"
echo "${FILE##*/}"        # app_backup_2026-07-13.tar.gz  (strip path)
echo "${FILE%.tar.gz}"    # strip extension from the end
echo "${FILE/backup/BKP}" # replace first match
VERSION="v2.1.3"
echo "${VERSION#v}"        # 2.1.3 (strip leading v)

# TRAP — always clean up, even when the script crashes or is Ctrl+C'd ⭐
TMPDIR=$(mktemp -d)
cleanup() {
    echo "Cleaning up $TMPDIR"
    rm -rf "$TMPDIR"
}
trap cleanup EXIT             # cleanup() runs on ANY exit — success, error, Ctrl+C

# HERE-DOC — write multi-line files from a script (configs!)
cat > /tmp/app.conf <<EOF
server_name myapp.com
listen 8080
env production
EOF
bash — debugging your scripts
bash -n script.sh               # syntax check only — does not run it
bash -x script.sh               # trace mode — prints every command as it runs ⭐
set -x                          # turn tracing on inside a script...
set +x                          # ...and off again (wrap the buggy part)

# shellcheck — the linter EVERY professional uses
sudo apt install shellcheck
shellcheck script.sh            # catches quoting bugs, typos, bad practices ⭐
💡 Professional Habit

Run shellcheck on every script before it goes anywhere near a server — many teams enforce it in CI. Combined with set -euo pipefail and trap cleanup EXIT, your scripts are now safer than most you'll find in production.

Bonus — Server Health Report Script (Line-by-Line)

A script you can genuinely run on any server today. Every technique above appears here:

bash — health-report.sh
#!/bin/bash
# health-report.sh — daily server health summary
set -euo pipefail

REPORT="/tmp/health_$(date +%F).txt"       # output file, dated
THRESHOLD=80                                # disk % that triggers a warning

{                                           # { } groups commands → one redirect at the end
    echo "===== HEALTH REPORT: $(hostname) ====="
    echo "Generated : $(date)"
    echo "Uptime    : $(uptime -p)"

    echo -e "\n--- CPU & Memory ---"
    top -bn1 | grep "Cpu(s)"              # -bn1 = batch mode, one sample
    free -h | awk 'NR<=2'                 # first two lines: header + RAM

    echo -e "\n--- Disk ---"
    df -h --output=target,pcent | tail -n +2 | while read -r mount pct; do
        usage=${pct%\%}                    # strip the % sign
        if [ "$usage" -ge "$THRESHOLD" ]; then
            echo "⚠ $mount at $pct — NEEDS ATTENTION"
        else
            echo "✔ $mount at $pct"
        fi
    done

    echo -e "\n--- Top 5 Memory-Hungry Processes ---"
    ps aux --sort=-%mem | head -6         # header + top 5

    echo -e "\n--- Failed systemd Services ---"
    systemctl --failed --no-legend || echo "None ✔"

    echo -e "\n--- Last 5 SSH Logins ---"
    last -5
} > "$REPORT"                              # everything above lands in the file

echo "Report saved to $REPORT"
cat "$REPORT"
✅ Make It Automatic

Schedule it with cron — 0 8 * * * /home/vikram/health-report.sh — and you've built your first tiny monitoring system, the ancestor of what Prometheus does in Module 16.

💡 Why Exit Codes Matter

CI/CD pipelines decide pass/fail purely from exit codes. set -e makes your script stop at the first failure instead of blundering on — exactly how Jenkins and GitHub Actions treat each step. Quote your variables ("$VAR") to survive spaces in values.

12. Environment Variables & PATH

Environment variables are key-value pairs available to processes — the standard way to configure applications without changing code (this idea becomes Kubernetes ConfigMaps and pipeline secrets later).

bash
echo $HOME $USER $SHELL       # built-in variables
env                             # list all environment variables

MY_VAR="hello"                  # shell variable — this shell only
export DB_HOST="10.0.1.5"       # exported — visible to child processes ⭐
DB_PASS=secret ./app.sh         # set for one command only
unset MY_VAR

# PATH — where the shell looks for commands, in order
echo $PATH                      # /usr/local/bin:/usr/bin:/bin...
export PATH="$HOME/bin:$PATH"   # add your own tools dir
which terraform                 # which binary will actually run?

# Make it permanent → add export lines to ~/.bashrc, then:
source ~/.bashrc
💡 Config Files Order

~/.bashrc runs for every interactive shell (aliases, PATH, exports go here). /etc/environment is system-wide. "Command not found" after installing a tool? 90% of the time it's a PATH problem — check with which.

13. Cron Jobs & Task Scheduling

cron runs commands on a schedule — backups, cleanups, report generation, certificate renewals. Each user has a crontab with five time fields:

crontab syntax
┌──────── minute        (0–59)
│ ┌────── hour          (0–23)

14. SSH Key-Based Authentication & Remote Access

SSH (Secure Shell) is how you manage every remote server. Key-based authentication replaces passwords with a cryptographic key pair: the private key stays on your machine (never share it), the public key goes on the server.

Key-Based Auth Flow
ssh-keygen (create pair) → Public key → server's authorized_keys → ssh user@server (no password)
bash
# 1. Generate a modern key pair (on YOUR machine)
ssh-keygen -t ed25519 -C "vikram@laptop"
#   → ~/.ssh/id_ed25519 (private, chmod 600) + id_ed25519.pub (public)

# 2. Install the public key on the server
ssh-copy-id vikram@192.168.1.50
#   (or manually append .pub content to ~/.ssh/authorized_keys on server)

# 3. Connect — no password prompt
ssh vikram@192.168.1.50
ssh -p 2222 vikram@server       # custom port

# Copy files over SSH
scp app.tar.gz vikram@server:/tmp/
rsync -avz ./site/ vikram@server:/var/www/   # sync — only changed files

# ~/.ssh/config — aliases save your sanity
# Host web1
#     HostName 192.168.1.50
#     User vikram
#     IdentityFile ~/.ssh/id_ed25519
# → now just:  ssh web1
✅ Server Hardening Basics

In /etc/ssh/sshd_config: set PasswordAuthentication no and PermitRootLogin no, then sudo systemctl restart sshd. Key-only SSH is also exactly how GitHub authentication and Ansible connections work in later modules.

└────┬──────┘   └──────────┬─────────────┘   └───────┬────────────────┘ └┬┘ └─┬┘
  client IP           timestamp                method + URL + protocol   │    └ response bytes
                                                                          └ HTTP status ⭐

The Analysis One-Liners (Real Incident Toolkit)
bash — log forensics
LOG=/var/log/nginx/access.log

# 1. How many requests per status code? (health overview)
awk '{print $9}' $LOG | sort | uniq -c | sort -rn
#  45231 200    ← healthy traffic
#    892 404    ← broken links or scanners
#    117 500    ← YOUR BUG

# 2. Which URLs are throwing 500s?
awk '$9 == 500 {print $7}' $LOG | sort | uniq -c | sort -rn | head

# 3. Top 10 IPs by request volume (DDoS / abuse check)
awk '{print $1}' $LOG | sort | uniq -c | sort -rn | head

# 4. Requests per minute — find the exact moment traffic spiked
awk -F'[' '{print substr($2, 1, 17)}' $LOG | uniq -c | tail -30

# 5. Everything one suspicious IP did
grep "203.0.113.45" $LOG | awk '{print $4, $6, $7, $9}' | less

# 6. Failed SSH login attempts — who is attacking me?
grep "Failed password" /var/log/auth.log | awk '{print $(NF-3)}' | sort | uniq -c | sort -rn | head

# 7. Errors in a specific time window (during the incident)
awk '/13\/Jul\/2026:10:2[0-9]/ && $9 >= 500' $LOG | wc -l
logrotate — Why Your Disk Isn't Full of Logs

Logs grow forever unless rotated. logrotate (runs daily via cron/systemd) renames the current log, compresses old ones, and deletes the oldest — that's why you see app.log, app.log.1, app.log.2.gz on every server.

/etc/logrotate.d/myapp
/var/log/myapp/*.log {
    daily                 # rotate every day (or: weekly, monthly, size 100M)
    rotate 14             # keep 14 old logs, delete older
    compress              # gzip rotated logs
    delaycompress         # keep newest rotation uncompressed (still being read)
    missingok             # no error if the log doesn't exist
    notifempty            # skip rotation if the file is empty
    create 0640 myapp myapp   # recreate the log file with these permissions
    postrotate
        systemctl reload myapp > /dev/null 2>&1 || true   # tell the app to reopen its log file
    endscript
}
bash — test & run
sudo logrotate -d /etc/logrotate.d/myapp    # -d = dry run, shows what WOULD happen
sudo logrotate -f /etc/logrotate.d/myapp    # -f = force a rotation right now
💡 The Bigger Picture

Everything here scales up later: these awk pipelines become LogQL queries in Loki (Module 16), and logrotate's job is done by log shippers and retention policies in Kubernetes. Learn it raw first — the concepts transfer exactly.

Interview Questions

Q1

A service is down on a Linux server. Walk me through your troubleshooting steps.

Check service state with systemctl status app; read recent logs with journalctl -u app -n 100 and application logs in /var/log; verify resources (df -h for disk full, free -h for memory, top for CPU); check the port isn't taken (ss -tulpn); validate recent config changes; then restart and watch logs live with journalctl -u app -f. Disk-full and port-conflict are the two most common real causes.
Q2

Explain chmod 755 vs 644. Why must an SSH private key be 600?

Digits are owner/group/others with r=4, w=2, x=1. 755 = rwx for owner, r-x for group and others — right for scripts and directories. 644 = rw- for owner, read-only for everyone else — right for data files. A private key must be 600 (owner read/write only) because the SSH client refuses keys readable by others — any other user on the machine could steal your identity.
Q3

What's the difference between a process and a service? How does systemd fit in?

A process is any running program with a PID. A service (daemon) is a long-running background process managed by the init system — on modern Linux that's systemd (PID 1). systemd starts services from unit files, restarts them on failure, handles dependencies and boot order (systemctl enable), and captures their logs into the journal (journalctl).
Q4

How would you find the top 5 IPs generating errors in an access log?

grep " 500 " access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head -5 — grep filters error lines, awk extracts the IP column, sort groups identical IPs together, uniq -c counts them, sort -rn orders by count descending, head takes the top 5. Explaining each stage of the pipe is exactly what the interviewer wants to hear.
Q5

What does set -euo pipefail do and why use it in every script?

-e exits the script when any command fails; -u errors on undefined variables (catches typos like rm -rf $TYPO/ becoming rm -rf /); -o pipefail makes a pipeline fail if any stage fails, not just the last. Without these, scripts continue past failures and cause damage silently — in CI/CD, a script that hides failure is worse than one that crashes loudly.
Q6

How does SSH key-based authentication work?

Asymmetric cryptography: ssh-keygen creates a private key (stays on the client, 600 permissions) and a public key (added to ~/.ssh/authorized_keys on the server). At login the server sends a challenge that only the holder of the private key can sign; the server verifies the signature with the public key. No secret ever crosses the network, which is why it's safer than passwords and standard for automation (Git, Ansible, CI/CD).
Q7

The disk is 100% full on a production server. What do you do?

Confirm with df -h (which mount?), then locate the biggest consumers: du -sh /var/* | sort -rh | head — it's usually logs in /var/log. Free space safely: truncate a huge active log (truncate -s 0 big.log) rather than deleting it (a deleted file still held open by a process frees nothing — find those with lsof | grep deleted). Long-term fix: configure logrotate and disk-usage alerts in monitoring.
