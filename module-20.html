2. AI-Assisted Coding & Infrastructure as Code

💡 How to use Copilot effectively for DevOps work

Don't just accept suggestions — treat Copilot like a fast junior engineer. It's excellent at: generating Terraform resource blocks from comments, writing Python boto3 scripts from a natural-language description (Module 14 payoff), converting shell one-liners to reusable functions. Where it fails: Terraform remote state configuration (the relationships between providers are subtle), Kubernetes RBAC (gets permissions wrong frequently), and anything requiring org-specific context it doesn't have.

python — LLM-augmented log triage (boto3 + OpenAI)
# ai_triage.py — feed recent CloudWatch errors to an LLM for probable root cause
import boto3, json, openai

cw = boto3.client('logs', region_name='us-east-1')
client = openai.OpenAI()  # assumes OPENAI_API_KEY env var

def get_recent_errors(log_group: str, minutes: int = 30) -> str:
    import time
    end = int(time.time() * 1000)
    start = end - (minutes * 60 * 1000)
    resp = cw.filter_log_events(
        logGroupName=log_group,
        startTime=start, endTime=end,
        filterPattern='ERROR'
    )
    return '\n'.join(e['message'] for e in resp['events'][:20])

def ai_triage(log_group: str):
    errors = get_recent_errors(log_group)
    if not errors:
        print("No recent errors found.")
        return

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{
            "role": "system",
            "content": "You are an SRE assistant. Analyze these logs, identify the probable root cause, and suggest specific kubectl or AWS CLI commands to investigate further."
        }, {
            "role": "user",
            "content": f"Recent errors from {log_group}:\n\n{errors}"
        }]
    )
    print(response.choices[0].message.content)

# Usage: python ai_triage.py
ai_triage('/aws/eks/techitfactory/api')

⚠️ AIOps ≠ no-ops

AIOps tools reduce alert noise and accelerate triage. They don't make decisions. "DevOps Guru flagged an anomaly in RDS CPU" still requires an engineer to determine whether it's a query that needs optimization, a traffic spike that needs scaling, or a backup job that's expected. The engineer who understands the system (Modules 10-19) uses AIOps as a faster compass — not a substitute for judgment.

8. Interview Framework — The STAR-SRE Method

DevOps/SRE behavioral interviews reward quantified, incident-anchored stories. Use STAR but add the SRE layer: what did the data say, what was the error budget impact, what systemic change did you make?

text — STAR-SRE template
Situation: "Our payment service was paging 8× per week — every on-call engineer
            was spending 3+ hours per shift on manual restart procedures."

Task: "As the on-call lead, I owned reducing page frequency and eliminating the
      manual toil."

Action: "I ran 5-Whys on 3 months of incident data. Root cause: a memory leak
        in the payment processor on high-traffic evenings. Short-term: added HPA
        with memory-based scaling + resource limits. Long-term: added prom-client
        instrumentation, identified the leak in a third-party library, pinned to
        the patched version in the Helm chart."

Result: "Pages dropped from 8/week to 0.5/week over 6 weeks. MTTR fell from
        42 minutes to 4 minutes for the remaining pages. On-call engineer
        satisfaction score improved from 2.1 to 4.3/5 in the next survey.
        Error budget consumption dropped from 120% to 18% per month."

SRE layer: "This also let us unfreeze the error budget policy — we'd been in
           feature freeze for 6 weeks. Product team shipped 3 features in the
           month after the reliability work."

Final Interview Questions

The hardest questions at Senior/Staff level. Click to reveal mastery-level answers.

Q: Walk me through your full DevOps stack.

Here's the production stack I built for TechitFactory: Code in GitHub (branch protection, PR reviews). Jenkins CI runs: Gitleaks secret scan, Semgrep SAST, Docker multi-stage build, Trivy container scan (fails on CRITICAL CVEs), generates SBOM, pushes to ECR. CI commits the image tag to a GitOps config repo. ArgoCD on EKS detects the commit, auto-syncs dev (selfHeal on), requires a manual sync for prod after a PR approval. The EKS cluster has OPA Gatekeeper enforcing resource limits and registry policies, Falco for runtime anomaly detection, HPA for autoscaling. Observability: kube-prometheus-stack with RED-method dashboards, 99.9% availability SLO, error-budget-burn AlertManager rules routing to PagerDuty. Rollback is git revert — ArgoCD syncs the revert in under 5 minutes. Infrastructure is Terraform with S3+DynamoDB remote state, organized as modules for reuse. The whole stack bootstraps from an ArgoCD App of Apps — disaster recovery is one kubectl apply.

Q: How does AI change your role as a DevOps engineer?

AI makes me faster at the parts that don't require judgment — boilerplate IaC, script scaffolding, first-pass CVE triage. I use Copilot for Terraform resource blocks and Python boto3 scripts; what used to take 30 minutes takes 5. For incidents, I've built an LLM-assisted log triage tool that feeds recent CloudWatch errors to GPT-4 and gets a probable root cause with suggested commands — it's not always right, but it narrows the search space. AIOps tools like AWS DevOps Guru handle anomaly detection that we'd have needed ML expertise to build ourselves. What AI can't replace: the decision to roll back a deploy at 2 AM when the error budget is burning, the architecture judgment call about whether to use DynamoDB or RDS for a given access pattern, or the organizational navigation to get a blameless post-mortem culture adopted. I see AI as eliminating the low-judgment toil — which frees more time for the engineering work that actually matters.

Q: What's the hardest technical problem you've solved?

[Frame around your capstone work] The hardest part of building the capstone was wiring Terraform, ArgoCD, and the EKS IRSA chain correctly. The problem: ArgoCD needs permissions to read ECR (to pull images), but the pod identity chain goes: ArgoCD pod → K8s service account → IRSA IAM role → ECR. Getting the aws-auth ConfigMap, the OIDC provider, the IRSA annotation, and the trust relationship all consistent took me several iterations. The lesson: IRSA failures are silent in logs — the pod starts normally but image pulls fail with a generic auth error. I learned to validate the IRSA chain independently using aws sts assume-role-with-web-identity before attaching it to ArgoCD. Now I document this in a runbook so future team members have the debugging path immediately. This kind of layered identity problem — where the failure mode is misleading — is exactly where deep system understanding beats surface-level tool knowledge.

Q: You're on call. API error rate just spiked to 40%. Walk me through your response.

First: assess severity and open the incident channel. 40% error rate = P1. Declare IC and get a scribe. Step 1: check for a recent deploy in ArgoCD — most incidents correlate with the last change. If there was one in the last 30 minutes, roll back first (git revert, verify ArgoCD syncs, watch error rate). If no recent deploy: Step 2: kubectl get pods — any CrashLoopBackOff or OOMKilled? Step 3: kubectl logs on failing pods — what's the error? Step 4: check Grafana — is this latency (possibly HPA underprovision), errors (upstream dependency), or both? Step 5: check downstream dependencies (RDS connection pool, external APIs). Throughout: keep the scribe logging the timeline. Communicate status to stakeholders every 10 minutes even if just "still investigating." Once mitigated, set a time-bounded monitoring period before clearing. Then: schedule post-mortem within 48 hours, not next sprint.

Q: How do you manage technical debt in infrastructure?

Infrastructure debt comes in a few forms: drift (live state ≠ Terraform state — GitOps and terraform plan in CI catch this), version lag (outdated Kubernetes, Terraform providers, Helm charts — I maintain a quarterly upgrade calendar), toil accumulation (manual processes that grew — I track toil ratio and treat anything over 30% as a P2 backlog item), and architectural debt (wrong tool for the job — this needs a proper ADR and migration plan). My practice: every 6 weeks, I run a "debt audit" — terraform plan on all workspaces, kubectl get nodes -o yaml for k8s version, trivy on base images. The output is a prioritized ticket list. The key cultural piece: debt reduction work gets sprint capacity, not just "get to it when you have time." Debt compounds; a minor version lag today becomes an emergency upgrade in 6 months when a CVE hits.

Q: What's your 30-60-90 day plan if you joined this team?

Days 1-30: Learn and listen. Read existing runbooks, post-mortems, architecture docs. Shadow 3 on-call shifts before taking primary. Map the system: what's the CI/CD path, where's the state, how is production traffic routed? Identify the top 3 pain points by asking the team, not assuming I know. Days 30-60: Deliver one thing. Pick the highest-pain toil item from my listening phase and eliminate it — a script, a pipeline stage, a monitoring gap. Prove I can ship in this environment. Days 60-90: Propose and start a medium-term initiative — likely an SLO implementation if the team lacks them, or a security stage if the pipeline is missing it. By day 90, be fully independent on-call primary and have at least one post-mortem authored. The goal: be a net positive to the team by day 45, not day 90.
